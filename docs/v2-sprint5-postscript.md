## ★ 이번 세션 핵심 교훈 (4개)
1. **잠복 마이그레이션은 첫 "쓰기"에서 터진다.** S1 컬럼 SQL이 이 DB엔 미반영이었는데 읽기(`?? 기본값`)가 가려서
   안 보이다가 **S4 INSERT(첫 쓰기)에서 표출**. 합본 멱등 SQL로 흡수. 교훈: 실제 쓰기 경로가 생기는 단계까지 가야
   마이그레이션이 진짜 검증됨. SQL 에디터 성공 ≠ 반영 확인.
2. **SQL 에디터는 RLS 우회 → UPDATE 정책 검증의 거짓양성.** anon 키 경로 .select()→data.length>0가 유일한 실증(S5).
3. **report jsonb 공존은 양방향 다 확인.** 같은 jsonb를 관찰·캐시가 나눠 쓰면 한쪽이 통째 교체 시 상대 데이터 조용히 증발.
4. **★값 컨벤션(한글 vs 영문)까지 raw로 봐야 한다 — S7이 잡은 Step5 잠복버그.** Step5의 PtConfirmBanner·nextAction이
   한글 `"성공"`/`"보류"`를 검사했는데 **실제 저장값은 영문 `"success"`/`"hold"`**였음 → **Step5 배너가 실데이터에선
   한 번도 안 떴음**(테스트 데이터가 한글이라 통과한 거짓양성). §1 성공≠PT 게이트의 UI가 프로덕션에서 구멍이었던 것.
   S7 writer 붙이며 reader 값 정렬하다 발견, step7-b(4692cb0)로 수정. 교훈: raw 검증 시 **값 자체가 실제 앱 컨벤션과
   같은지**까지 볼 것(status 있는지만 보지 말고).

  ⚠️ **구독료 방어를 '수정 막기'로 하지 말 것** — 수정 막아도 등록 무제한이라 방어 안 되고 정직한 트레이너만 불편. 돌려쓰기·공유 방어는 ⑦(로그인·trainer_id·계정정책)의 몫.
  착수 = B 후/③ 언저리.
- **2차 "황현진 4유형"을 1차/①에도 노출**(추가기능). ⚠️ 관찰 기반이면 1차 정확도 낮음 주의.
- **관찰 기록 탭 1차 클로징 입력 순서 재검토**(현장 동선 확인 후). ⚠️ ⭐ "관찰→결과" 철학 주의 — UI 순서만.

## ② 종료 후 = 현장 사용성 패스 (③ 가기 전 — 최우선)
- **코드 진도 멈추고**, 트레이너 본인이 **실기기(폰)에서 실제 회원 한 명 전체 흐름**(등록 → 1차 → 관찰 → 2차 → 확정,
  그리고 보류→재접근→홈 카드)을 시뮬레이션으로 끝까지 써본다. 개발용 강제 주입 말고 진짜 수업 흐름.
- 뽑을 것 = **"매일 못 쓰겠는 치명적 마찰"만**(버튼 위치, 글씨 크기, 입력 마찰, 수업 중 못 쓰는 지점). 사소한 카피·순서는 ⑥으로.
- 이유: 지금까지 검증은 전부 "기능이 도는가"(build·end-to-end)였고 **"엄지로 편한가"는 실측 0**. daily 앱 판정은 트레이너 손에서만.
- 여기서 나온 치명 마찰은 ③ 전에 먼저 처리. (홈 카드 위치 격상·회원 dedup 등 S7에서 열어둔 판단도 여기서 실사용 보고 결정.)

### 결과 (폰 실측 완료 — 이번 세션)
- **경로 A·B 폰 실측 완료.** 검증축이 "기능이 도는가"(build·end-to-end) → **"엄지로 되는가"**(실기기)로 이동.
- **치명 마찰 — 처리 완료(위 커밋 체인):**
  - **배너 stale refetch:** PtConfirmBanner 조회 useEffect 의존성이 `[member?.id]`뿐이라, 같은 회원에서 클로징 'success'
    저장(형제 컴포넌트)해도 배너가 옛 rounds를 들고 있어 안 떴다(재선택/새로고침해야 뜸) → 부모 `closingVersion` 카운터 +
    ObservationTab·SecondOTTab 저장 성공(하드닝 `data.length>0` 통과) 직후 `onClosingSaved` 콜백 → 배너 즉시 재조회. 1·2차 both 실증.
  - **배너 개발자언어 자연어화**("성공 기록 ≠ 자동 등록 · §1" → "성공을 기록해도 자동으로 등록되진 않아요").
  - **확정 중앙 확인 모달**(인라인): '확정' → 회원명 확인 모달 → [확정]에서만 `toPtActive`(로직 무변경, 앞 게이트만). 실수 확정 방지 + 배치(가운데라 안 놓침).
  - **1차 ① 안내 자연어화**(캐시/세션/0행 → 자연어) + **입력 무변경 시 재생성 소프트 억제**(`firstInputHash` 동일 → 버튼 흐리게+힌트,
    **클릭은 됨** = 하드락 아님·재시도 살림). 반복 Haiku 호출 비용 방어.
- **배너 노출 정책(버그 아님, 확정):** `PtConfirmBanner`는 `view==='ot'`에서 **탭 위에 상주** → 모든 OT 탭에서 보인다 = **의도**.
  탭별로 좁히지 말 것(놓침 위험 + 확정은 회원 단위 결정). 게이트(`status==='ot_active' && (round1||round2 success)`) 정상 —
  빈/미성공 회원엔 안 뜬다.
- **치명 마찰 — 남음(B로 처리):** **2차 OT 탭 흐름이 수업 동선과 역순**(실행/기록이 위, 준비가 아래). 결정: **준비(수업 전)→기록(수업 후)**.
  "수업 중" 실시간 블록은 성립 안 함(트레이너가 수업 중 폰을 못 봄). → 아래 "2차 OT 탭 재편(B)".

## 2차 OT 탭 재편 (B) — 현장 사용성 패스의 최대 산출물
> 폰 실측에서 나온 "2차 동선 역순" 치명 마찰의 처리. **B1(구조) → B2(내용)** 분할. ③ 전 우선.

### B1 — 구조 재편 (다음 세션 착수)
- **삭제:** "실시간 자극 결과" 블록(`SECOND_ACT`). 수업 중 폰 못 봄 = 실시간 입력 전제 붕괴. **세션 토글이라 삭제해도 데이터 안 잃음**(DB 미저장).
  미커밋 A(라벨 교체)는 여기서 흡수(어차피 지울 블록).
- **재배치:** **"AI 지원 준비(수업 전) → 클로징 결과 기록(수업 후)"** 순서. 등록 당위성·ARC·거절대처(황현진 4유형)·"이렇게 하면 더 좋아져요"를 **AI 지원 아래로** 묶음.
- **생성 로직 무변경:** 기존 `report.brief` 캐시 유지, 1차 FirstOTAssist처럼 "AI 지원" 눌러 자동 생성 UX.
- **캐시 방식 확정:** `report.brief` 캐시 **유지**(재방문 재호출 0·네트워크 안정·일관성). **1차 무캐시와 다른 게 맞다** — 2차는 관찰이 저장되니 캐시 근거 있음.
  스테일 배지+재생성으로 갱신.

### B2 — 재료·내용 강화 (B1 후)
- **관찰일지(1차 ObservationTab)에 트레이너 자유서술 필드 신설**(`report.trainer_note`, **새 컬럼 0**). "입력=사고" 철학상 1차에 미리 적어둠.
- **ot-brief 2차 프롬프트 재료 3종:** 관찰 정형필드 + 자유서술 + 성향.
- **자극반응 3갈래 대처 시나리오를 AI 생성 섹션으로**(삭제한 실시간 블록 개념의 재탄생): "자극 잘 옴/약함/안 옴" 각 대처를 **수업 전 브리핑**으로. 회원별 관찰 기반(고정문구 X).
- **세일즈 강도 = 1차보다 세게, 단 "근거의 세기"지 "압박의 세기" 아님.** 등록 당위성·논리 클로징까지(마스터플랜 추가기능B).
  ⭐ 압박 대본 금지. "안 옴"일 때 "그러니 더 등록" 몰이 금지 → **원인 짚고 방향**.
- **프롬프트 튜닝은 harness A/B 실측 영역** → B1(재배치)과 분리.
