## 0. 정체성 — 이 앱은 무엇인가

프롬프트·스키마·UI 모든 결정이 여기서 갈리므로 먼저 박아둔다.

- **궁극 목적 = 트레이너 역량 성장.** 이 앱은 **세일즈 도구가 아니라 트레이너 성장 엔진**이다.
  세일즈 부담을 덜어 그 시간·에너지를 실력 향상에 쓰게 하고, AI 출력의 **'왜'(원리·근거)** 로
  트레이너를 공부시킨다. **세일즈는 입구, 성장이 목표.**
- **타깃 = 매출이 어려운 베테랑·중급 + 영업 안 해본 신입 트레이너.** 특히 **하루 12~14개 수업하는
  바쁜 트레이너.** 고성과 트레이너는 자기 방식대로 하니 타깃 아님.
  → 방식 중립으로 밍밍하게 X. **검증된 방식을 명확히 "이렇게 하세요"로 떠먹여주되,
  "당신 스타일로 응용 가능"의 여지만 열어둔다**(그 여지의 장치가 `principle`). 낭독 대본과는 다르다 —
  **예시(대화·큐잉)는 흐림, 원리·방향은 명확.**
- **사업 모델 = 트레이너 상향평준화 → 센터 매출 상승 → 전국 센터 도입.** 도입 경로는 **바텀업**
  (트레이너 긍정 반응 → 센터 도입). 대형 센터 설득의 핵심 무기는 **관리자용 통계 대시보드**
  (로드맵 ④ 클로징 통계와 연계).
- **설계 원칙 추가(⭐⭐ "정답 대본 X, 근거+방향 O"의 강화판): 모든 AI 출력에 '왜'가 따라붙는다.**
  답을 주는 동시에 트레이너가 이해하고 **다음엔 스스로** 하게 한다(단순 의존 X). `principle`이 대표 예.

> **⚠️ 생존 조건 2개 (타깃이 신입·바쁜 트레이너라 필수):**
> 1. **3초 안에 핵심이 눈에 들어와야 한다** — 안 열어보면 끝. → 화면 핵심 3줄(§7 화면 구조).
> 2. **환각 한 번에 신뢰가 무너진다** — **가드레일은 윤리가 아니라 제품 생존 문제.**
>    FMS 채점 세부·PT 금액 등을 **지어내지 않는다**(§9 환각 고위험 지점).

### 0.1 큰 결정 — 탭1 전면 교체

**1차 강화 = 탭1 하드코딩 전면 은퇴 → AI 교체.**

S1~S3은 탭1을 건드리지 않고 위에 ①(`FirstOTAssist`)만 얹는 **추가형(additive)** 이었다. 이 스프린트는
그 원칙을 명시적으로 종료한다. 탭1의 김철수 전용 하드코딩 스캐폴드
(타이머·`PHASES`·성향3줄·`SALES_SCRIPT`·`buildClosing`·`ROUTINE`·`MOBILITY` + 헤더 타이머 컨트롤)를
**전면 은퇴**시키고, 그 자리를 `FirstOTAssist`(실 AI)를 척추로 하는 6단계 AI 흐름으로 교체한다.

근거: 탭1 legacy는 대부분 **`FirstOTAssist`가 이미 회원별 실 AI로 생성하는 것의 하드코딩 쌍둥이**다.
대본은 문자 그대로 "개발하실 때 로그 안 보고 디버깅…", "철수님 무릎"이라 다른 회원에겐 틀린 내용이고,
`ROUTINE`/`MOBILITY`는 세트 수치를 담아 v1.5 철학(숫자 처방 금지)에 어긋난다.

| 탭1 legacy 블록 | 처리 | 대체 / 비고 |
|---|---|---|
| 60분 타임라인 + 라이브 타이머 | **은퇴** | 6단계 arc 길잡이로 대체 (§2) |
| 헤더 `tab===1` 타이머 컨트롤 | **은퇴** | 타이머 제거의 종속 작업 — 회귀 위험 ㉠ (§9) |
| `PHASES` (4단계 하드코딩) | **은퇴** | arc가 흐름을 대신함 |
| 성향 3줄 `member.summary` | **은퇴** | `FirstOTAssist.hypothesis` (회원별 실 AI) |
| `SALES_SCRIPT` 대본 | **은퇴** | `FirstOTAssist.arc` (왜+방향, 회원 인지) |
| `buildClosing` + 자극/무릎/기분 빌더 | **은퇴** | `closing_compass` / `soft_closing` (하드코딩은 낭독 대본 → 위반) |
| `ROUTINE` / `MOBILITY` 운동(세트 수치) | **은퇴** | `movement_cues` 치트키 1개 — 방향+원리+큐잉+대화 (§4·§5) |
| `FINANCE` 추정 연소득 | **보류** | 회원기반화는 윤리 검토 필요 → 별도 논의 (§10) |
| `PACKAGES` 가격 제안 | **유지** | 세일즈의 패키지 제안 재료로 활용(금액 지어내기 금지). 실 가격은 ③ 소관 |
| 현장 메모(`notes`, 세션 로컬) | **유지** | 경량 편의 기능, 변경 없음 |

> **원칙 확인:** ①은 계속 '가설'이다(관찰 아님). 은퇴하는 건 "정답 대본"이고, 남기는 건 "근거+방향"이다.

### 0.2 ⚠️ 구현 우선순위 (제일 중요)

아래 설계는 **1차 OT 완성형 청사진**으로 문서에 전부 담되, **한 스프린트에 다 구현하지 않는다.**
2차(S3)의 '압축된 강력함'을 잃을 위험이 있다. **핵심에 먼저 집중하고, 나머지는 실사용하며 튜닝**한다.

| 구분 | 항목 |
|---|---|
| **핵심 (이번 구현)** | **탭1 → `FirstOTTab.jsx` 추출(코드 다이어트)** · 6단계 arc · 자각/평가동작 · 직업·나이 세일즈 · 화면 핵심3줄/펼치기 · `principle`('왜' 원칙) |
| **단계적 (핵심 후 얹기)** | 미지근 4원인 세분화 · 기구 회원 경로(`machine_only`) · MBTI 가설 · PT 패키지 제안 · 라포 팁 심화 |

본문 각 항목에 **[핵심]** / **[단계적]** 을 표시한다.

## 1. 배경 — 왜 ①부터인가

여정의 첫 관문이다. 1차가 얕으면 그 뒤(관찰·2차 브리핑)의 재료가 부실해진다. "정보를 앞단에서
최대한 모으고 단계마다 쌓아 다음 단계의 방향을 만든다"(MASTERPLAN ⭐ 파이프라인)의 출발점을 두껍게
만드는 작업. 세로 슬라이스(1차 → 2차 → PT)를 끝까지 완성하기 위해 **완성도 먼저.**

## 2. 타이머 → 6단계 길잡이 (arc 재구성)

타이머는 경과 시간을 잴 뿐 수업의 질·클로징 확률과 무관하다. 걷어내고, `arc`를 탭1의 **척추**로
승격해 **트레이너 실제 1차 OT 흐름**을 담게 한다.

이제 arc는 트레이너 실제 흐름을 담는 **6단계**다. **인바디 측정은 별도 단계가 아니라 ②문진·공감
안에 자연스럽게 녹인다**(상담하며 인바디를 함께 짚는 실제 흐름). 직전 초안의 '인바디 placeholder
(7번째 위치)'는 ②로 흡수돼 사라졌다 — 6개 비트로 깔끔해졌다.

> **⚠️ 모든 비트 관통 원칙 — AI 출력은 정답이 아니라 '발판(scaffold)':**
> 각 비트의 방향·예시는 "이렇게 하면 된다"는 **참고일 뿐, 최종 판단·실행은 트레이너 것**이다.
> **목적(무엇을 이루려는가)은 명확히 제시**하되(밍밍한 방치 X), **수단은 강요하지 않는다**(획일 X).
> 톤: *"목적은 이것, 예를 들면 이렇게, 본인 방식 있으면 그대로 쓰세요."* ⭐⭐ 스파링 파트너이자
> §0 '성장 엔진'(**의존이 아니라 발판**)의 톤을 arc 전체에 관통시킨다.

| # | arc 비트 | 핵심 내용 | 우선순위 |
|---|---|---|---|
| 1 | **라포 형성** | 회원 정보 기반 라포 팁 — 뭘로 말문 열지·첫 화제. 라포 어려운 트레이너용, **힌트로(대본 X, 발판)** | [핵심] (팁 심화는 [단계적]) |
| 2 | **문진·공감 (+ 인바디 측정)** | 목적·불편함·왜·수술이력 파악. **상담하며 인바디를 자연스럽게 녹인다** — 흐름 안내만, **수치 해석·설명은 앱이 생성 X(트레이너 몫, §3)** | [핵심] |
| 3 | **회원이 스스로 문제 자각** ← **핵심 가치** | **목적(회원이 스스로 자각) 고정, 수단은 자유.** 평가 동작은 **'방법의 한 예시(참고)'** — "자각이 목적, 예를 들면 이런 동작(오버헤드 스쿼트 등), 본인 방식 있으면 그대로 쓰세요". 시각 판단 못 하는 트레이너에겐 동작이 발판. `observe_targets`와 연결 | [핵심] |
| 4 | **치트키 운동** | `movement_cues`. 즉효 체감 + 라포 + 니즈 충족. **평가동작(진단) ↔ 치트키(해결) 짝**. 치트키도 예시 발판 | [핵심] |
| 5 | **수업 피드백** | 오늘 몸에서 일어난 것 되짚기 | [핵심] |
| 6 | **세일즈** | `closing_compass`. 라포 활용 + 회원 목적 축. 예시 화법은 발판 | [핵심] (machine 경로·MBTI·미지근4·패키지는 [단계적]) |

> **⚠️ 자각 비트 환각 가드:** FMS 등 평가 도구는 **가리키기만** 하고 **채점 기준의 세부를 지어내지
> 않는다.** "이런 동작을 시켜 이런 반응을 보라"까지가 AI의 몫(그것도 예시), 정밀 채점표는 트레이너/
> 실제 도구의 몫.

### closing_compass — 세일즈 비트의 나침반

"이 방향으로 가면 클로징 확률이 왜 높아지나"를 명시. `approach_tag`(목적 축) · `why_higher_odds`
(사실 기반 근거) · `watch_for`(안 먹힐 신호 = 브레이크). 확장 필드는 §6·C.

> **⚠️ 톤 가드:** "확률 높은 방향"은 **압박 플레이북이 아니라 공감 가설**이다. PREAMBLE이 이미 허위
> 긴급성·FOMO·죄책감을 금지하고 "강한 클로징의 원천은 압박이 아니라 '나를 정확히 이해했다'는 공감"
> 이라 못 박아뒀으니 그 톤을 상속한다. `watch_for`는 이 화법의 **안전핀** — 공감이 안 통하는 신호를
> 미리 줘서 억지 클로징을 막는다.

## 3. 인바디 = ②문진·공감에 통합 (입력폼·저장 없음)

인바디는 **별도 단계가 아니라 ②문진·공감 흐름 안에 녹인다**(상담하며 인바디를 자연스럽게 짚는 실제
흐름). arc는 ②에서 **"이 흐름으로 상담하며 인바디를 함께 다루라"** 는 안내만 하고,
**수치 해석·설명 내용은 앱이 생성하지 않는다**(트레이너 몫).

> **⚠️ 이전 설계 폐기 (S4 개정):** 직전 초안의 **인바디 유연 입력폼 · `ot_log` round-0 저장 ·
> `intakeHash` · upsert 병합**은 **전부 제거**한다. 인바디 입력폼도, round-0도, 별도 인바디 단계도 없다.

**결과:**
- **① 생성 재료 = 회원 기본정보(`user_table`)** — age·job·residence·mbti·pain·goal·machines.
  별도 사전 입력폼 없음.
- **① 캐시/저장 없음 = 세션 생성**(S3의 `FirstOTAssist` 방식 유지). 탭 이동 시 재생성. `ot_log`에
  ① 쓰기 없음 → **신규 저장/RLS 작업 0**, round-0 공존·시퀀싱 리스크 소멸.
- `observe_targets`(자각 비트에서 나온 관찰 항목)는 **트레이너가 '관찰 기록' 탭(round-1)에 수동
  이관**(기존 방식 그대로). ① → 관찰 → ② 파이프라인은 유지.
- *(캐시가 실사용에서 필요해지면 별도 논의 — 이번 범위 밖.)*

## 4. 치트키 운동 (`movement_cues`) — 치트키 + 큐잉 + 대화 + 클로징 연결  [핵심]

트레이너가 원하는 건 "이 운동 해라"가 아니라, **"이 운동을 이런 큐잉·대화와 함께 풀면 회원이 니즈
충족을 몸으로 느끼고, 그게 클로징으로 이어진다"** 는 길잡이다. **평가동작(진단)에 대한 해결책**으로
짝을 이룬다(§2 자각 비트 ↔ 치트키). 즉효 체감 + 라포 + 니즈 충족을 한 번에.

- **치트키 1개만.** 여러 개 나열해 레시피가 되면 사고 정지 → 하향평준화(v1.5).
- **필드:**
  - `exercise` — 동작(방향까지만. 세트·횟수·각도 등 **숫자 처방 금지**)
  - `why_instant` — 왜 즉효인가(회원이 그 자리에서 '되네?'를 느끼는 이유)
  - `cueing` — 현장 큐잉(**흐림 처리** — 낭독기 방지)
  - `dialogue` — 함께 건넬 대화 결(**흐림 처리**)
  - `connects_to_closing` — 이 몸의 경험이 어떤 클로징 논리로 이어지나(`closing_compass` 연결)
  - `principle` — §5 참조
- **숫자 처방 금지**(PREAMBLE이 차단). 큐잉·대화는 '예시'로 낮춰 트레이너가 자기 스타일로 채우게.

## 5. `movement_cues.principle` — 원리로 응용, 링크는 금지  [핵심]

- **`principle`:** 이 동작이 통하는 **원리**를 설명한다. 목적은 "이거 따라 해"가 아니라, 트레이너가
  **같은 원리로 자기만의 필살기를 응용**할 수 있게 하는 것. §0 "모든 출력에 '왜'" 원칙의 대표 장치.
- **🚫 운동 영상 링크(URL) 절대 생성 금지.** AI가 URL을 환각한다(존재하지 않는 링크·잘못된 영상).
  링크 대신 **원리 설명**으로 응용을 돕는다.
- **로드맵 메모:** 실제 영상 연동은 나중에 **웹검색 도구로 별도 구현**(AI 생성 URL이 아니라 검증된
  검색 결과). 이번 스프린트 범위 아님.

### C. 세일즈(`closing_compass`) 설계 상세

- **목적 유형별 접근 축 분기** — 불편함 개선 목적 → **통증/개선 축**[핵심]. 기구사용법 목적
  (`goal_type=machine_only`) → **효율/안전 축**("혼자 하면 비효율·부상, 제대로 배우면 목표에 빨리")
  **+ 표면 니즈 뒤의 진짜 목표 탐색**[단계적]. `goal_type`에 따라 compass·자각 비트가 갈린다.
- **직업·나이 적극 반영**[핵심] — 좌식→어깨, 40대→건강 등 **사실 기반**.
- **MBTI는 있으면 참고, 없으면 생략**[단계적] — 세일즈 뼈대는 목적·불편함·직업·나이. MBTI는 있을
  때만 보조. **있어도 단정 금지**("~성향이면 ~할 수 있어요" 가설 + "실제 반응으로 확인"). 없으면
  지어내지 말 것(회원이 몰라 빌 수 있음).
- **미지근한 회원 대처**[단계적] — 반응 미지근할 때 **원인 진단 후 대응**. ⚠️ "뚫는 법"이 아니라
  "**원인별 대응**". **신뢰 부족이면 '지금 미루기'가 정답**(`watch_for` 안전핀, 압박 방지).
- **PT 패키지 제안**[단계적] — ⚠️ **AI가 금액 지어내기 금지.** 센터 등록 패키지(현 `PACKAGES` 상수,
  추후 실데이터화)에서 고른다. '뽑아내기'가 아니라 '**이 회원 목표에 맞는 최소 필요량 + 왜**'(회원
  이익 기반). 예산 부담 신호 시 부담 적은 옵션부터(`watch_for`). 이번엔 추천 로직·방향까지, 정확한
  금액은 센터 가격 실데이터화(③) 후.

### 흐름 매핑 (arc ↔ 필드)

`arc`가 6단계 흐름(라포 → 문진·공감**(인바디 포함)** → 자각 → 치트키 → 피드백 → 세일즈)을 담고,
그중 '자각' 비트 → `observe_targets`, '치트키 운동' 비트 → `movement_cues`, '세일즈' 비트 →
`closing_compass`와 맞물린다. 평가동작(자각·진단) ↔ 치트키(해결)가 짝. AI가 *"이렇게 라포 열고 →
상담·인바디로 공감하고 → 이 동작(예시)으로 자각시키고 → 이 치트키로 해결 체감시키고 → 이 축으로
세일즈하면 클로징 확률이 높다"* 를 하나의 흐름으로 길잡이하는 구조. **모든 비트의 방향·예시는 정답이
아니라 발판** — 목적은 명확히, 수단은 트레이너 것(§2 관통 원칙).

### ① 프롬프트 확정본 (route.js `firstPrompt` — 톤 4개 반영 · **haiku 확정**)

> **상태:** ✅ **확정.** haiku(`claude-haiku-4-5-20251001`) 실측으로 톤 3종 통과 —
> ① `why_higher_odds` **공감 톤**('등록 확률'이 아니라 회원이 '내 문제 해결로' 느끼게),
> ② `watch_for` **미루기 → 2차 연결**('포기'가 아니라 '전략적 선택', 다음 접점의 씨앗),
> ③ **MBTI 공란 무언급** + 직업·나이 사실 기반 엮음. → **haiku로 확정, sonnet 불필요.**
> 아래가 route.js에 박을 **확정본 전문**이다. 반영된 "톤 4개 편집":
> (a) 정체성 프리앰블(성장 엔진·발판·공감>압박) (b) `why_higher_odds` '회원 위함' 방향 재정의
> (c) `watch_for` 미루기=2차 다리 (d) `soft_closing.logic` '오늘 클로징 없이 다음 약속만'도 정답.

**firstPrompt(member) (user, `${...}`는 회원 기본정보 주입):**

```text
[상황] 1차 OT 전/중. 아직 관찰 데이터가 없다 — 아래는 관찰이 아니라 '사전 정보(회원 기본정보)'다.
모든 판단은 '가설'임을 전제한다("~일 가능성" 톤). 없는 관찰·수치·에피소드를 지어내지 않는다.
[회원 기본정보] name=${name}, age=${age}, job=${job}, residence=${residence},
               mbti=${mbti}, pain=${pain}, goal=${goal}, machines=${machines}

[이 도구의 정체성 — 모든 출력에 관통]
이건 '세일즈 도구'가 아니라 '트레이너 성장 엔진'이다. 아래 출력은 정답 대본이 아니라 '발판'이다.
목적(무엇을 이루려는가)은 명확히 제시하되(밍밍한 방치 금지), 수단은 강요하지 않는다(획일 금지).
톤 원칙(모든 방향·예시에 적용): "목적은 이것, 예를 들면 이렇게, 본인 방식 있으면 그대로 쓰세요."
특히 세일즈·클로징은 압박이 아니라 '이 트레이너가 나를 정확히 이해했다'는 공감에서 나온다(PREAMBLE
상속). 회원을 '설득 대상'이 아니라 '이해할 사람'으로 다뤄라.

기본정보만으로 1차 OT를 처음부터 끝까지 길잡이하라. 6단계 흐름(arc)이 척추다.

[arc — 1차 OT 6단계 흐름] 각 비트: when / intent(왜) / direction(무엇을, 참고 방향) /
example(예시 문장 — 발판, 낭독기 아님) / tone(말투).
 1) 라포: 회원 정보(직업·거주·나이 등)로 말문 열 첫 화제·라포 팁. 힌트로(대본 아님).
 2) 문진·공감: 목적·불편함·'왜'·수술이력을 끌어내는 흐름. 인바디 측정을 이 상담 안에 자연스럽게
    녹인다(흐름 안내만) — 인바디 '수치의 해석·설명'은 생성하지 말 것(트레이너 몫).
 3) 자각(핵심): 목적은 '회원이 스스로 문제를 자각'하는 것(고정). 수단은 트레이너 자유다.
    자각을 유도할 '평가 동작'을 어디까지나 '예시(참고)'로 제시하라(예: 오버헤드 스쿼트 등) —
    "이렇게 시켜 이런 반응이 나오면 회원이 자각"까지. 단, FMS 등 평가도구의 정밀 채점 기준·세부
    점수는 지어내지 말 것(가리키기만). "본인 평가 방식이 있으면 그대로 쓰라"는 여지를 남겨라.
 4) 치트키 운동: 아래 movement_cues와 연결(평가동작=진단, 치트키=해결의 짝).
 5) 수업 피드백: 오늘 몸에서 일어난 것을 되짚는 흐름.
 6) 세일즈: 아래 closing_compass와 연결.

[movement_cues — 치트키 1개] 자각에서 드러난 문제를 즉효로 체감시키는 치트키 '1개만'.
숫자 처방(세트·횟수·각도) 금지, 움직임 '방향'까지만.
 - exercise: 치트키 동작(방향까지).  - why_instant: 회원이 그 자리에서 '되네?'를 느끼는 이유.
 - cueing: 현장 큐잉(예시 — 흐림/발판).  - dialogue: 함께 건넬 대화 결(예시 — 흐림/발판).
 - connects_to_closing: 이 몸의 경험이 어떤 클로징 논리로 이어지나.
 - principle: 이 동작이 통하는 '원리'(트레이너가 같은 원리로 자기 필살기를 응용하도록).
   ※ 운동 영상 링크(URL)는 절대 생성 금지 — 원리 설명으로 대체.

[closing_compass — 세일즈 나침반] 이 방향으로 공감하면 회원이 등록을 '압박'이 아니라 '내 문제
해결'로 느끼게 되는 나침반이다. (등록은 목표가 아니라, 회원이 깊이 이해받았을 때 따라오는
결과다.) 압박 플레이북이 아니다.
 - approach_tag: 회원 목적 축(pain=불편함개선 / appearance=외형 / value=가치 / other).
 - why_higher_odds: '이 방향으로 공감하면 회원이 등록을 압박이 아니라 내 문제 해결로 느끼게
   되는 이유'. 회원의 목적·불편함·'직업'·'나이'를 사실 기반으로 엮어라(좌식→어깨·거북목,
   40대→건강·회복). MBTI는 '있을 때만' 가설 보조("~성향이면 ~할 수 있다"), 단정 금지, 비어
   있으면 언급 금지. ※'등록 확률을 올리는'(회원 대상) 방향이 아니라 회원이 '해결로 느끼게'
   하는(회원 위함) 방향을 써라 — 등록은 이해받았을 때 따라오는 결과다.
 - watch_for: 이 방향이 '안 먹힐' 신호 = 밀어붙이지 말라는 브레이크(안전핀). 신뢰가 아직이면
   오늘 세일즈를 미루고 라포로 돌아가는 게 정답이다.
   ★ 단, 미루기는 '포기'가 아니라 2차로 넘기는 '전략적 선택'임을 반드시 담아라:
     · 오늘은 몸으로 느낀 경험을 남기고 '다음 접점'을 만든다(다음 약속의 씨앗:
       "오늘 느낀 이 감각이 다음에 어떻게 발전하는지 보여드릴게요" 식).
     · 관찰한 것을 '관찰기록'에 남기면 → 2차 지원 AI가 그 재료로 더 강한 클로징을 준비한다.
     · 1차에서 무리하게 미는 것보다 신뢰 쌓고 2차에서 제대로 클로징하는 게 등록률이 높다
       (1차 미루기 = '보류' = 2차 클로징 대상 — 미루기가 1차와 2차를 잇는 다리다).

[soft_closing — 부담 없는 1차 마무리] approach_tag / logic / example(예시·흐림).
 - logic: 강요·긴급성 금지. 미지근하거나 신뢰가 덜 쌓였으면 '오늘은 클로징 없이 다음 약속만'도
   정답임을 담아라(위 watch_for 미루기 연결과 일관되게).

[data_gaps — 성장 프레임] 지금 정보로 위 전부를 '반드시' 생성하라("정보 부족으로 못 한다" 금지).
data_gaps는 결핍이 아니라 '더하면 좋아지는 것'이다("○○를 관찰해오시면 △△까지 짚어드릴 수 있어요"
형태의 긍정 코칭). 기본정보가 충실하면 빈 배열 또는 1개 이하로(억지로 채우지 말 것).

[출력 언어·형식] 모든 텍스트는 자연스러운 한국어. 영문 코드값(pain, appearance 등)·필드명을 출력에
노출 금지. 반드시 아래 JSON 스키마만 출력. 설명·마크다운·코드펜스 금지.
{
  "data_gaps": ["..."],
  "hypothesis": "회원 기본정보 기반 접근 가설 (관찰 아님, '~일 가능성' 톤)",
  "observe_targets": ["자각(평가 동작)에서 1차에 확인해올 것 (관찰 단위)"],
  "arc": [ { "when": "라포|문진·공감|자각|치트키 운동|수업 피드백|세일즈", "intent": "...", "direction": "...", "example": "...", "tone": "..." } ],
  "movement_cues": { "exercise": "...", "why_instant": "...", "cueing": "...", "dialogue": "...", "connects_to_closing": "...", "principle": "..." },
  "closing_compass": { "approach_tag": "pain|appearance|value|other", "why_higher_odds": "...", "watch_for": "..." },
  "soft_closing": { "approach_tag": "pain|appearance|value|other", "logic": "...", "example": "..." }
}
```

> **확정본 스키마 = 핵심 필드만.** 위 §6 청사진 JSON의 **[단계적]** 하위 필드
> (`closing_compass.lukewarm_response` · `closing_compass.package_suggestion` ·
> `approach_tag`의 `machine_efficiency`)는 **확정본엔 아직 없다** — 실사용 튜닝 때 얹는다(§0.2).
> **실측 하네스:** `scripts/test-first-prompt.mjs`(리포 제외, `.gitignore`) — 이 확정본과 동일한
> PREAMBLE·firstPrompt를 담아 톤 바꿀 때마다 재실측(`node --env-file=.env.local scripts/test-first-prompt.mjs`).
> 출력이 ~5.5k 토큰이라 실측 시 `max_tokens`는 넉넉히(8192) — 4096이면 잘림.

## 9. 구현 위험 (회귀 지점)

- **㉠ 헤더 타이머 컨트롤이 `PHASES`/`active`와 얽힘.** 헤더의 `tab===1` 블록(경과·재생/일시정지)은
  `active`·`elapsed`·`running`·`overtime`에 의존한다. 타이머를 제거하면 **헤더도 함께 손봐야**
  컴파일·렌더가 깨지지 않는다(은퇴 종속 작업).
- **㉡ 환각 고위험 지점(§0 생존 조건 ② 직결).** 프롬프트가 반드시 막아야 할 두 곳:
  - **FMS·평가동작:** 가리키기만, **채점 기준 세부 지어내기 금지**(§2 자각 가드).
  - **PT 패키지 금액:** **지어내기 금지**, `PACKAGES`(센터 등록 패키지)에서만(§6 C).
  - PREAMBLE의 환각 금지를 상속하되, 이 두 지점은 프롬프트 본문에서 **명시적으로** 재차 못 박는다.
- *(직전 초안의 round-0 관련 위험 ㉠(report 공존)·㉢(upsert 시퀀싱)은 round-0 폐기로 소멸.)*
