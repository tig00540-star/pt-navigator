## 1. 다음 = ③ Phase 2 — 회원앱 종목별 무게 그래프

**무엇:** Phase 1로 쌓이는 `sets_structured`를 **회원앱(`/m`)에 "종목별 무게 증가 그래프"**로 노출. "벤치프레스 20 → 30kg" 성과 가시화 = **재등록 유도 강력**.

**왜 지금:** 재료 다 준비됨 — 구조화 데이터가 이미 저장 중이고, `buildExerciseSeries`·`Sparkline`이 공용 헬퍼로 나와 있음(회원앱에서 그대로 재사용). Phase 1로 트레이너가 추출 정확도 검증도 끝.

**이미 있는 것(재료):**
- `daily_workout_log.sets_structured`(jsonb) — 실데이터 축적 중.
- `lib/workout.js` `buildExerciseSeries(logs)` + `components/ui/Sparkline.jsx` — 회원앱에서 import만 하면 됨.
- 회원 안전뷰 `member_workout_log`(현재 `id·created_at·ai_summary`) + `app/m/[token]/page.jsx`(인바디 추이 섹션이 이미 Sparkline/delta 패턴으로 동작 = 그래프 붙일 자리).
- 회원앱 규약(§2) 확립: **definer 안전뷰만·민감컬럼 제외.**

**새로 필요한 것:**
- **안전뷰에 `sets_structured` 노출** — `member_workout_log` 뷰에 컬럼 추가(숫자만이라 안전) 또는 전용 unnest 뷰. 기록본 `docs/migrations/2026-07-1x-member-exercise.sql`.
- **회원앱 그래프 섹션** — `/m` 페이지 "인바디 변화" 아래 "종목별 무게 변화". `buildExerciseSeries(logs.map(l => ({ ...l, session_at: l.created_at })))`(뷰가 created_at만 주므로 매핑). 회원용 톤(글씨 크게·정보밀도 낮게).

**착수 전 정할 것(웹Claude가 스펙 전에 물어볼 것):**
1. **뷰 방식** — `member_workout_log`에 `sets_structured` 컬럼만 추가(간단·클라 집계) vs 서버에서 종목·날짜·무게로 펼친 전용 뷰(`member_exercise_series`). 잠정 추천 = **컬럼 추가**(공용 헬퍼 재사용·가벼움).
2. **그래프 지표** — Phase 1과 동일하게 **최고중량(topWeight)** 그대로 vs 볼륨(Σwt×reps)·추정1RM 추가. 잠정 추천 = **최고중량 유지**(직관적·트레이너 화면과 일관).
3. **노출 범위** — 전 종목 vs 대표 몇 개(회원 화면 과밀 방지). 잠정 추천 = 무게 데이터 있는 전 종목, 최근활동순(그리드).
4. **빈 데이터 회원** — 구조화 기록 0인 회원은 섹션 숨김/빈상태 문구 규약(노쇼·손입력 미기재만 있는 경우 포함).
5. **회원 표시 문구** — 성과·격려 톤("무게가 이만큼 올랐어요") 넣을지 vs 숫자만.

**참고 문서:** `docs/v2-스펙-수업일지-구조화-구현.md` §4(Phase 2 아웃라인) · §2(데이터 계약) · `docs/pt-navigator-총정리.md` §4(스키마).

## 2. 회원 앱 규약 (S1~S3에서 확립 · 그래프 붙일 때 준수)

- **회원 데이터 노출 = 안전 뷰만**(원시 테이블 회원 정책 없음). 새 회원 화면은 definer 뷰로만, 민감 컬럼 제외. Supabase "security definer view" 경고는 의도된 설계(무시). `sets_structured`는 숫자만이라 노출 안전(단 `raw_voice_text`·`note` 등은 계속 제외).
- **격리:** 회원 auth 유저는 trainer 행 없음 → `auth_account_id()`/`auth_is_owner()`=NULL → 매출·타회원·트레이너 데이터 0행. 회원은 `auth_member_id()` 기반만.
- **회원 라우트=`/m`**, 회원 클라 `lib/memberSupabase.js`(storageKey 분리).
