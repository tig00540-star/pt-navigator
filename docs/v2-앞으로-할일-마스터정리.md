# pt-navigator — 앞으로 할 일 마스터 정리

> 작성 기준: 회원앱 S1·S2·S3(발급·PT종료) 완결 시점. 흩어진 계획(MASTERPLAN·로드맵·백로그·미착수 설계·각 핸드오프 미결)을 한 문서로 종합.
> 성격: 살아있는 백로그(=우선순위는 바뀔 수 있음). "무엇을 / 왜 / 상태 / 선행"만 짧게. 구현 스펙은 착수 시 따로.

---

## 0. 지금 위치 (이미 완결 — 참고용)

- **회원 앱 v1** — S1(인증·RLS)·S2(홈·열람·안전뷰)·S3(링크 발급·복사·PT 종료 완전차단). 격리·컬럼누출까지 닫음.
- **OT 여정 코어** — 1차 강화·2차 등록당위성 브리핑·재등록·급한불 = 실 AI(`ot-brief`). 음성일지 실 STT+요약.
- **OT/PT 뷰 분리** — `member_status`/`viewFor` 뷰 스위치, 클로징 결과 기록, 수동 PT 확정.
- **로그인·격리** — Supabase Auth·`account`/`trainer`·RLS 헬퍼(`auth_account_id`/`auth_is_owner`). solo(1인) 모드.
- **PT 관리·정산** — `session_log` 계약·수업차감·급여/매출/클로징 집계·스케줄 보드·인바디 추이.
- **디자인·앱화** — 레드 브랜드 통일·`ui/` 프리미티브·하단 탭바·PWA.

> 아래는 "앞으로"만. 위 완결분은 손대지 않는다는 전제.

## 2. 수익화 B계열 — "돈 받는 스위치" (지금 우선순위 1순위 후보)

> 근거: `v2-설계-구독등급+접근통제+카카오로그인.md`. S3로 "프리미엄으로 팔 물건(회원관리)"이 완성됐으니 이제 스위치를 깐다. 두 층 잠금(층1=결제활성=앱사용 / 층2=premium=회원관리).

- **B1 — 구독 스키마 + 층1 잠금(수동)** [중]
  `account`에 `subscription_status`·`plan` 딱지 + `auth_account_id()`에 subscription AND + **기존 계정 전부 active 백필**(안 하면 본인 잠김) + `my_account_status()`(결제벽/온보딩 분기) + 화면 분기. 활성은 처음엔 **수동 스위치**. ★광범위 잠금이라 단계 배포(리트머스 1계정 → 전체).
- **B2 — 프리미엄 게이트(층2)** [소] · 선행 B1 + 회원앱S3
  `auth_account_plan()` + 회원 링크발급(S3 `issue()`)·`member-auth`에 premium 체크 + UI 🔒. S3 발급이 유일 교체점이라 자연스럽게 얹힘.
- **B3 — 카카오 로그인(트레이너)** [중] · 나머지와 독립
  Supabase Kakao 설정 + 로그인 버튼 + **온보딩 프로비저닝 라우트**(OAuth는 가입 트리거 안 걸림 → 이메일 경로와 통일). 신규=inactive로 층1 연결.
- **B4 — 결제 자동 활성** [상] · 선행 B1
  결제업체(토스/포트원 등) + `billing-webhook`(서명검증) → subscription/plan 자동 전환. B1의 수동 스위치를 대체.

> 추천 순서: **B1 → B2 → (B4 또는 B3 급한 것)**. 결제업체 선정은 B4 때.

## 3. 회원 앱 심화 (S3 이후)

- **회원 자가입력** [중] — 비포애프터 사진 업로드(스토리지) → 유산소·개인운동 체크. 뷰가 아니라 회원 INSERT가 생기므로 **회원 쓰기 정책·검증 별도 설계**.
- **회원앱 AI 기능** [상] — 식단 AI·루틴 AI 등(무거운 것 마지막).
- **★AI 게이트(비용 통제)** [중] · AI 기능과 함께
  AI 기능은 "활성 PT(**남은 세션>0**)일 때만" 열리게 게이트. → 남은 0이면 AI만 자동 잠금(비용 누수 차단), **열람(성과·히스토리)은 유지**(재등록 유도), 재등록해 남은>0 되면 자동 재개. account 2층 잠금과 같은 원리를 회원 기능 레벨에. (S3 스펙 §10에서 확정한 방향 — 앱 전체를 끄지 않고 돈 드는 것만 잠금.)

## 4. OT 여정 완성 (제품 세로 슬라이스 · roadmap-next 잔여)

> ①1차강화·②뷰분리·⑦로그인RLS는 완료. ③PT관리 뷰도 대체로 실데이터. 남은 것:

- **④ 클로징 통계 대시보드** [중] — 트레이너 본인 클로징률 + 나이·성별·직업·`closing_approach`별 강점 분석. 관리자 "행동 가능한 목표 대시보드"(오늘 재등록 대상→예상 매출)와 연결. 원천 데이터(`closing_result`/`approach`)는 이미 축적 중.
- **⑤ 치트키 운동 추천** [중] — 1차(즉각 피드백 동작)·PT(세션 흐름 추천)에 **대칭**으로. **방향까지만**(숫자 처방 금지). 독립 스프린트 아니라 각 뷰에 얹음.
- **남은 데모 → 실데이터** — 재등록 CRM '정적 대본' 일부 · admin **QC 섹션**(TRAINERS 하드코딩, 행동 계측 선행) · admin **카피봇** 문장 템플릿 · 회원 성향요약 Hero(김철수만 하드코딩).

## 5. 큰 UX/AI 재편 (미착수 설계 2건 · 방향은 합의됨, 스펙 미착수)

- **OT 여정 '전/후' 재편** [중~대] — `v2-설계-OT여정-전후재편-제안.md`
  2차·재등록이 정보 과부하("한눈에 안 들어옴") → **3층 셸**(L0 지금 한마디 / L1 핵심 3카드 / L2 접힘 상세) + **2차 실시간 분기 선택기**(잘옴/약함/아직 중 하나만 크게). 공통 컴포넌트(`BriefShell`·`NowLine`·`KeyCards`·`BranchPicker`·`ResultForm`) 먼저 → 2차 → 재등록 → 1차 → 1차후. **1단계=렌더 3층+분량상한**(스키마 유지·캐시 호환), 2단계=필드 축소는 나중. → 착수 전 §0 전제(A~E) 답 필요.
- **자기진화 루프(AI 맞춤 세일즈 선생님)** [중~대] — `v2-설계-자기진화루프-트레이너맞춤세일즈선생님.md`
  트레이너 과거 결과·피드백을 구조화 저장 → 브리핑마다 few-shot 주입(인컨텍스트 학습, 재학습 아님). 조각: ⓐ1차 `to_capture`(2차 대비 얻어올 정보) ⓑ1차 실패원인 분석 ⓒ`app_followed`(앱대로/변형/내방식) 필드 ⓓ트레이너 브리핑 피드백 캡처 ⓔD-3 케이스 주입을 재등록까지 확장. 절반은 이미 뼈대 존재(`closing_*`·`closing_profile`·D-3). PII 최소화(익명 프로파일만) 필수. → 착수 전 4개 확정(학습방식·app_followed값·피드백항목·PII수준).

## 6. 미완 UI·기능 백로그 (총정리 §5.3)

- **회원 기본정보 수정 UI** — 등록은 있으나 수정 동선 확인/보강.
- **초대 메일(SMTP)** — 현재 트레이너 초대 임시비번 수동 전달 → 메일 자동화.
- **스케줄 확장** — 반복 예약·시간 변경·완료 취소.
- **pay_policy 편집 UI** — 원장이 급여 구간·인센 수정(현재 DB 직접인지 확인 · AdminPayrollSettings 있으면 완료 여부 점검).
- **오프보딩 UI** — 트레이너 비활성 + 접근 차단 흐름(§7 보안과 연결).
- **테스트 프레임워크** — 러너·테스트 파일 0 → 회귀 방어용 최소 세팅(선택).

## 7. 보안 하드닝 잔여 (총정리 §5.1 · 파일럿 규모선 급하지 않음)

- **trainer_id 스코프 강화** — `daily_workout_log`·`ot_log`·`inbody_log`가 계정 스코프만(트레이너별 격리 아님). 정상 UI엔 안 새나 직접 REST 방어 없음.
- **오프보딩 RLS** — 트레이너 `active=false`가 RLS 헬퍼에 반영되게(라우트 인증엔 이미 `active` 확인). B1의 subscription AND 넣을 때 함께 정리 가능.
- **service_role 키 노출 점검** — 빌드/로그/클라 번들에 안 새는지.
- **API 라우트 인증·레이트리밋** — `requireTrainer` 도입됨(voice-log 등). member-revoke 포함 전 라우트 커버 확인.
- **프롬프트 인젝션** — 회원/트레이너 입력이 프롬프트에 직삽입(내부도구라 위험 낮으나 점검 가치).

## 10. 추천 진행 순서 (내 판단 · 바꿔도 됨)

1. **1번 위생 잡일**(teardown·intake SQL 확인·git add) — 30분짜리, 지금 털기.
2. **B1(구독 스키마 + 층1 수동 잠금)** — 돈 받는 스위치. S3로 팔 물건이 생겼으니 지금이 적기. ★단계 배포 주의.
3. **B2(프리미엄 게이트)** — B1 위에 S3 발급을 premium 전용으로. 작고 자연스러움.
4. 이후 갈림길:
   - 매출 자동화 급하면 → **B4(결제 웹훅)** 또는 가입마찰 줄이려면 **B3(카카오)**.
   - 제품 깊이 급하면 → **④ 클로징 통계** 또는 **OT 전/후 재편**(현장 "한눈에" 불편 해소).
   - 회원앱 차별화 급하면 → **회원 자가입력(사진)** → **AI 기능 + AI 게이트**.

> 큰 원칙(MASTERPLAN): 뼈대(데이터)부터 깔고 AI를 얹는다 · 데모/실연동 섞지 않기 · 보안·다듬기는 흐름 굳은 뒤. B계열은 광범위 잠금이라 항상 기존행 백필 + 단계 배포.
