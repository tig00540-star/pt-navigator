## 1. 한 줄 정의

트레이너가 현장에서 **AI가 하달하는 OT 세일즈 대본 · 맞춤 루틴**으로 수업하고, 관리자는 **지점 매출과 트레이너 QC를 한눈에 감시**하며, 회원은 **음성 AI 일지**를 카톡으로 받는 — Next.js + Tailwind + Supabase 기반의 피트니스 세일즈 플랫폼.

### 트레이너 화면 — 5개 탭 (app/page.jsx)

| 탭 | 이름 | 핵심 기능 |
|---|---|---|
| 0 | 회원 | Supabase 회원 목록(카드) · 검색 · 신규 등록 모달 · 탭하면 회원 선택 |

| 2 | 2차 OT | 피드백 AI 분석 · 둔근 100% 조정팁 · 2차 루틴 · 2차 세일즈 흐름 |
| 3 | 재등록 CRM | 리스크&기회 분석 · 재등록 타이밍 판정(슬라이더) · 세일즈 방향·흐름 |
| 4 | 음성일지 | 🎤 녹음 → AI 요약 → 📋 카톡용 복사 → DB 저장 |

## 4. 데이터베이스 구조도 (Supabase / PostgreSQL)

**user_table** — 회원 정보
```

created_at   timestamptz
name         text (필수)
age          int
job          text
residence    text          -- 거주지
mbti         text
pain         text          -- 불편 부위
goal         text          -- 목적
machines     text[]        -- 사용 머신
phone_number text          -- 카톡 전송용 (음성일지 기능)
```

**center_machine** — 센터 보유 머신
```
id, created_at, brand, name
```

**ot_log** — OT 리포트 (초기 설계, 현재 미사용)
```
id, created_at, user_id(FK), ot_round, report(jsonb), note
```

**daily_workout_log** — 음성 AI 일지
```
id           uuid (PK)
created_at   timestamptz
user_id      uuid (FK → user_table)
raw_voice_text  text       -- STT 원본 (v2에서 실제 채움)
ai_summary   text          -- AI 요약 리포트 본문
```

### 보안 (RLS) — ⚠️ 현재 프로토타입 상태

모든 테이블이 **anon 키로 읽기/쓰기 개방**되어 있음. 주소·키만 알면 누구나 접근 가능.
→ **v2에서 반드시 로그인 기반으로 잠글 것.** (특히 매출이 보이는 /admin, 회원 개인정보)

## 7. v2 작업 지도 (우선순위)

### 🥇 1순위 — 코드 구조 리팩터링
`app/page.jsx` 단일 파일이 2,500줄 이상으로 비대함. 기능 추가 시 반복적으로 깨짐.
→ 탭별 컴포넌트를 별도 파일로 분리 (`components/tabs/`, `components/ui/`), 데이터 상수 분리 (`data/`), 커스텀 훅 분리.

### 🥈 2순위 — 중복 기능 통합
- 재등록 CRM ↔ 음성일지 ↔ 회원 상태: 겹치는 부분 정리.
- v1에서 보류한 **회원 라이프사이클 상태(OT/PT 구분)** 를 이 단계에서 제대로 재설계해 통합.

### 🥉 3순위 — 진짜 데이터화
- 결제/세션 테이블 신설 → 관리자 KPI를 실데이터로.
- 실제 AI 연동: OT 대본 생성 · 음성 STT · 일지 요약 (Gemini/Claude API + 서버 함수).

### ⭐ 세일즈 심리 설계 (제품의 영혼 — v2에서 핵심 반영)
트레이너의 세일즈 저항은 스킬이 아니라 **정체성 충돌**에서 온다. "나는 파는 사람이 아니다 / 거절이 두렵다." 제품은 파는 법이 아니라 **"세일즈 = 회원을 책임지는 행위"라는 재정의**를 제공해 세일즈를 트레이너 본질과 화해시킨다. 4가지 설계 원칙:
1. **회원 목표에서 출발하는 대화** — "할인해요"(파는 말)가 아니라 "지금 멈추면 원점"(책임의 말).
2. **거절을 데이터로** — "예산 때문에 미룸 → 3개월 뒤 재접근"으로 CRM 기록. 거절이 관계의 끝이 아닌 과정의 한 점.
3. **제안 안 한 비용 보여주기** — 제안 실패의 두려움보다 방치의 손실이 크다는 인식.
4. **작은 성공으로 거절 내성** — 클로징을 잘게 쪼개(오늘 결과 한 줄 브리핑) 성공 경험 축적.
→ 모든 세일즈 대본·CRM 문구를 이 4원칙에 맞춰 재작성.

**추가 기능 A — 거절 사유별 대응 (Objection Handling)**
거절 순간 말문 막히는 게 트레이너 공포의 핵심. 사유별로 "반박"이 아니라 "회원의 걱정 해소" 방향을 제시:
- 💰 금전 부담 → 할인이 아니라 회당 가치 환기 + "지금 멈추면 원점"(손실 회피) + 최소 필요량 옵션.
- ⏰ 시간 부족 → 빈도 줄인 플랜(주1회+홈트) + "바쁠수록 혼자 하면 부상 위험" 재프레이밍.
- 📅 스케줄 → 대개 진짜 이유를 숨긴 핑계. 유연한 시간대 + "진짜 장애물"을 드러내는 질문 유도.
- 🤔 보류("생각해볼게요") → 밀어붙이지 말고 "무엇이 가장 고민되세요?"로 진짜 이유 파악 → 재접근 타이밍 CRM 기록.
> 원칙: "이렇게 반박하세요"가 아니라 "이 걱정을 이렇게 풀어주세요"로 제시(죄책감 없는 세일즈).

**추가 기능 B — PT 등록 당위성 자동 브리핑 (1차 피드백 기반)**
2차 OT 클로징 직전, "왜 이 회원이 등록해야 하는지"를 1차 데이터 기반으로 정리해 트레이너에게 브리핑. 감이 아니라 논리·근거로 클로징. 예:
- 1차에서 확인된 것(통증·체형 패턴) → 혼자 하면 악화될 위험 → 2차에서 증명된 것(정확한 세팅 시 통증 없이 자극) → 클로징 논리("혼자서는 이 세팅을 못 잡습니다. N주는 함께 잡아야 몸이 기억합니다").
- "설득이 아니라 증명"의 완성형. 1차 데이터 → 2차 피드백 → 클로징 논리의 자동 연결. (1차 피드백 실저장·연동이 선행 조건.)

### ⭐ 단계별 데이터 파이프라인 (제품 흐름의 빠진 연결고리 — v2 핵심)
"정보를 앞단에서 최대한 모으고, 단계마다 쌓아 다음 단계의 방향을 만든다." 정보를 미리 알수록 대화가 맞춤이 되어 세일즈가 쉬워진다.

```
[등록/데스크] 기본정보(+인바디 수동입력) 수집
        ↓
[1차 OT] 트레이너가 현장 관찰 정보 추가 기입 → DB 저장
        ↓
   ★ AI 종합 분석 → 2차 OT 진행 방향 자동 설정 ★
        ↓
[2차 OT] 방향대로 '증명' 수업 → 등록 당위성 브리핑 → 클로징
```

구성 요소와 난이도:
| 기능 | 난이도 | 시점 |
|---|---|---|
| 데스크 정보 수집 (기존 등록 폼 확장, 데스크 직원용 활용) | 낮음 | v2 초반 |
| 1차 OT 관찰 정보 기입 → DB 저장 (하드코딩 제거) | 중간 | v2 핵심 배관 |
| AI 종합 분석 → 2차 방향 자동 설정 | 높음(실 AI 필요) | v2 후반 |
| 인바디 수동 입력 기반 방향 설정 | 중간 | v2 |
| 인바디 자동 연동(API) | 매우 높음(협약) | v3 |

핵심 원칙: **수동 입력 + DB 저장 뼈대를 v2에서 먼저 깔고**, 그 위에 AI 자동 분석과 인바디 자동화를 얹는다. 뼈대(데이터) 없이 AI부터 붙이면 분석할 게 없어 헛돈다. 삼각 편대에 '데스크' 접점이 추가되는 셈이나, 별도 화면이 아니라 기존 등록 폼의 활용처를 넓히는 방식이라 부담이 적다.

### ⭐⭐ AI 활용 철학 — "정답 제공"이 아니라 "스파링 파트너"
상향평준화 철학의 AI 버전. AI가 방향을 뚝딱 정해주면 트레이너는 사고를 멈추고 의존하며 하향평준화된다. 그래서 순서가 결정적이다:

```
[회원] 등록 시 인바디+기본정보를 '하나의 양식'에 직접 작성
        ↓
[데스크] 파일 철에 끼워 배정 트레이너에게 전달 (아날로그 그대로 OK — 협약 없이 인바디 활용하는 우회로)
        ↓
[트레이너] 종이를 보며 앱에 직접 입력 ← 입력하는 행위 자체가 회원 이해·방향 고민의 시간(인지적 각인)
        ↓
[AI] 트레이너 입력 기반으로 피드백 제시
        ↓
[트레이너] 내 생각 vs AI 비교 → 방향 확정 (성장의 순간)
```

- **왜 '직접 입력'인가:** 자동으로 받으면 훑고 잊지만, 손으로 입력하면 "이 회원은 허리가 안 좋고 좌식이니…"를 자연히 생각하게 된다. 입력이 곧 사고 과정. 자동화가 이 소중한 과정을 없애서는 안 된다.
- **개발적 강점:** ①종이→앱 하이브리드로 인바디 협약 없이 즉시 도입 ②AI가 0에서 창조가 아니라 입력에 '피드백'이라 헛소리 확률↓, 프롬프트 설계 쉬움 ③회원 작성 양식 = 데이터 표준 템플릿의 출발점(→디지털 폼→인바디 자동연동으로 진화).
- **균형점(현장 검증):** 트레이너가 직접 입력을 귀찮아하지 않도록 — 회원이 종이에 미리 다 써오게 해 트레이너는 빠르게 옮겨적기만. 입력 항목은 '사고를 유발하는 핵심'만, 나머지는 간소화. "생각하게 하는 입력"은 살리되 입력 자체는 가볍게.
- **한 줄 원칙:** AI는 정답을 주는 도구가 아니라, 트레이너가 먼저 고민한 뒤 자기 생각을 비교·검증하는 스파링 파트너다.

### v1.5 철학 반영 수정 리스트 (v2 리팩터링 시 함께)
> "방향과 원리(왜)는 주되, 구체적 실행 레시피(중량·세트·각도)는 빼거나 '예시'로 낮춘다."
1. **[1순위] 2차 OT 루틴** — `15kg·15회·4세트`, `상체 15°·템포 2-1-3` 같은 정확한 처방 → "둔근 우세 자극이 목표 / 대퇴사두 개입 차단 방향" 등 방향+원리로.
2. **[2순위] 1차 OT 루틴·스트레칭** — 세트 수치 → 방향 위주로.
3. **[유지] 음성일지 AI 요약의 수치** — 트레이너가 실제 말한 걸 정리하는 자리라 수치 OK.
4. **[4순위] 관리자 QC 티칭 지표 추가** — 통증 데이터 확인율·맞춤 루틴 적용률·재등록율(하향평준화 방지).
5. **[5순위] "왜(원리)" 강조** — 방향에 항상 원리를 붙여 트레이너의 공부를 유도.
