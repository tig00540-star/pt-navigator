# v2 설계 — 자기진화 루프 (AI가 트레이너 맞춤 세일즈 선생님으로)

> 성격: 방향 합의용 설계 제안(구현 스펙 아님). "가능한가?"에 대한 답 + 요청 매핑(이미있음/확장/신규) + '학습'의 실체 + 보안 + 순서.
> 전제: 1차 사전무장 4블록은 이미 배포. 이 루프는 그 위에 얹는다.

---

## 0. 결론 먼저

**가능하다. 절반 이상이 이미 뼈대로 존재한다.**
- 이미: 클로징 결과·방향·사유·3박자(접근/반응/결과)·**익명 프로파일** 기록(`ot_log` round2 `closing_*`·`closing_detail`·`closing_profile`) + **D-3 "내 과거 케이스 거울"**(`closingCasesForTrainer`·`closingCaseGate`·`closingSuccessCount`) = 자기진화 엔진의 씨앗. 재등록도 `reg_result`/`reg_reason` 기록 중.
- 신규로 얹을 것: ⓐ1차 전에 '2차 대비 얻어올 정보' 예고 ⓑ1차 후 실패원인 분석 → 2차 셋업 ⓒ'앱 방향대로 vs 개인 방법' 기록 필드 ⓓ트레이너 브리핑 피드백 캡처 ⓔ이 신호들을 다음 브리핑 프롬프트에 few-shot으로 배선 + 재등록까지 확장.

---

## 1. ★"AI가 스스로 학습·진화"의 실체 (기대치 정렬 — 제일 중요)

이 앱의 진화 = **모델 재학습(fine-tune)이 아니라, 트레이너의 과거 결과·피드백을 구조화 저장했다가 브리핑 생성 때마다 '맥락(few-shot)'으로 주입**하는 인컨텍스트 학습.
- **효과:** "이 트레이너에게 통한/막힌 접근"을 매번 근거로 → 출력이 점점 이 트레이너에 맞춰짐 = "나를 알아가는 세일즈 선생님."
- **왜 이게 맞나:** 즉시 반영(1건만 쌓여도 다음부터) · 저렴 · **트레이너별 격리 쉬움(보안)** · 되돌리기 쉬움. 가중치 재학습은 이 규모에 과하고 느리고 프라이버시 부담 큼 → 비추.
- **이미 D-3(case_feedback)가 정확히 이 구조.** 우리는 넓히고 다듬는다(재등록 확장 + '앱/개인 방법'·트레이너 피드백을 재료로 추가).

## 2. 루프 전체 그림

```
[1차 전]  사전무장 4블록  +  (신규) 2차 대비 '얻어올 정보' 예고
   │ 수업
[1차 클로징]  결과 + 방법(앱대로/변형/내 방식) 기록  →  (실패/보류면) 사유
   │
[1차 후]  (신규) 실패원인 분석 → 2차 방향 셋업 + 관찰 보강
   │
[2차 전]  1차 정보·실패분석 기반 → '방향'은 유지하되 클로징에 더 포커스(운동·세일즈 확률↑)
   │ 수업
[2차 클로징]  결과 + 방법 + 3박자 기록   (재등록 트라이도 동일 패턴)
   │
[축적]  결과·방법·트레이너 피드백 → 트레이너별 익명 코퍼스
   │
[다음 브리핑]  코퍼스를 few-shot 주입 → 점점 맞춤 (= 진화)
```

---

## 3. 요청별 매핑

| # | 트레이너 요청 | 상태 | 무엇을 |
|---|---|---|---|
| 1 | 1차 전에 '2차 대비 얻어올 회원 정보' 미리 알려주기 | **신규(작음)** | 1차 사전무장 출력에 `to_capture`(2차 클로징 대비 1차에서 확보하면 좋을 관찰·정보 2~3개) 블록 추가. *(초기 firstPrompt의 observe_targets를 이 목적으로 부활 — '관찰용'이 아니라 '2차 클로징 준비용'으로 재프레이밍.)* |
| 2 | 1차 클로징 실패 원인 분석 | **확장** | `closing_reason`(이미 저장) + `closing_detail` + 관찰을 근거로, 1차 후 '왜 안 됐나 + 2차에서 뒤집을 지점'을 AI가 정리(신규 phase 또는 2차 프롬프트에 주입). |
| 3 | 2차 전 = 1차 정보로 클로징 포커스 | **이미있음+튜닝** | 2차 브리핑이 이미 1차 관찰 기반. 여기에 (2)의 실패분석을 근거로 얹고, 프롬프트를 '방향+클로징 확률'로 더 기울임. 전/후 4블록 재편과 합류. |
| 4 | 클로징 시 '앱 방향대로 vs 개인 방법' 기록 | **신규 필드** | 결과기록 폼에 `app_followed`(followed 그대로 / adapted 참고+변형 / own 내 방식) + (선택)어느 제안을 썼나. → **진화의 핵심 신호**(앱 제안이 실제로 통했는지 측정). |
| 5 | 성공/실패 데이터 저장 → AI 자기진화 (1·2차·재등록) | **확장** | round2 closing_* 이미 저장. 재등록도 reg_* 저장 중. D-3 케이스 주입을 **재등록까지 확장** + `app_followed`·피드백을 few-shot 재료에 추가. |
| 6 | 트레이너 브리핑 피드백(아쉬움/보완/현실충돌) | **신규** | 브리핑 직후 1탭 피드백 캡처(자유서술 3~4항목 + 별점?) → 저장 → 다음 프롬프트에 "이 트레이너가 지난번 지적한 것" 맥락으로 반영. |
| 7 | 보안(회원·트레이너 PII 유출 방지) | **정책** | §4. |

## 4. 보안 / PII (이 루프의 필수 전제)

- **트레이너별 격리:** 코퍼스·피드백은 **그 트레이너의 프롬프트에만** 주입(RLS `trainer_id` 스코프). 트레이너 간 유출 0 — 남의 케이스가 내 브리핑에 안 섞임.
- **회원 PII 최소화:** 학습 코퍼스·프롬프트엔 **이름·전화 절대 제외**(현재 `closing_profile`이 이미 이름/전화 없음 ✓). `residence`·`mbti`는 세일즈에 꼭 필요치 않으면 **추상화(도시급) 또는 제외** 권장 — 익명 프로파일(나이대·직업군·목적·거절사유·통한 접근)만 남겨 '누구인지' 특정 불가하게.
- **LLM 제공자 전송:** 서버에서 **익명 프로파일만** 프롬프트에 실어 보냄. 트레이너 개인정보(이름·연락처 등)는 프롬프트에 안 들어감.
- **동의·표기:** 실회원 데이터 축적이므로, 배포 전 개인정보 수집·활용 동의(기존 배포 체크리스트 항목)와 정합.

---

## 5. 순서 (배포한 1차 위에 얹기 · 다스프린트)

1. **[완료]** 1차 전 4블록 사전무장.
2. **1차 전 `to_capture` 추가**(요청1) — 사전무장에 '2차 대비 얻어올 정보' 블록. 작은 프롬프트+렌더 확장.
3. **2차 전 4블록 + 클로징 포커스**(요청3) — 전/후 재편의 '2차 전'을 4블록 결로. 1차 관찰·(있으면)실패분석 주입.
4. **후 화면 통일**(요청2·4) — 결과기록 폼 + `app_followed` + (1차 실패 시)원인분석. 2차·재등록 공통 컴포넌트.
5. **트레이너 피드백 캡처**(요청6) — 브리핑 직후 1탭.
6. **진화 배선**(요청5) — D-3를 재등록까지 확장 + `app_followed`·피드백을 few-shot 재료로. (매니징 대시보드 = `closing_reason`·`app_followed` 집계로 "어떤 거절·어떤 방식이 매출 저하 핵심인지".)

> 각 단계는 독립 배포 가능. 2~3부터 현장 반응 보며 진행. 지금 배포한 1차의 폰 스모크가 먼저.

---

## 6. 확정하면 좋을 것 (내 추천 포함)

- **학습 방식 = 인컨텍스트(§1)** — 확인만.
- **`app_followed` 값** — 추천: `followed`(제안 그대로) / `adapted`(참고+변형) / `own`(내 방식) 3분류 + (선택) 어느 블록을 썼나(오프닝/타겟운동/비유/거절대응). → 진화 신호의 해상도.
- **피드백 캡처 항목** — 추천: 아쉬움 · 보완했으면 하는 점 · 현실충돌(실제 진행 중 안 맞았던 것) 3개 자유서술 + 5점 만족도. 브리핑 직후 접이식 1탭.
- **PII 최소화 수준** — 추천: 코퍼스에서 `residence`·`mbti` 제외(또는 도시급 추상화). 세일즈 근거엔 나이대·직업군·목적·거절사유·통한 접근이면 충분.

이 4개만 정해지면 요청1(1차 `to_capture`)부터 구현 스펙으로 내려갈 수 있음(작고, 이미 배포한 1차에 바로 얹힘).
