# v2 설계 — 회원 앱 MVP (아키텍처 · 인증 · 보안 · 스프린트)

> 성격: 방향 합의용 설계 제안(구현 스펙 아님). 회원 전용 앱 신설의 뼈대 — 인증·RLS·화면·순서.
> 확정: **인증 = 카톡 초대링크 + 휴대인증** · **v1 = 열람 우선 MVP**(내 수업일지·인바디 변화량 읽기전용).
> ★대전제(보안): 회원은 **'내 데이터만'** 본다. 매출·급여·타 회원·트레이너 화면은 **절대 접근 불가**.

---

## 0. 큰 그림

지금 앱 = 트레이너/관리자만 로그인(Supabase Auth). 회원(`user_table`)은 **로그인이 없음.** 회원 앱을 열려면 세 가지가 새로 필요:
1. **회원 인증**(카톡 링크 + 휴대 확인 → 회원 세션),
2. **회원 스코프 RLS**(회원 세션이 자기 행만 읽게),
3. **회원 전용 라우트/화면**(읽기전용 홈).

기존 데이터(트레이너가 쓴 수업일지 `daily_workout_log`, 수기 인바디 `inbody_log`) 위에 **읽기전용**으로 얹는 거라 v1은 가볍다.

---

## 1. 인증 흐름 (카톡 링크 + 휴대인증)

```
[트레이너] 회원 카드 → '회원 링크 생성·복사' → 카톡으로 전송(기존 카톡 흐름 그대로)
      │  링크 = /m/<member_token>  (긴 랜덤 토큰)
[회원]  링크 열기 → '휴대폰 뒷 4자리' 입력(본인 확인)
      │
[서버]  member-auth 라우트: token + 휴대끝4자리를 user_table와 대조 → 일치하면 회원 세션 발급
      │
[회원]  '내 페이지' 입장 (이후 재방문은 세션 유지 · 만료 시 재확인)
```

- **왜 휴대 확인을 붙이나:** 링크만으로 열리면 링크 유출 = 남이 봄. 트레이너가 그 회원 번호로 보냈으니, **휴대 끝자리 일치**를 2차 확인으로 둬서 링크만으론 못 열게. (더 강하게 원하면 후속에 문자 OTP로 승급 — v1은 끝자리 대조로 충분.)
- **링크 폐기/재발급:** 트레이너가 토큰 재발급하면 이전 링크 무효(분실·양도 대비).

---

## 2. DB 변경 (마이그레이션)

- `user_table`에 컬럼 추가:
  - `member_token uuid`(초대 링크 토큰 · 트레이너 생성 · nullable),
  - `member_auth_id uuid`(회원이 최초 인증 시 연결되는 auth 유저 id · nullable).
- **회원 세션 메커니즘(내 추천):** 회원도 **실제 Supabase auth 유저**를 갖는다. member-auth 서버 라우트가 token+휴대 확인되면, 그 회원용 auth 유저를 (최초 1회) 만들고 `user_table.member_auth_id`에 연결 → 세션 발급. 이후 RLS는 `member_auth_id = auth.uid()`로 스코프.
  - 대안: 커스텀 JWT(member_id 클레임) 또는 서버 프록시(service_role) — 둘 다 가능하나, **실제 auth 유저**가 RLS·세션관리가 제일 깔끔해서 추천. (member-auth 라우트는 create-trainer처럼 service_role 서버전용.)
- **회원 스코프 RLS(신규):** SECURITY DEFINER 헬퍼 `auth_member_id()` = 세션 → 내 `user_table.id`. 정책:
  - `user_table` SELECT: `id = auth_member_id()` (내 행만),
  - `daily_workout_log` SELECT: `user_id = auth_member_id()` (내 수업일지만),
  - `inbody_log` SELECT: `user_id = auth_member_id()` (내 인바디만).
  - ★회원 세션은 트레이너 헬퍼(`auth_account_id`/`auth_is_owner`)가 **null** → 매출·급여·타 회원·account 스코프 데이터에 **원천 접근 불가**. (이중 방어: 회원 라우트/화면에서 그런 쿼리 자체를 안 함.)

---

## 3. 회원 화면 v1 (읽기전용 · 라우트 `/m`)

새 라우트 `app/m/`(트레이너 앱과 분리 · 회원 세션 전용). 홈 한 화면:
- **내 프로필 헤더:** 이름·목표·담당 트레이너(간단).
- **내 수업일지 타임라인:** `daily_workout_log`의 `ai_summary`(트레이너가 음성일지로 쓴 것)를 날짜순 카드로. → 요청 '운동일지 수집/열람' + '수업일지 연동' 해결.
- **내 인바디 변화량:** `inbody_log` 추이(체중·골격근·체지방 등 + 증감 색). → '인바디 변화량 파악(회원에게 안 물어봐도)' 해결.
- (후속 자리만 잡아둠: 비포애프터·식단·유산소·개인운동 체크 — v2에서 자가입력.)

톤·디자인: 트레이너 앱과 같은 디자인 시스템(레드 브랜드·Tailwind 토큰) 재사용. 회원용이라 더 심플·큰 글씨.

---

## 4. 트레이너 측 추가 (작음)

- 회원 카드/상세에 **'회원 링크 생성·복사'** 버튼 → `member_token` 생성(없으면) → `/m/<token>` 링크 클립보드 복사(기존 카톡 복사 패턴 재사용). '재발급'도.
- (선택) 링크 발급 여부 배지("회원앱 연결됨").

---

## 5. 보안 가드 (이 앱의 필수 전제)

- **격리 3중:** ① RLS(회원 세션은 내 행만) ② 회원 라우트는 트레이너/관리자 쿼리 자체를 안 함 ③ 트레이너 헬퍼가 회원 세션엔 null.
- **매출·급여·QC·타 회원 = 회원에게 절대 노출 금지**(RLS + 라우트 분리로 원천 차단).
- **service_role**은 member-auth 서버 라우트에서만(서버 전용·깃/클라 노출 금지 — create-trainer와 동일 원칙).
- **개인정보 동의**: 회원 앱에서 자기 데이터 열람 = 수집·활용 고지와 정합(배포 전 체크리스트).
- **링크 보안**: 긴 랜덤 토큰 + 휴대 끝자리 2차 확인 + 재발급으로 폐기. (민감도 더 높이려면 문자 OTP 승급 — 후속.)

---

## 6. 스프린트 분해 (하나씩 배포·검증)

1. **S1 — 인증·RLS 뼈대:** SQL(member_token·member_auth_id·회원 RLS·`auth_member_id()`) + `member-auth` 서버 라우트(token+휴대 확인→세션). 화면 없이 인증만 검증.
2. **S2 — 회원 홈(열람):** `app/m/` 라우트 + 홈(수업일지 타임라인·인바디 추이). 읽기전용.
3. **S3 — 트레이너 링크 발급 UI:** 회원 카드 '링크 생성·복사·재발급'.
4. **(이후) 자가입력:** 비포애프터 업로드(스토리지) → 유산소·개인운동 체크 → 식단AI·루틴AI(무거운 것 마지막).

> S1이 뼈대라 제일 조심스럽고(보안), S2·S3는 그 위에 얹으면 빠름. 셋 다 되면 "회원 앱 v1(열람)" 완성.

---

## 7. 남은 기술 결정 (1개 · 내 추천대로 가도 됨)

- **회원 세션 = 실제 Supabase auth 유저(추천)** vs 커스텀 JWT vs 서버 프록시. → §2대로 **실제 auth 유저** 권장(RLS·세션 제일 깔끔). 이견 없으면 이대로 S1 스펙 잡음.

이 설계 OK면 **S1(인증·RLS 뼈대)부터 구현 스펙**으로 내려갈게 — SQL + member-auth 라우트부터.
