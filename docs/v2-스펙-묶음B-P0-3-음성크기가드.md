## 0. 문제 (확인됨)
- `VoiceLogTab.jsx:27` 주석이 `10분 상한(25MB 방어)` — **OpenAI 25MB만 방어**. 실제 경로는 OpenAI 직행이 아니라 **Vercel 함수 경유**(`fetch("/api/voice-log")`)이고 **Vercel body 상한은 4.5MB**(플랜 무관 · 초과 시 함수 실행 **전에** 413 `FUNCTION_PAYLOAD_TOO_LARGE`). **[확인]**
- `audioBitsPerSecond` 미지정(`:224-227`) → 브라우저 기본(대략 128kbps) → **10분 ≈ 9.6MB** 추정. **[확인]**
- 업로드 전 크기검사 없음(`:263`은 `size===0`만) · 서버측 검사도 없음(`route.js`). **[확인]**
- **가장 나쁜 점:** 4.5MB 초과는 함수가 **실행조차 안 되므로** `console.error` 한 줄도 안 남고 → 클라 `res.json()` 파싱 실패 → `data.error` undefined → "AI 처리 실패"만 뜨고 **녹음이 통째로 소실**. 원인 추적 불가. **[확인]**

## 1. 설계 — 3겹 방어
| 겹 | 무엇 | 왜 |
|---|---|---|
| ① **bitrate 고정** | `audioBitsPerSecond = 48000`(48kbps) | 애초에 작게 굽는다. 48kbps opus/aac는 음성 STT에 충분(숫자·중량 발음 보존). **10분 ≈ 3.6MB** → 4.5MB 한도 아래. |
| ② **업로드 전 클라 크기 가드** | `blob.size > 4MB`면 전송 안 하고 안내 | **하드 가드.** 브라우저가 bitrate를 무시/미지원해도(iOS 등) 여기서 doomed 업로드를 막는다 → 413·소실을 원천 차단. |
| ③ **서버 백스톱** | `audio.size > 4MB`면 STT 전에 413 + 로깅 | 구버전 캐시 JS 등으로 클라 가드를 우회한 **4~4.5MB 구간**을 STT 비용 쓰기 전에 차단 + 서버 로그 남김. |

**bitrate 산수:** 48kbps × 600s ÷ 8 = **3.6MB**(+컨테이너 소량) < 4MB(클라 가드) < 4.5MB(플랫폼). ✔
→ 정상 10분 녹음은 ①로 이미 통과. ②는 bitrate가 안 먹힌 예외를, ③은 클라를 우회한 예외를 잡는 안전망.
→ **64kbps는 안 됨**(10분=4.8MB > 4.5MB). 48kbps 유지.

**하지 않는 것(이 묶음 범위 밖):** `maxDuration` 상향은 **묶음 D**. 녹음 blob 오프라인 큐(IndexedDB 재시도)는 별도 스프린트(리포트 §10-3). `MAX_RECORD_SEC`(10분)은 유지 — 48kbps에서 3.6MB라 줄일 필요 없음(주석만 정정).

### 2-4. 변경 후 (413 전용 분기 추가 — "잠시 후 다시" 꼬리표 제거)
```js
      if (!res.ok) {
        // 503 = AI 키 미설정(데모 환경) → 미리보기용 데모 유지. 그 외 실패는 가짜 리포트 안 채움.
        if (res.status === 503) { setNotice("AI 미설정 — 데모 리포트로 표시합니다."); runDemo(); return; }
        const data = await res.json().catch(() => ({}));
        // 413 = 용량 초과: 같은 녹음 재전송은 무의미 → 재시도 문구 없이 크기 안내만.
        if (res.status === 413) { failReal(data.error || "녹음 파일이 너무 큽니다. 더 짧게 나눠 다시 녹음해 주세요."); return; }
        failReal((data.error || "AI 처리에 실패했습니다.") + " 잠시 후 다시 시도해 주세요.");
        return;
      }
```

```js
    console.error(`[voice-log] 오디오 초과: ${audio.size} bytes (한도 ${MAX_AUDIO_BYTES})`);
    return Response.json(
      { error: "녹음 파일이 너무 큽니다. 더 짧게 나눠 다시 녹음해 주세요." },
      { status: 413 }
    );
  }
```

### 3-3. 로직 근거
- `form.get("audio")`는 Web FormData의 **File**(Blob 서브클래스) → `.size`(bytes) 보유. `typeof === "number"` 방어 후 사용.
- STT(OpenAI) 호출 **이전**에 차단 → 초과 파일에 비용 안 씀 + `console.error`로 서버 로그 남김(P0-3 "무음" 해소 · §6-1 관측성 정신).
- 앱레벨 413은 의미상 정확(Payload Too Large). 플랫폼 413과 코드 일치.
- **Next.js App Router route handler엔 자체 bodyParser 한도가 없음**(그건 구 Pages API 얘기) → `next.config` 변경 불필요. 실효 한도는 Vercel 플랫폼 4.5MB뿐.

## 5. 회귀 검증목록
**로컬(클로드코드):**
- `npm.cmd run build` exit 0 · `npm.cmd run lint` green.

**배포 후(폰 — 실기기 녹음):**
- **정상 경로:** 1~3분 녹음 → 정상 STT·요약 생성(회귀 없음). 콘솔에 413 없음.
- **bitrate 확인(선택·개발자도구):** 3~5분 녹음의 `blob.size`가 대략 (분수×0.36MB) 근처면 48kbps 적용된 것. 10분도 ~3.6MB로 4MB 미만인지.
- **가드 확인(선택):** `MAX_UPLOAD_BYTES`를 잠깐 아주 작게(예: `50*1024`) 낮춰 빌드 → 짧은 녹음도 **"녹음 용량이 커서…"** 안내가 뜨고 **업로드 자체가 안 나가는지**(네트워크 탭) 확인 후 원복.
- **STT 품질:** 48kbps에서도 중량·횟수·세트 숫자가 정확히 잡히는지(시끄러운 환경 1회) — 떨어지면 `AUDIO_BPS`만 상향 여지(단 10분×bitrate < 4.5MB 유지: 48k 안전, 56k도 4.2MB로 가능하나 마진 축소).

**불변 확인(안 바뀌어야 하는 것):**
- `handleStop`의 STT/요약 성공 경로·`onResult` payload·`buildText` 불변.
- 데모 폴백(503)·`failReal`·데모 워터마크 동작 불변.
- 서버 STT·요약·`parseReport`·응답 스키마(`{raw_text, report}`) 불변. `maxDuration=60` 불변(상향은 묶음 D).
