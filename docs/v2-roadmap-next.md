# v2 이후 로드맵 — 다음 스텝 (스프린트 3 대화 정리)

> 스프린트 3(관찰 기반 실 AI: `app/api/ot-brief` · `SecondOTTab` · `FirstOTAssist`) 완료 시점에
> 트레이너와 정리한 향후 방향. 상위 설계는 `docs/MASTERPLAN.md`, 직전 스프린트는
> `docs/v2-sprint3-ot-brief.md`. 이 문서는 "무엇을 / 왜 / 무엇이 선행돼야 하나"의 순서 지도다.

## 우선순위 원칙 — "완성도 먼저"

기능 개수를 늘리기 전에 **하나의 흐름(1차 → 2차 → PT)을 끝까지 완성**한다.

```
① 1차 OT 강화
   └─▶ ② OT/PT 뷰 분리  +  ③ PT 관리 뷰   (같이 묶여 감)
          └─▶ ④ 클로징 통계 대시보드
⑤ 치트키 운동 추천 = ①·③에 얹는 대칭 기능(각 뷰 완성될 때 함께)
⑥ 다듬기 스프린트(탭 이름·순서·프롬프트)  → 뒤로
⑦ 로그인 / RLS                            → 맨 마지막
```

- **①→②③→④** 순으로 세로 슬라이스를 완성한다.
- **⑤**는 독립 스프린트가 아니라 ①(1차)·③(PT)에 각각 얹는 대칭 기능 — 해당 뷰를 만들 때 함께.
- **⑥ 다듬기**와 **⑦ 로그인/RLS**는 기능이 자리 잡은 뒤 마지막. (지금 anon 전면 개방이 개발엔 편하므로 보안은 끝에 한 번에 잠근다.)

## ① 1차 OT 강화 (탭1 전면 개편 + 인바디)

- **무엇:** 스프린트 3에서 탭1은 **건드리지 않고** ① AI 블록(`FirstOTAssist`)만 얹었다. 이제 탭1의
  하드코딩 스캐폴드(타임라인·대본·루틴·클로징 빌더)를 **AI/실데이터 기반으로 전면 교체**하고,
  **인바디 구조화 입력**(등록 폼 확장)을 붙여 ①의 입력을 사전메모 textarea에서 정식 데이터로 승격.
- **왜:** 여정의 첫 관문. 1차가 얕으면 그 뒤(관찰·2차 브리핑)의 재료가 부실해진다. 또 지금 ①은
  사전메모가 비저장이라 캐시가 불가능(스프린트3에서 세션 재생성으로 둔 이유) — 인바디/사전정보를
  구조화 저장해야 ① 캐시·통계·개인화가 깨끗해진다.
- **선행 조건:**
  - 등록 폼 확장(인바디 항목 = 데이터 표준 템플릿의 출발점). MASTERPLAN "회원 작성 양식 → 디지털 폼" 라인.
  - 탭1 legacy 제거 시 회귀 주의(현재 타이머·`buildClosing`·PHASES 등 하드코딩과 얽힘).
  - MASTERPLAN ⭐⭐ 철학 유지(정답 대본 X, 근거+방향). ①은 '가설'임을 계속 명시.

## ② OT/PT 뷰 분리 (member_status, 하나의 앱 내 뷰 분리)

- **무엇:** 회원 라이프사이클 상태(OT 진행 중 / PT 등록 완료 등)를 `member_status`로 모델링하고,
  **하나의 앱 안에서 OT 뷰와 PT 뷰를 분리**한다(별도 앱 아님 — 상태에 따라 보이는 화면/탭이 갈림).
- **왜:** 지금은 모든 회원이 OT 관점 탭(1·2차 OT)으로만 보인다. 등록이 끝난 PT 회원은 다른 관심사
  (재등록·세션 관리)를 갖는다. 상태로 뷰를 가르면 트레이너가 "이 회원 지금 뭘 해야 하나"가 명확해진다.
  ③ PT 관리 뷰의 전제이기도 하다.
- **선행 조건:**
  - `member_status` 테이블/컬럼 신설(OT/PT 구분 + 단계). MASTERPLAN v2 예상 테이블.
  - ①로 1차/관찰 흐름이 안정된 뒤(상태 전이의 출발점이 OT 완료 → PT 등록).
  - ㉣ 1차/2차 클로징 결과(`ot_log.closing_result`)가 상태 전이 트리거의 재료로 이미 쌓이는 중.

## ③ PT 관리 뷰 (재등록 탭 편입, 실데이터화)

- **무엇:** 현재 하드코딩인 **재등록 CRM 탭**을 PT 뷰로 편입하고, 재등록 대상 판정·타이밍·대본을
  **실데이터**로 구동. (지금 `CRMTab`은 전부 데모 숫자/대본.)
- **왜:** OT의 목적지는 PT 등록·유지. PT 회원 관리(잔여 세션·만기·재등록 타이밍)가 실데이터로 돌아야
  제품이 "등록 후"까지 책임진다. ②로 뷰가 갈린 다음의 자연스러운 채움.
- **선행 조건:**
  - ② `member_status` + **`payment` / `session_log`** 테이블(만기일·잔여 세션·결제) 실데이터화.
  - 이게 없으면 하드코딩(가짜 숫자)로 회귀 = MASTERPLAN "뼈대 없이 붙이면 헛돈다" 위반.

## ④ 클로징 통계 대시보드

- **무엇:** 트레이너 본인 **클로징률** + 나이·성별·직업·`closing_approach`별 강점 분석. 관리자용
  "행동 가능한 목표 대시보드"의 축(오늘 수업 잡힌 재등록 대상 → 예상 매출)과 연결.
- **왜:** 스프린트3부터 쌓기 시작한 `closing_result`/`closing_approach`(1차·2차)가 원천 데이터.
  트레이너가 "내가 어떤 방향(통증/외형/가치)에서 강한가"를 알면 자기 클로징을 교정할 수 있다(상향평준화).
- **선행 조건:**
  - ㉠ 클로징 결과가 **충분히 축적**돼야 의미 있음(그래서 ①②③ 뒤).
  - 관리자 대시보드 확장은 `payment`/`session_log`(③) + `trainer`(⑦) 위에 선다.

## ⑤ 치트키 운동 추천 (①·③ 대칭 기능)

- **무엇:** 상황별 **운동/움직임 추천**을 ①(1차 지원)과 ③(PT 관리)에 **대칭으로** 얹는다.
  1차엔 즉각 피드백 동작, PT엔 세션 흐름용 추천. **방향까지만** — 숫자 처방(각도·템포·세트·중량) 금지.
- **왜:** 트레이너가 현장에서 "이 상태엔 이 움직임"을 빠르게 꺼내 쓰는 치트키. 단 v1.5 원칙대로
  레시피(숫자)를 주면 사고 정지 → 하향평준화. "왜(원리)+방향"만 줘서 스파링 파트너로 유지.
- **선행 조건:**
  - 독립 스프린트가 아니라 ①·③ 각 뷰가 생길 때 함께 구현(대칭 UI/프롬프트).
  - 관찰 데이터(observe_targets / movements)·PT 맥락이 재료. `app/api/ot-brief` 프롬프트 패턴 재사용.

## ⑥ 다듬기 스프린트 (탭 이름·순서·프롬프트)

- **무엇:** 탭 이름·순서 재배치, 프롬프트 문구 다듬기, 카피·톤 정리 등 **표면 정돈**.
- **왜:** 기능이 자리 잡기 전에 이름·순서를 확정하면 계속 뒤집힌다. 흐름(①②③④)이 굳은 뒤 한 번에
  정돈하는 게 효율적. (예: 관찰 기록 탭 위치, 2차 OT 탭 내 섹션 순서, ① 블록 카피 등.)
- **선행 조건:** ①~④가 기능적으로 완성돼 "최종 흐름"이 보일 때. 그 전엔 부분 다듬기만.

## ⑦ 로그인 / RLS (맨 마지막)

- **무엇:** Supabase Auth 로그인 + `trainer_id` 소유권 + RLS를 `trainer_id = 로그인 사용자` 기준으로
  잠금 + `/admin` 권한 분리. "트레이너가 로그인하면 본인 담당 회원만 보인다"는 이 작업의 결과물.
- **왜:** 현재 anon 전면 개방(주소·키만 알면 접근) = 현장 배포 전 필수 청산 대상. 개발 중엔 개방이
  편하므로 **맨 마지막에 한 번에** 잠근다. (스프린트별로 열어둔 `ot_log` anon UPDATE 등 부채 포함.)
- **선행 조건:**
  - ②의 `member_status`, ③/④의 `payment`·`session_log`, 그리고 `trainer` 테이블이 자리 잡은 뒤
    (RLS를 어떤 소유권 축으로 잠글지가 그 스키마에 달림).
  - 운영 방식 확정 필요(담당 배정 주체·시점, 담당 변경/이관, 다중 트레이너 열람, 관리자 권한 차등).
  - **미리 해둘 값어치:** `user_table.trainer_id` 컬럼만 선반영(값은 나중) — 스프린트2 ot_log 컬럼 선반영과 같은 논리.
