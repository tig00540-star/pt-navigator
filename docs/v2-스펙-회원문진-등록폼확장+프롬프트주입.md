## 2. `app/page.jsx` — MemberForm 필드 추가

기존 `fields` 배열(name/age/job/residence/mbti/pain/goal — 전부 text input)에 아래 9개를 **이어서 추가**(같은 text input 패턴 그대로). 트레이너가 종이 문진표 보고 옮겨적기용이라 자유 텍스트로 충분.

```js
// (기존 7개 아래에 이어서)
{ k: "goal_deadline",  label: "목표 시점·계기",   ph: "예: 8월 결혼 / 없으면 비움" },
{ k: "training_pace",  label: "원하는 페이스",     ph: "가볍게 / 제대로 / 집중해서" },
{ k: "injury_history", label: "부상·수술 이력",   ph: "없음 / 2년 전 무릎 수술 등" },
{ k: "exercise_level", label: "운동 경험",         ph: "처음 / 가끔 / 꾸준히" },
{ k: "quit_reason",    label: "예전 중단 이유",    ph: "시간·동기·효과·부상·혼자 막막 등" },
{ k: "past_exercise",  label: "받아본 유료 운동",  ph: "PT, 필라테스 등 / 없음" },
{ k: "availability",   label: "가능 빈도·시간대",  ph: "주 2회 · 저녁" },
{ k: "activity_level", label: "하루 활동량",       ph: "주로 앉아서 / 활동적" },
{ k: "member_note",    label: "바라는 점(선택)",   ph: "회원이 미리 남긴 말" },
```

- **폼 UX:** 기존 7개(핵심) 뒤에 시각 구분선 + 소제목 `— OT 사전 문진 (선택) —`을 넣어, 사전 문진 9개는 선택 입력임을 표시(모달이 길어지니 구분). 구현 부담되면 일단 그냥 이어 붙여도 됨(후속에 접이식 가능).
- **저장 배선:** MemberForm의 insert payload가 `fields`의 `k`를 키로 구성한다면 자동 반영. **직접 확인 필요** — 폼 state·insert가 field.k를 순회하는지 읽고, 아니면 신규 9키를 payload에 명시 추가. (전부 nullable이라 빈 값이면 null/미포함 OK.)
- **회원 로드:** 목록/선택 시 `select("*")`면 신규 컬럼이 member에 자동 실림(firstOTAssist가 member 통째 전송). select 컬럼을 명시 지정하는 곳이 있으면 신규 9개 추가.

## 3. `app/api/ot-brief/route.js` — firstPrompt 주입

### 3-a. `[회원 기본정보]` 아래에 `[OT 사전 문진]` 블록 추가
`firstPrompt`의 `[회원 기본정보]...machines=${machines}` 줄 **바로 아래**(가설 톤 안내 위)에 삽입:
```
[OT 사전 문진 — 회원이 등록 때 작성(있는 것만 활용, 없으면 무시)]
 목표시점/계기=${g(m.goal_deadline)}, 원하는페이스=${g(m.training_pace)}, 부상·수술=${g(m.injury_history)},
 운동경험=${g(m.exercise_level)}, 예전중단이유=${g(m.quit_reason)}, 받아본유료운동=${g(m.past_exercise)},
 가능빈도·시간=${g(m.availability)}, 하루활동량=${g(m.activity_level)}, 바라는점=${g(m.member_note)}
```

### 3-b. 활용 지침 추가 (각 블록에 어떻게 쓰는지)
`[★운동목적(goal) = 최우선 렌즈]` 블록 **아래**에 삽입:
```
[사전 문진 활용 — 클로징 근거 강화]
 - goal_deadline(시점): session_logic의 기간 역산과 '왜 지금'에 최우선 반영("D-데드라인까지 N개월이라
   주2회는 필수"). 없으면 일반 기간 논리로.
 - availability(가능 빈도): recommended_program.frequency의 '가능한 현실 빈도' 근거로 삼고, 그 빈도가
   목표에 부족하면 그 격차를 objection_defense의 time(시간부족) 선제 방어로 연결.
 - quit_reason(예전 중단 이유): objection_defense.doubt(효과 의심)와 클로징 손실 프레이밍의 직접 근거
   ("혼자 하다 ○○로 멈추셨잖아요 — 이번엔 그 지점을 제가 잡아드려요"). 없는 이유 창작 금지.
 - injury_history·exercise_level·activity_level: target_exercise의 동작 선택·난이도·안전(부상 부위는
   피하거나 우회)과 비유 소재로. injury_history 있으면 의료경계 유지(숫자·의료 단정 금지 그대로).
 - training_pace·member_note: 오프닝·전반 톤 보조(있을 때만).

[★재정·투자 신호 → 세션·단가 제시 강도] job·past_exercise·training_pace로 이 회원의 지불 여력·투자
 의향을 읽어라.
 - 여유·의향이 높아 보이면(전문직 등 안정 직군 · 과거 PT/필라테스 등 유료 운동 경험 · '제대로/집중해서'
   선택): recommended_program에서 위축되지 말고, 목표 달성에 '필요한 만큼' 넉넉한 회차의 패키지(pick_ref)를
   확신 있게 골라라. session_logic도 당당하게(할인·최소량으로 미리 물러서지 말 것).
 - 금전 제약이 커 보이면(관련 신호 부재 · '가볍게 시작'): 최소 필요량부터 제안하고 이후 확장 여지를 남겨라.
 - ⚠️ 이 판단은 pick_ref 선택과 session_logic '톤'에만 반영한다. 회원에게 '여유 있어 보인다/돈' 식의
   재정 언급은 값 텍스트에 절대 금지(추정을 입 밖에 내지 말 것).
```

> 스키마·`REQUIRED_FIRST`·렌더 무변경(입력만 늘고 출력 스키마 동일). FIELD_TERMS도 무변경(신규 필드는 프롬프트 '입력'이라 출력 누출 위험 낮음).

## 4. 검증 체크리스트
1. **SQL 먼저 Supabase 실행 → 검증쿼리로 컬럼 9개 확인** 후 코드.
2. `npm.cmd run lint` green.
3. **등록/수정 왕복:** MemberForm에서 신규 9필드 입력·저장 → 새로고침 재진입 시 값 유지(insert 배선 확인). 빈 값 저장도 OK.
4. **AI 반영(실계정):** 데드라인·가능빈도 채운 회원으로 1차 '다시 생성' → session_logic이 **데드라인 기준으로 기간·회차 역산**하는지, quit_reason이 거절 doubt에 반영되는지 눈검.
5. **재정 신호:** 과거 PT 경험+전문직 회원 vs 미기재 회원 비교 → 전자에 **더 넉넉한 회차 패키지·당당한 session_logic**이 나오는지. 단 출력에 '재정' 직접 언급 0(가드 확인).
6. 기존 회원(문진 미입력)은 신규 필드 전부 null → **기존과 동일 동작**(회귀 안전).
