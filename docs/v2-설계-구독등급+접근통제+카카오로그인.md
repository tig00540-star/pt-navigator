# v2 설계 — 구독 등급 · 가입 후 접근 통제 · 트레이너 카카오 로그인

> 성격: 방향 합의용 **설계도**(구현 스펙 아님 — 스프린트별 구현 스펙은 착수 시 따로). 세 가지를 한 그림으로: ① 구독 등급(프리미엄=회원관리), ② 가입만으론 못 쓰고 **결제해야 자동 활성**, ③ **트레이너 앱 카카오 로그인**.
> 확정(이번 논의): 카카오 로그인 = **트레이너(앱 로그인)**. 접근 통제 = **결제하면 자동 활성**.
> 전제: 지금 구조 = 계정(`account`)마다 소유·과금 단위 1행, 트레이너(`trainer`)에 `active` 스위치, RLS 헬퍼(`auth_account_id()`/`auth_is_owner()`)가 **`active`일 때만** 값을 돌려줌(비활성 → NULL → 전 데이터 잠금). 이 `active`가 이미 **마스터 스위치**라 여기에 얹으면 된다.

---

## 0. 한눈에 — 잠금은 "두 층"이다

```
[층 1] 앱을 쓸 수 있나?      = 계정이 '결제 활성'인가            ← 가입만으론 X, 결제해야 O
        └ 비활성이면: 로그인은 되지만 데이터 전부 잠김 → "구독하세요" 벽

[층 2] 회원관리를 쓸 수 있나? = 계정 요금제가 'premium'인가        ← 일반 구독은 X, 프리미엄만 O
        └ 일반이면: 회원 링크 생성·회원앱 로그인 자체가 서버에서 거절

로그인 방법(누가 들어오나): 이메일/비번  또는  ★카카오  (트레이너)
```

세 가지 다 **`account` 행에 딱지 두 개 붙이고, 서버에서 그 딱지를 확인**하는 같은 패턴이야. 화면에서 숨기는 건 편의고, **진짜 잠금은 서버**(회원앱 RLS 때랑 똑같은 원칙 — UI만 숨기면 우회당함).

## 1. 스키마 — `account`에 구독 딱지 두 개

```
account (기존: id · type[center/solo] · name · created_at)
  + subscription_status  text  'inactive' | 'active'          ← 층1: 앱 사용 가능?
  + plan                 text  'basic'    | 'premium'         ← 층2: 회원관리 가능?
  + (선택) current_period_end  timestamptz                    ← 구독 만료일(결제연동 때)
  + (선택) billing_customer_id text                           ← 결제업체 고객 id(웹훅 매칭용)
```

- 신규 가입 계정 = `subscription_status='inactive'`, `plan='basic'`으로 태어남 → **가입 직후엔 잠긴 상태**(결제 전).
- ★**기존 계정(너 포함) 보호:** 마이그레이션에서 **이미 있는 account를 전부 `active`로 백필**(grandfather). 안 하면 너부터 잠긴다. (모든 마이그레이션의 첫 안전수칙 = 기존 행 보호.)

> 등급을 더 잘게(예: 기능별 on/off) 팔고 싶어지면 나중에 `entitlements` 테이블로 확장. **지금은 `plan` 2단(basic/premium)이면 충분** — "회원관리 = premium 전용" 하나만 필요하니까.

## 2. 층 1 — "결제해야 앱을 쓴다" (접근 통제)

**메커니즘 = 이미 있는 킬스위치 재사용.** 헬퍼 `auth_account_id()`가 지금은 `trainer.active`만 보는데, 여기에 **"그 계정이 subscription_status='active'인가"**를 AND로 더한다.

```
auth_account_id():  trainer.active = true  AND  account.subscription_status = 'active'  일 때만 내 account_id, 아니면 NULL
```

효과: 비활성(미결제) 계정으로 로그인하면 `auth_account_id()`=NULL → **회원목록·탭·음성일지·매출 전부 0행/차단**. 앱은 뜨지만 알맹이가 다 잠긴 "구독하세요" 화면이 됨. 별도 게이트 코드 없이 **RLS 하나로 전면 잠금**.

**"구독하세요/온보딩" 화면은 뭘 보고 그리나:** 잠긴 상태에선 account 행도 못 읽어(정책이 NULL이라). 그래서 **상태 조회용 `security definer` 함수 하나**(`my_account_status()`)를 둬서, active 여부와 무관하게 `{ 계정있음?, subscription_status, plan }`만 안전히 돌려줌 → 클라가 "① 계정 없음=온보딩 / ② 미결제=결제벽 / ③ 활성=정상"을 구분해 화면 분기.

---

## 3. 층 2 — "프리미엄만 회원관리" (프리미엄 게이트)

회원 앱(회원관리)을 여는 **두 동작**에 `plan='premium'` 체크를 건다(이중):

1. **트레이너가 회원 링크 발급**(회원앱 S3) — 링크 생성 버튼 옆에 잠금(UI) + **서버에서 발급 시 plan 확인**. 발급을 `security definer` RPC(`issue_member_token(member_id)`)로 감싸 "premium 아니면 거절"하면 UI 우회도 차단.
2. **회원이 링크로 로그인**(`/api/member-auth`) — 라우트가 그 회원의 **소속 계정 plan을 확인**, `premium` 아니면 로그인 거절. → 프리미엄 해지하면 **이미 나간 링크들도 그날부터 안 열림**(다운그레이드가 자동 반영).

- 헬퍼 하나 추가 추천: `auth_account_plan()` (`auth_account_id()`와 동일 틀) → RLS·UI에서 재사용.
- **UI:** 일반 구독이면 회원관리 진입점 자체를 🔒 프리미엄 배지로 표시(+ "업그레이드" 유도).

> 이 틀을 한 번 깔면 앞으로 나올 기능(식단 AI·루틴 AI 등)도 "이건 premium, 이건 basic도" 하고 **자유롭게 나눠 팔 수 있음** — 수익 모델 전체의 스위치가 생기는 셈.

## 4. 결제 → 자동 활성 흐름 (확정 방식)

```
[트레이너] 구독 결제(토스페이먼츠/스트라이프 등)
     │
     ▼ 결제업체가 "결제 완료" 웹훅을 우리 서버로 쏨
[서버] /api/billing-webhook (service_role · 서명 검증)
     │   → 해당 account.subscription_status='active' (+ plan=결제한 등급, current_period_end 갱신)
     ▼
[트레이너] 다음 요청부터 auth_account_id()가 값 반환 → 앱 알맹이 열림. (프리미엄 결제면 회원관리까지)
   해지·미납 웹훅 → subscription_status='inactive' 또는 plan='basic'으로 자동 강등.
```

**MVP 팁(중요):** 결제업체 연동은 무거우니 **1단계는 "수동 활성"으로 시작**해도 됨 — 초기엔 고객 몇 명이라, 네가 대시보드(또는 관리자 화면 버튼)에서 `active`/`premium`을 직접 켜주는 걸로 충분. 위 스키마·게이트는 그대로 두고 **웹훅만 나중에 얹으면** 자동화됨. 즉 "지금은 수동 스위치 → 나중에 결제 자동" 매끄럽게 승급.

## 5. 카카오 로그인 (트레이너 앱)

Supabase Auth가 **카카오를 기본 지원**(OAuth 제공자 목록에 있음)이라 새 인증서버 만들 필요 없음. 붙이는 순서:

1. **카카오 개발자 콘솔:** 앱 생성 → 카카오 로그인 활성화 → **Redirect URI에 Supabase 콜백**(`https://<프로젝트>.supabase.co/auth/v1/callback`) 등록 → Client Secret 발급 → 동의항목 설정.
2. **Supabase 대시보드:** Auth → Providers → **Kakao 켜고** Client ID/Secret 붙여넣기.
3. **클라(로그인 버튼):** `supabase.auth.signInWithOAuth({ provider: 'kakao', options: { redirectTo } })` → 카카오 동의 → 콜백으로 돌아오며 세션 발급. 기존 이메일/비번 로그인은 **그대로 두고 버튼만 추가**(둘 다 지원).

**★놓치기 쉬운 핵심 — 카카오 최초 로그인은 "계정 자동 생성"이 안 걸린다.** 지금 셀프가입은 이메일 `signUp` 때 넘기는 `account_type` 메타데이터를 트리거(`handle_new_signup`)가 읽어서 account+trainer를 만듦. 그런데 **OAuth(카카오)는 그 메타데이터가 없어** → 카카오로 처음 들어온 사람은 auth 유저만 생기고 **trainer 행이 없음** → (마침) `auth_account_id()`=NULL이라 자동으로 잠긴 상태가 됨. 그래서:

- **카카오 최초 로그인 → 온보딩 화면 1장**("개인 / 센터 선택 + 이름") → **프로비저닝 라우트**(service_role)가 account(inactive)+trainer(owner) 생성. 이메일 가입과 **같은 프로비저닝 경로로 통일**하는 게 깔끔(트리거는 이메일 전용으로 남기거나, 프로비저닝을 라우트로 일원화).
- 이렇게 만든 신규 계정도 **subscription_status='inactive'** → 온보딩 끝나면 자연스럽게 **결제벽**으로 이어짐(층1과 맞물림).
- (참고) 카카오는 이메일 미제공일 수 있음(비즈앱 심사 필요) → **이메일에 의존하지 않게** 설계(우린 account를 auth uid로 잇지 이메일로 안 잇음 → 무관). 같은 사람이 이메일 계정과 카카오를 **한 계정으로 합치는 것(identity linking)**은 후속 니스투해브.

## 6. 기존 계정·회귀 보호 (필수)

- 마이그레이션에서 **기존 account 전부 `subscription_status='active'`, 필요 등급으로 백필**(너=premium). 안 하면 본인부터 잠김.
- `auth_account_id()`에 subscription 조건 추가 = **광범위 영향**(전 테이블 잠금 좌우) → 반드시 **center_machine 1개 리트머스 → 나머지** 방식(⑦-b 때처럼 단계 배포)으로.
- 트레이너 초대(`create-trainer`)는 owner의 account를 물려받으므로 별도 결제 없이 그 계정 등급을 따름(좌석 개념) — 좌석 과금은 후속.

## 7. 스프린트 로드맵 (순서 · 난이도 · 의존성)

| # | 스프린트 | 하는 일 | 난이도 | 의존 |
|---|---|---|---|---|
| (진행) | 회원앱 S1→S3 | 회원 인증·열람·링크발급 = **회원관리 기능 실체** | — | — |
| B1 | **구독 스키마 + 층1 잠금(수동)** | account에 딱지 2개 + `auth_account_id()`에 subscription AND + 기존 active 백필 + `my_account_status()` + 결제벽/온보딩 화면 분기. **활성은 아직 수동 스위치.** | 중 | — |
| B2 | **프리미엄 게이트(층2)** | `auth_account_plan()` + 회원 링크발급·`member-auth`에 premium 체크 + UI 🔒. **회원앱 S3와 함께 붙이면 자연스러움.** | 소 | B1 · 회원앱S3 |
| B3 | **카카오 로그인(트레이너)** | Supabase Kakao 설정 + 로그인 버튼 + **온보딩 프로비저닝 라우트**(이메일 경로와 통일). 신규=inactive로 층1과 연결. | 중 | (B1 있으면 결제벽까지 매끄러움) |
| B4 | **결제 자동 활성** | 결제업체(토스 등) 연동 + `billing-webhook`(서명검증)로 subscription/plan 자동 전환. B1의 수동 스위치를 대체. | 상 | B1 |

**추천 순서:** 회원앱(S1~S3) 마무리 → **B1(수동 구독 잠금)** → **B2(프리미엄 게이트)** → **B4(결제 자동)** 또는 **B3(카카오)** 중 급한 것부터. B3(카카오)는 나머지와 **독립적**이라, 가입 마찰을 먼저 줄이고 싶으면 앞으로 당겨도 됨(단 B1이 있어야 신규가 결제벽으로 이어짐).

> 이유: **돈 받는 스위치(B1)를 먼저** 깔면, 프리미엄(B2)·결제자동(B4)·카카오신규(B3)가 전부 그 스위치에 얹혀 일관됨. 회원앱을 먼저 끝내는 건, 프리미엄으로 팔 "물건(회원관리)"이 있어야 B2가 의미 있어서.

## 8. 결정 포인트 (착수 전 정해두면 좋음)

- **결제업체:** 국내 위주면 토스페이먼츠/포트원(아임포트), 해외 포함이면 스트라이프. (B4 때 선택 — 지금 안 정해도 B1~B3 진행 무관.)
- **미결제 상태 UX:** 완전 잠금(결제벽만) vs 최소 열람 허용(예: 회원목록만 read-only 맛보기). 확정 방식은 "결제해야 활성"이라 **완전 잠금**이 기본 — 맛보기를 줄지 여부만 나중에.
- **프리미엄 범위:** 지금은 "회원관리=프리미엄" 하나. 식단AI·루틴AI 나올 때 각 기능을 basic/premium 어디 둘지는 그때 `plan`/entitlements로.

이 그림 OK면, **B1(구독 스키마 + 층1 잠금)부터 구현 스펙**(SQL: account 딱지·헬퍼 확장·백필·`my_account_status()` + 화면 분기)으로 내려갈게. 카카오(B3)를 먼저 당기고 싶으면 그쪽부터 스펙 잡아도 되고 — 순서만 말해줘.

## 9. 가격 모델 (초안 · 원가 실측 반영 · 2026-07)

> 성격: 방향 초안. 확정 숫자는 시장·경쟁사·초기 고객 반응으로 검증(재무 자문 아님). 타깃 = **솔로 우선 → 센터 확장.**

### 9.1 과금 축 (요약)

- **주축 = 트레이너 좌석(seat)당 월정액.** 솔로 = 1좌석(= account), 센터 = 활성 트레이너 N좌석(볼륨 할인). §1 account→trainer 좌석 구조에 그대로 얹힘.
- **기능 등급 = basic / premium**(§3 층2 그대로 · premium = 회원앱 = 회원관리).
- **회원 수 = 과금 단위 아님.** 오직 후속 **회원 AI(식단·루틴) 포함량 상한**에만 사용(§9.4 · entitlements 훅).
- **수업 수 종량제 금지** — 음성일지 사용을 억제해 성과그래프·재등록 데이터 flywheel과 lock-in을 스스로 훼손. 정액에 넉넉히 포함이 정석.

> 근거: basic 좌석 원가(트레이너-쪽 AI)는 **한 트레이너가 칠 수 있는 수업 수에 물리적 상한**이 있어 자연히 bounded → 정액 포함이 안전. 회원 AI만 회원 수에 정비례(§9.4).

### 9.2 원가(COGS) 실측 — 2026-07 현행 단가

모델(코드 기준): 음성일지 요약·OT 전 phase = `claude-sonnet-5`, STT = `gpt-4o-mini-transcribe`.

| 항목 | 현행 단가 |
|---|---|
| Claude Sonnet 5 | 입력 **$2** / 출력 **$10** per MTok (**도입가 · 2026-09-01부터 $3/$15**) |
| Claude Haiku 4.5 | 입력 $1 / 출력 $5 per MTok |
| gpt-4o-mini-transcribe | $0.003/분 |

건당 실비(환율 ≈ ₩1,400/$ 가정):

- **음성일지 1건** ≈ STT(2분 $0.006) + Sonnet 요약(입력~3k·출력~1.5k ≈ $0.021) ≈ **$0.027 ≈ ₩40**.
- **OT 1차 1건** ≈ Sonnet(입력~4k·출력~5.5k, `max_tokens=8192`) ≈ **$0.063 ≈ ₩88**.
- **OT 2차 1건** ≈ Sonnet(입력~3k·출력~3.5k) ≈ **$0.041 ≈ ₩57**.

### 9.3 솔로 basic 월 원가·마진 (못박기)

바쁜 솔로(월 음성일지 ~150 + OT 브리핑 ~30, 재생성 포함):

- 음성일지 150 × ₩40 = ₩6,000 · OT 30 × ₩75 = ₩2,250 · 인프라(Supabase·Vercel) 소액 → **월 COGS ≈ ₩8,000~10,000**(헤비 ~₩13,000).
- 목표가 **₩50,000~70,000** → **COGS ~13~20% · 매출총이익 ~80~87%.** SaaS 기준 매우 건강.
- ⚠️ **2026-09-01 Sonnet 5 도입가 종료(+50%)** → COGS ~1.5배(월 ~₩12,000~19,000). 헤비유저 @₩50,000이면 COGS ~35%까지 → 그때 원가 레버(§9.5) 발동 또는 헤비유저 상한.

### 9.4 premium · 센터 · 회원 AI

- **premium** = basic + 회원앱. 현재 회원앱은 **읽기전용이라 회원당 원가 ≈ 0** → 통째 번들 안전. 가격 = basic + α(회원앱 가치분).
- **센터** = account 다좌석 × 좌석단가(볼륨 할인) + 센터 전용(admin 매출 KPI·트레이너 QC·통합 페이롤).
- **회원 AI(식단·루틴 · 후속) = 유일한 회원수 비례 원가.** premium에 **활성 AI회원 N명 포함 + 초과 상위밴드/애드온**으로 가드(§8 프리미엄 범위·entitlements). **런칭 시엔 미라이브** → B1 스키마에 활성 AI회원 카운트 **자리만** 잡아두면 됨.

### 9.5 원가 레버 (마진 압박 시 · 지금은 대기)

- **음성일지 요약 → Haiku 4.5**: `voice-log/route.js`의 `SUMMARY_MODEL` **한 줄** 교체(코드 주석에 명시). 건당 ₩40→~₩23(~40%↓). 루틴 로그엔 충분 → 헤비유저·하위티어용.
- **OT 1차 출력 축소**(`max_tokens=8192`가 최대 원가 덩어리) — 세일즈 품질 트레이드오프라 신중.
- 지금 ₩5~7만·80%+ 마진에선 **안 건드림**(세일즈 클로징 품질 = 제품 가치). Sonnet 도입가 종료 후 헤비유저 상한용으로 대기.

### 9.6 B1 스키마 반영 포인트

- `account.plan`(basic/premium)은 §1대로. 추가로 **좌석 개념**(활성 트레이너 수 = 이미 `trainer.active`로 집계 가능)과, 후속 **활성 AI회원 쿼터** 자리를 entitlements로 열어둠.
- 숫자(등급별 가격·포함량)는 B1 구현엔 불필요(`plan` 2단이면 충분) — 위 마진 근거로 런칭 직전 확정.
