## 0. 무엇을 만드나 (한 줄)

`account`에 구독 딱지를 달고, **층1 잠금**(결제/활성 안 된 계정은 로그인은 되지만 데이터 전부 잠김 = "구독하세요" 벽)을 켠다. **베타 = 네가 고른 10명만 수동 활성 + 7일 자동 만료.** 프리미엄 게이트(B2)·결제 자동화(B4)·카카오(B3)는 **별도 스프린트**.

**핵심 안전장치(먼저 각인):**
1. 컬럼을 `not null default 'inactive'`로 추가하면 **기존 행이 전부 inactive로 채워짐** → **같은 마이그레이션에서 즉시 active 백필**(안 하면 트레이너 본인부터 잠김).
2. `auth_account_id()` 변경은 **전 테이블 잠금을 좌우**하는 광범위 작업 → **백필 확인 후 별도 단계**로, `center_machine` 리트머스 먼저.

## 2. 스키마 — 마이그레이션 A (컬럼 + 백필 · 행동 무변경)

> A는 **헬퍼를 안 건드리므로 아직 아무것도 안 잠긴다**(안전). 잠금은 B에서 켬. A와 B를 나누는 게 안전핵심.

**`docs/migrations/2026-07-16-b1a-account-subscription.sql`**

```sql
-- =============================================================================
-- B1-A — account에 구독 딱지 + 기존행 active 백필 (헬퍼 미변경 = 이 단계선 잠금 안 켜짐)
-- 실행: Supabase SQL Editor(수동) · git 기록본. 멱등.
-- ★ not null default 'inactive'는 기존 행을 전부 inactive로 채우므로, 같은 파일에서 즉시 백필.
-- =============================================================================
alter table account add column if not exists subscription_status text not null default 'inactive';
alter table account add column if not exists plan               text not null default 'basic';
alter table account add column if not exists current_period_end timestamptz;  -- 접근 유효기한(체험/결제; null=무기한)

-- ★기존 계정(너 포함) grandfather — 안 하면 본인부터 잠김.
update account
  set subscription_status = 'active',
      plan = 'premium',
      current_period_end = null
  where true;   -- 이 시점 기존 행 전체(신규는 이 마이그레이션 이후 default로 태어남)

-- 검증: select id, subscription_status, plan, current_period_end from account;  -- 전부 active/premium/null
-- 롤백: alter table account drop column if exists subscription_status, drop column if exists plan, drop column if exists current_period_end;
```

> 신규 계정은 이 마이그레이션 **이후** default(`inactive`/`basic`)로 태어남 → B에서 잠금 켜지면 자동으로 결제벽.

## 3. 층1 잠금 — 마이그레이션 B (`auth_account_id()` 확장 · 광범위 · 백필 후에만)

> **A 검증(전 기존계정 active) 끝난 뒤에만** 실행. 이걸 켜면 inactive/만료 계정은 전 데이터 0행.

**`docs/migrations/2026-07-16-b1b-subscription-gate.sql`**

```sql
-- =============================================================================
-- B1-B — auth_account_id()에 구독 활성 + 만료 조건 AND (층1 잠금 켜짐)
-- ★ 반드시 B1-A 백필 확인 후. 광범위(전 RLS 좌우) → 적용 후 center_machine 리트머스.
-- 기존 정의는 trainer.active만 봤음 → 여기에 account 구독 조건을 AND.
-- =============================================================================
create or replace function auth_account_id()
returns uuid
language sql stable security definer set search_path = public as $$
  select t.account_id
  from trainer t
  join account a on a.id = t.account_id
  where t.auth_user_id = auth.uid()        -- ← 실제 trainer↔auth 링크 컬럼명에 맞출 것(스키마 확인)
    and t.active = true
    and a.subscription_status = 'active'
    and (a.current_period_end is null or a.current_period_end > now())
  limit 1
$$;
-- ⚠️ 위 select 본문(조인·컬럼명·active 조건)은 **현재 auth_account_id() 정의를 그대로 가져와**
--    subscription 3줄만 AND로 얹는 것. 기존 로직을 재작성하지 말고 확장만 할 것.
-- ⚠️ auth_is_owner()가 auth_account_id()와 독립적으로 계산되면, 거기에도 같은 subscription AND를
--    넣거나 auth_account_id() 기반으로 유도해 owner 권한도 잠기게 할 것(확인 필요).

-- 리트머스(적용 직후, 본인=active 세션): GET /rest/v1/center_machine → 행 반환되면 OK.
--   0행이면 백필 누락/조인 오류 → 즉시 롤백.
-- 롤백: create or replace function auth_account_id() ... (구독 3줄 뺀 기존 정의 그대로 복원).
```

## 4. 상태 조회 — `my_account_status()` (결제벽 화면용)

잠긴 상태에선 `account` 행도 못 읽음(헬퍼 null) → 화면 분기용 **SECURITY DEFINER 함수**로 활성 여부와 무관하게 안전히 상태만 반환.

**`docs/migrations/2026-07-16-b1c-account-status.sql`**

```sql
-- =============================================================================
-- B1-C — my_account_status(): 로그인 유저의 계정 상태(잠김이어도 조회 가능) → 화면 분기용
-- =============================================================================
create or replace function my_account_status()
returns table (has_account boolean, subscription_status text, plan text,
               current_period_end timestamptz, is_expired boolean, access boolean)
language sql stable security definer set search_path = public as $$
  select
    (t.id is not null)                                             as has_account,
    a.subscription_status,
    a.plan,
    a.current_period_end,
    (a.current_period_end is not null and a.current_period_end <= now()) as is_expired,
    (a.subscription_status = 'active'
       and (a.current_period_end is null or a.current_period_end > now())) as access
  from trainer t
  left join account a on a.id = t.account_id
  where t.auth_user_id = auth.uid()      -- ← trainer↔auth 링크 컬럼(§3과 동일)
  limit 1
$$;
grant execute on function my_account_status() to authenticated;
-- 계정 없는 유저(향후 카카오 최초): 0행 → 클라에서 has_account=false로 처리.
-- 검증: 본인 세션에서 select * from my_account_status();  → access=true.
```

## 5. 클라 화면 게이트 (트레이너 앱 루트)

**진입 시 `my_account_status()` 호출 → 분기.** 탭 렌더 **전에** 판정.

- `!data || !has_account` → 온보딩/안내(현재 셀프 signup은 트리거로 account가 생기므로 드묾 · fallback 문구 "계정 준비 중" + 재로그인).
- `access === false` → **결제벽 화면**:
  - `is_expired` → "무료 체험이 종료됐어요"(연장·전환 안내).
  - else(inactive) → "구독이 필요해요 / 담당자에게 문의"(베타는 문의 CTA · 결제 버튼은 B4에서).
- `access === true` → **정상 앱**(기존 탭 렌더).
- **데모 모드(`supabase` null)** → `access=true`로 간주(개발·데모 유지).

**배치:** 트레이너 앱 최상위(`app/page.jsx` 렌더 초입 또는 감싸는 게이트 컴포넌트). 로딩 중 스피너 → 판정 후 분기. 상태는 async effect에서 set(set-state-in-effect 회피 패턴, 회원앱 `/m`과 동일).

**UI:** 결제벽은 기존 토큰(레드 브랜드·`bg-bg`·`Card`·`Button`) 재사용. 새 색·동적 클래스 금지.

## 6. 베타 운영 — 수동 활성 + 7일 자동 만료 (UI 없음 · 대시보드 SQL)

베타는 별도 화면 불필요. **트레이너가 셀프 가입 → 이메일 알려줌 → 네가 SQL로 활성.**

```sql
-- (1) 이메일로 account_id 찾기 (조인 컬럼은 실제 스키마에 맞춰 — 클로드코드가 채움)
select a.id as account_id, u.email, a.subscription_status, a.current_period_end
from account a
join trainer t on t.account_id = a.id
join auth.users u on u.id = t.auth_user_id
where u.email = '베타트레이너@example.com';

-- (2) 활성 + 7일 체험창
update account
  set subscription_status = 'active', plan = 'premium',
      current_period_end = now() + interval '7 days'
  where id = '<account_id>';

-- 연장:  update account set current_period_end = now() + interval '7 days' where id='<id>';
-- 해지/만료 강제:  update account set subscription_status='inactive' where id='<id>';
```

- 대시보드 SQL = service_role → **RLS 우회 → account UPDATE 정책 불필요**(교훈1 회피).
- 7일 만료는 **자동**(§3 헬퍼가 `current_period_end`를 `now()`와 비교 → 지나면 다음 요청부터 잠김, 크론 불필요).
- **베타 계정 태깅(권장):** 나중에 실데이터와 구분하려면 `plan` 외에 메모용 표식(예: `account`에 `note text` 또는 별도 `is_beta boolean`)을 둘지 선택 — 없어도 되지만 teardown 편의상 추천. (원하면 A 마이그레이션에 `is_beta boolean default false` 한 컬럼 추가.)

## 9. 주의·엣지

- **`not null default 'inactive'` 함정:** 컬럼 추가 즉시 기존행이 inactive. **A 마이그레이션 안에서 바로 백필**(그 사이 헬퍼 미변경이라 잠금은 안 켜짐 = 무해 윈도우).
- **A/B 분리 이유:** A는 잠금 안 켜짐(헬퍼 그대로) → 안전히 배포·검증. B에서만 잠금 → 문제 시 B만 롤백.
- **헬퍼 확장은 재작성 금지:** 현재 `auth_account_id()` 정의를 그대로 가져와 subscription 3줄만 AND. 조인·`t.auth_user_id` 등 **실제 컬럼명 확인**(스펙의 컬럼명은 예시).
- **오프보딩 겸수:** `trainer.active=false` 잠금도 이 헬퍼 경로 → 마스터 §7 오프보딩 RLS 항목이 여기서 함께 정리됨.
- **크론 불필요:** 만료는 조회 시점 `now()` 비교라 자동. 별도 스케줄러 없음.
- **B2 예고:** `plan='premium'`은 이번엔 아무것도 게이트 안 함(회원관리 잠금은 B2에서 `auth_account_plan()`으로). 지금은 컬럼만 준비.
- **B4 예고:** 유료 전환 시 `billing-webhook`이 `subscription_status`/`current_period_end`를 갱신(이 필드 그대로 재사용).
