# v2 스펙 — 회원 앱 S2 (회원 홈 · 열람) · 구현

> 성격: **구현 스펙**(클로드코드 실행용). 전제 = S1(인증·RLS) 완료·배포·격리 실증 통과.
> 스코프: `/m/<member_token>` 회원 라우트 — 휴대 끝4 로그인(S1 라우트 재사용) → **읽기전용 홈**(내 프로필 · 수업일지 타임라인 · 인바디 추이). 자가입력 없음.
> ★S2에서 반드시 닫는 것: S1 §5 **컬럼 누출**(회원이 자기 행 열면 `member_note`·`quit_reason`·`raw_voice_text`까지 보임) → **안전 뷰**로 교체.
> 흐름: 이 스펙 → 트레이너 → 클로드코드(diff) → 트레이너(git diff) → 웹Claude(검토) → 폰 검증. 커밋은 트레이너 직접.

---

## 0. 한눈에

```
[회원] /m/<token> 열기
   │  기존 회원 세션 있으면 → 바로 홈
   │  없으면 → "휴대폰 뒤 4자리" 입력 → POST /api/member-auth(S1) → setSession → 홈
   ▼
[홈] (읽기전용 · 회원 JWT로 '안전 뷰'만 조회)
   · 프로필 헤더   ← member_me         (이름·목표·담당 트레이너)
   · 수업일지 타임라인 ← member_workout_log (날짜 + ai_summary)   ← raw_voice_text 제외
   · 인바디 추이   ← member_inbody      (측정일 + 수치 + 증감색)  ← note 제외
```

**핵심 안전 원리:** 회원은 원시 테이블(`user_table` 등) SELECT를 **아예 안 받는다**(S1의 raw 회원 정책은 제거). 대신 **안전 컬럼만 담은 뷰 3개**(내부에서 `auth_member_id()`로 자기 행만 필터)에만 접근 → 트레이너 메모·음성원본 컬럼은 **구조적으로 못 가져옴.** S1의 격리(트레이너 데이터 0행)는 그대로 유지되고, 여기에 **컬럼 차단이 더해져 더 강해짐.**

## 1. PART A — SQL (안전 뷰 · §5 해소) · 기록본 `docs/migrations/2026-07-16-member-views.sql`

> S1의 raw 회원 SELECT 정책 3개를 **제거**하고, 안전 컬럼 뷰 3개로 교체. 멱등. **security definer 뷰**(`security_invoker=false`)라 뷰가 기반 테이블 RLS를 우회해 읽되, 뷰 내부 `where ... = auth_member_id()`로 자기 행만 스코프(`auth.uid()`는 정의자 뷰 안에서도 호출자 JWT 기준이라 회원별로 정확히 갈림).
>
> ⚠️ **왜 굳이 definer 뷰인가(중요 · Supabase 경고 예상):** 회원은 원시 테이블 행 접근이 없으니(정책 제거) `security_invoker=true`면 뷰가 회원 권한으로 돌아 **0행**이 된다. 그래서 뷰 소유자(postgres) 권한으로 읽고 `where auth_member_id()`로 스코프하는 **definer 뷰가 필수**. 대신 Supabase 대시보드 **Security Advisor가 "security definer view"를 경고**로 띄우는데, 이건 **의도된 설계**(컬럼 은닉 목적)라 무시해도 됨 — 뷰가 단순 SELECT라 주입면 없음. 전제: **Postgres 15+**(`security_invoker` 옵션 · Supabase 기본).

```sql
-- =============================================================================
-- 회원 앱 S2 — 회원 열람용 안전 뷰 (§5 컬럼 누출 차단)
-- 실행일: 2026-07-16 · 실행: Supabase SQL Editor(수동) · 이 파일 = git 기록본
--
-- ★ 하는 일:
--   1) S1의 raw 회원 SELECT 정책 3개 제거(원시 테이블에 회원 직접 접근 없앰).
--   2) 안전 컬럼만 담은 뷰 3개 생성(내부에서 auth_member_id()로 자기 행만).
--   3) authenticated에 뷰 SELECT 부여. (원시 테이블은 여전히 트레이너 스코프 → 회원=0행)
-- ★ 제외 컬럼(회원에 노출 금지): user_table.member_note·quit_reason·phone_number·member_token·member_auth_id 등,
--   daily_workout_log.raw_voice_text, inbody_log.note.
-- 전제: S1(auth_member_id() 헬퍼) 완료. 멱등: drop policy if exists / create or replace view.
-- 롤백: 파일 하단.
-- =============================================================================


-- ========== 1) S1 raw 회원 정책 제거 (뷰로 대체) ==============================
drop policy if exists "member_read_own_user"      on user_table;
drop policy if exists "member_read_own_daily_log" on daily_workout_log;
drop policy if exists "member_read_own_inbody"    on inbody_log;


-- ========== 2) 안전 뷰 3개 (security definer · 내부 auth_member_id() 스코프) ===
create or replace view member_me
  with (security_invoker = false) as
  select u.id, u.name, u.goal, u.goal_deadline, t.name as trainer_name
  from user_table u
  left join trainer t on t.id = u.trainer_id
  where u.id = auth_member_id();

create or replace view member_workout_log
  with (security_invoker = false) as
  select id, created_at, ai_summary
  from daily_workout_log
  where user_id = auth_member_id();

create or replace view member_inbody
  with (security_invoker = false) as
  select id, measured_at, weight, skeletal_muscle, body_fat_mass,
         body_fat_pct, bmr, visceral_fat_level
  from inbody_log
  where user_id = auth_member_id();


-- ========== 3) 뷰 SELECT 권한 (회원=authenticated만 · anon 차단) ==============
grant select on member_me, member_workout_log, member_inbody to authenticated;
-- anon(비로그인)엔 불필요 — 혹시 Supabase 기본권한으로 새 뷰에 anon select가 붙어도
-- auth_member_id()가 NULL이라 0행이지만, 명시적으로 회수해 표면 줄임.
revoke select on member_me, member_workout_log, member_inbody from anon;


-- =============================================================================
-- 검증
-- (A) 뷰 3개 생성 확인:
--   select table_name from information_schema.views
--     where table_schema='public' and table_name like 'member_%';
-- (B) ★회원 세션(S1 access_token)으로 (PowerShell REST):
--   - GET /rest/v1/member_me            → 1행(내 프로필, 안전컬럼만)
--   - GET /rest/v1/member_workout_log   → 내 일지(ai_summary 있고 raw_voice_text 없음)
--   - GET /rest/v1/member_inbody        → 내 인바디(note 없음)
--   - GET /rest/v1/user_table?select=member_note  → 0행 또는 컬럼 접근 불가(누출 없음 확인)
-- =============================================================================


-- =============================================================================
-- ROLLBACK (S1 raw 정책 복원 + 뷰 제거)
-- drop view if exists member_me;
-- drop view if exists member_workout_log;
-- drop view if exists member_inbody;
-- create policy "member_read_own_user"      on user_table       for select to authenticated using (id = auth_member_id());
-- create policy "member_read_own_daily_log" on daily_workout_log for select to authenticated using (user_id = auth_member_id());
-- create policy "member_read_own_inbody"    on inbody_log        for select to authenticated using (user_id = auth_member_id());
-- =============================================================================
```

## 2. PART B — AuthGate 우회 (회원 라우트는 트레이너 게이트 밖)

`app/layout.js`가 전 라우트를 `<AuthGate>`로 감싸므로, `/m`이 트레이너 로그인 폼에 걸린다. **회원 라우트는 자체 인증**이니 게이트를 완전 우회. `components/AuthGate.jsx`에서 훅 선언(useState/useEffect) **뒤, `if (!ready)` 앞**에 아래를 추가:

```jsx
// (기존) const pathname = usePathname();
// ↓ 훅들 모두 선언된 뒤에 배치(Rules of Hooks). 회원 라우트는 트레이너 세션 판정 자체를 건너뜀.
const isMemberRoute = pathname === "/m" || pathname.startsWith("/m/");
if (isMemberRoute) return <>{children}</>;
```

> 효과: `/m/<token>`은 트레이너 세션 로딩("불러오는 중…")·로그인 폼·강제 비번변경과 무관하게 children만 렌더 → 회원 페이지가 자기 흐름대로 뜬다. 기존 `/signup` 공개 처리는 그대로.

## 3. PART C — 회원 전용 Supabase 클라 `lib/memberSupabase.js` (신규)

트레이너용 `lib/supabaseClient.js`와 **세션 저장소를 분리**해야 함(같은 storageKey면 회원 로그인이 트레이너 세션을 덮어씀). 별도 `storageKey` + 세션 지속/자동갱신:

```js
// lib/memberSupabase.js — 회원 앱(/m) 전용 Supabase 클라. 트레이너 세션과 저장소 분리.
import { createClient } from "@supabase/supabase-js";

const url = process.env.NEXT_PUBLIC_SUPABASE_URL;
const key = process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY;

// storageKey 분리 → 같은 브라우저에서 트레이너 세션과 충돌 안 함. 세션 지속+자동갱신(재방문 유지).
export const memberSupabase =
  url && key
    ? createClient(url, key, {
        auth: {
          storageKey: "pt-member-auth",
          persistSession: true,
          autoRefreshToken: true,
        },
      })
    : null;
```

> 데모 모드(키 없음)면 null → 페이지에서 가드(트레이너 앱 패턴과 동일).

## 4. PART D — 회원 페이지 `app/m/[token]/page.jsx` (신규)

로그인 게이트(끝4 입력 → S1 라우트 → setSession) + 홈(안전 뷰 3개 조회). 아래는 **로직 골격**(완전) — 프레젠테이션은 기존 디자인 시스템(레드 브랜드·`ui/` 프리미티브·큰 글씨)으로 맞춰 채운다.

```jsx
"use client";

import { useEffect, useState, useCallback } from "react";
import { useParams } from "next/navigation";
import { memberSupabase } from "@/lib/memberSupabase";

export default function MemberHome() {
  const { token } = useParams();
  const [phase, setPhase] = useState("checking"); // checking | login | home
  const [me, setMe] = useState(null);
  const [logs, setLogs] = useState([]);
  const [inbody, setInbody] = useState([]);
  const [last4, setLast4] = useState("");
  const [busy, setBusy] = useState(false);
  const [err, setErr] = useState("");

  const loadHome = useCallback(async () => {
    if (!memberSupabase) return;
    const [meRes, logRes, inbodyRes] = await Promise.all([
      memberSupabase.from("member_me").select("*").maybeSingle(),
      memberSupabase.from("member_workout_log").select("*").order("created_at", { ascending: false }),
      memberSupabase.from("member_inbody").select("*").order("measured_at", { ascending: true }),
    ]);
    setMe(meRes.data ?? null);
    setLogs(logRes.data ?? []);
    setInbody(inbodyRes.data ?? []);
    setPhase("home");
  }, []);

  // 기존 세션 있으면 바로 홈, 없으면 로그인
  useEffect(() => {
    if (!memberSupabase) { setPhase("login"); return; }
    let alive = true;
    memberSupabase.auth.getSession().then(({ data }) => {
      if (!alive) return;
      if (data.session) loadHome();
      else setPhase("login");
    });
    return () => { alive = false; };
  }, [loadHome]);

  const submit = async () => {
    if (busy) return;
    if (!/^\d{4}$/.test(last4)) { setErr("휴대폰 뒤 4자리를 입력하세요."); return; }
    setBusy(true); setErr("");
    try {
      const res = await fetch("/api/member-auth", {
        method: "POST",
        headers: { "Content-Type": "application/json" },
        body: JSON.stringify({ token, phoneLast4: last4 }),
      });
      const json = await res.json().catch(() => ({}));
      if (!res.ok || !json.ok) { setErr(json.error || "확인에 실패했습니다."); setBusy(false); return; }
      await memberSupabase.auth.setSession({
        access_token: json.access_token,
        refresh_token: json.refresh_token,
      });
      await loadHome();
    } catch {
      setErr("네트워크 오류. 잠시 후 다시 시도하세요.");
    }
    setBusy(false);
  };

  const signOut = async () => {
    if (memberSupabase) await memberSupabase.auth.signOut();
    setMe(null); setLogs([]); setInbody([]); setPhase("login");
  };

  if (phase === "checking") return <ScreenMsg>불러오는 중…</ScreenMsg>;
  if (phase === "login")
    return <LoginCard last4={last4} setLast4={setLast4} onSubmit={submit} busy={busy} err={err} />;
  return <HomeView me={me} logs={logs} inbody={inbody} onSignOut={signOut} />;
}
```

**화면 컴포넌트(같은 파일 내 정의 or `components/member/`로 분리 — 클로드코드 판단):**

- `ScreenMsg` — 중앙 로딩/안내 한 줄.
- `LoginCard` — 로고/이름 + **"휴대폰 뒤 4자리"** 숫자 입력(`inputMode="numeric"`, maxLength 4, Enter 제출) + 확인 버튼(`Button variant="primary" fullWidth`) + 에러줄. 톤은 AuthGate 로그인 카드 재사용감(레드).
- `HomeView`:
  - **프로필 헤더** — `me.name` 크게 + "목표 {me.goal}" + "담당 {me.trainer_name}"(있을 때). `me.goal_deadline` 있으면 D-day 보조표기(선택).
  - **수업일지 타임라인** — `Eyebrow`(아이콘 `NotebookPen`) "내 수업일지" + `logs`를 날짜 내림차순 카드로: 상단 `created_at`(KST 날짜) + `ai_summary` 본문. 비면 `EmptyState`("아직 기록된 수업일지가 없어요").
  - **인바디 추이** — `Eyebrow`(아이콘 `Scale`) "인바디 변화" + `inbody`(오래된→최신)로 지표별 최신값 + **직전 대비 증감색**. 시각 패턴은 기존 `PtInbodyTab`의 `deltaTone`(good=레드강조/bad=rose/flat=muted)·`Sparkline` 재사용(공용 필요하면 `ui/`로 추출). 비면 `EmptyState`.
  - 하단에 작게 **로그아웃**(선택 · AuthGate 로그아웃 버튼 결).
- 지표 라벨·goodDir은 기존 `lib/labels`의 `INBODY_FIELDS` 재사용(체중·골격근량·체지방량·체지방률·기초대사량·내장지방레벨).

> 톤: 트레이너 앱과 같은 토큰(`bg`·`card`·`ink`·`line`·`primary`)이되 **회원용이라 글씨 크게·여백 넉넉·정보 밀도 낮게.** 회원은 매출/세일즈 요소 일절 없음.

## 5. 세션·엣지 처리 메모

- **재방문 유지:** `memberSupabase`가 `persistSession + autoRefreshToken`이라, 회원은 링크 재클릭만으로 로그인 유지(리프레시 토큰 만료 전까지). 만료 시 자동으로 `login` 단계로 떨어져 끝4 재입력.
- **토큰 폐기(S3 재발급/해지):** 트레이너가 `member_token`을 바꾸면 이전 링크로는 **재로그인 불가**(기존 세션은 만료까지 유효 — 즉시 끊어야 하면 후속에 강제 만료 추가).
- **딴 회원 링크를 같은 폰에서 열면:** 이미 세션 있으면 그 세션의 홈이 뜸(URL 토큰은 재로그인 때만 씀). MVP 허용 — 명시적 "다른 계정" 전환은 로그아웃 후 재입력.
- **데모 모드(키 없음):** `memberSupabase=null` → 바로 `login` 단계 + 안내(실제 로그인 불가). 트레이너 앱 데모 가드와 동일 결. **`submit()`도 `memberSupabase` null 가드**(위 골격은 `setSession`에서 터질 수 있으니 상단에서 막고 안내).
- **`me` null 가드:** `member_me`가 0행이면(세션-링크 엣지) `me`가 null → `HomeView`는 `me` 없을 때 "정보를 불러오지 못했어요 · 다시 로그인" 폴백(끝4 재입력 유도). `.maybeSingle()`이 0행에 에러 안 내므로 렌더 단계에서 가드.

## 6. 검증 절차

1. **PART A SQL** Supabase 실행 → 검증 (A) 뷰 3개 확인. **(B) 회원 access_token(S1에서 쓰던 것 or 재로그인)으로 REST**: `member_me` 1행 · `member_workout_log`에 `ai_summary` 있고 `raw_voice_text` 필드 없음 · `user_table?select=member_note` **누출 없음**.
2. **PART B~D 배포**(Vercel Ready) 후 폰/브라우저에서 **`https://pt-navigator.vercel.app/m/<테스트 토큰>`** 열기 → 끝4(`5678`) 입력 → 홈 렌더 확인(프로필·수업일지 1건·인바디 빈 상태).
3. **재방문 유지:** 새로고침/재접속 시 끝4 재입력 없이 홈 유지되나.
4. **회귀:** 트레이너 앱(`/`·`/admin`) 로그인·탭 정상(AuthGate 우회는 `/m`에만 적용 — 다른 라우트 무영향).
5. **격리 재확인:** 회원 홈에서 매출·타회원·트레이너 화면 접근 경로가 전혀 없음(라우트 분리 + 뷰만 조회).
