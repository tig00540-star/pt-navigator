## 0. 이번 세션 성과 (배포·검증 완료)

**S3 — 회원앱 링크 발급·복사·PT 종료(세션 완전 차단):**
- `MemberAppLink`(PT 뷰 상단 바) — [링크 생성](`crypto.randomUUID()`→`member_token` UPDATE·복사) / [링크 복사] / [PT 종료](2단계 확인).
- PT 종료 = 서버 라우트 `app/api/member-revoke/route.js`(service_role) — 회원 auth 유저 삭제 → 세션 무효 + `member_token`·`member_auth_id` null. 재로그인 불가 + 기존 세션 다음 요청부터 0행. **회원 기록(수업일지·인바디·계약)은 보존**(user_table.id에 매달림). 재발급하면 옛 기록 그대로 재개통.
- 파일: `components/views/MemberAppLink.jsx`·`app/api/member-revoke/route.js`·`lib/clipboard.js` + `app/page.jsx`(mapMemberRow에 member_token·member_auth_id 매핑)·`components/views/PTView.jsx`(바 삽입, `key={member.id}`).

**기능 ② — 보유 장비 등록 → OT/재등록 루틴 반영:**
- **②-A** 설정탭 '보유 장비' CRUD(`components/views/CenterMachineSettings.jsx`) — 대표(owner)/solo 편집·트레이너 열람. `center_machine`에 `kind`(머신/프리웨이트/맨몸)·`spec`(규격·중량범위, 덤벨은 한 항목+범위) 컬럼. **owner RLS**(insert/update/delete = `auth_account_id() and auth_is_owner()`), `brand` NOT NULL 해제, account_id 기본값·인덱스. 회원 등록폼 '보유머신' 필드는 제거(중앙관리로 이관).
- **②-B** `app/api/ot-brief/route.js` — 트레이너 JWT로 그 계정 `center_machine` 조회 → 프롬프트 앞에 **[보유 장비] 소프트 제약** 주입(우선 활용하되 목록 밖도 필요하면 제안 · 0개면 미부착 = 종전대로). 빌더 4개 무변경.
- 마이그레이션: `docs/migrations/2026-07-16-center-machine-manage.sql`·`2026-07-16-center-machine-spec.sql`. 스펙: `docs/v2-스펙-센터장비-루틴반영-구현.md`.
- **현장 검증 통과** — 대표 저장·수정·삭제, 덤벨+규격 표시, AI 루틴에 등록 장비 반영 확인.

## 1. 다음 = ③ 수업일지 구조화 정리 (종목별·날짜별 무게·횟수·세트)

**무엇:** 수업일지를 종목별로 자동 정리 — 예) "벤치프레스: 7/16 20kg×12·30kg×12 → 7/20 20kg×15·30kg×15" 처럼 종목별 날짜순 추이. → **회원앱에 '종목별 무게 증가 그래프'**로 붙이면 성과 가시화 = 재등록 유도 강력.

**왜 지금:** 로드맵 Phase 1. 음성일지 AI가 이미 종목·수치를 뽑고 있어 재료는 있음. 회원앱 심화·AI 게이트·자기진화와 결.

**이미 있는 것(재료):**
- `app/api/voice-log/route.js` — STT→요약. **출력 스키마 이미 `{ machines:[{name, detail, method}], feedback, homework }`** (종목명 + detail에 중량·세트가 자유텍스트로 들어감).
- `daily_workout_log`(user_id·contract_id·session_at·source[manual/voice/noshow]·`ai_summary`(정리 텍스트)·`raw_voice_text`·sent_at·voided). — 지금은 **텍스트라 종목별 집계 불가.**
- `components/views/PtWorkoutTab.jsx` `saveLog` — 저장 지점(여기서 daily_workout_log insert). 음성일지 프리뷰→일지채우기→저장(차감).
- `components/views/PtInbodyTab.jsx` — **지표별 추이 + delta색 + Sparkline** 패턴(종목별 무게 그래프에 그대로 재사용 가능).
- 회원앱 S2: `app/m/[token]/page.jsx` + 안전뷰(`member_workout_log`=created_at·ai_summary). 그래프 붙이면 여기 + **새 안전 뷰** 필요.

**새로 필요한 것:**
- **구조화 저장** — `daily_workout_log`에 `structured jsonb`(수업 행에 `[{exercise, sets:[{weight,reps}]}]`) 또는 별도 `workout_set` 테이블(정규화). ★설계 결정.
- **AI 출력 구조화** — voice-log가 detail을 자유텍스트 대신 `{weight, reps, sets}` 숫자 필드로 뽑게 강화(또는 저장 시 파싱).
- **종목명 정규화** — 벤치/인클라인벤치 등 canonical 통일.
- **종목별 집계 뷰/차트** — 트레이너 PT 뷰(운동일지 탭 추이) + 회원앱(그래프 · 새 definer 안전 뷰).

**착수 전 정할 것(웹Claude가 스펙 전에 물어볼 것):**
1. **저장 방식** — `daily_workout_log.structured jsonb`(간단·수업단위) vs `workout_set` 테이블(집계 SQL 깔끔·정규화). 내 잠정 추천 = jsonb로 시작(가벼움), 집계 무거워지면 테이블로.
2. **추출 시점** — voice-log 프롬프트를 구조화 출력으로 강화 vs 저장 시 detail 파싱. (voice-log 강화가 정확·재사용 쉬움.)
3. **손입력(manual)·노쇼 수업** — 구조 데이터 없음 → 그래프에서 제외/스킵 규약.
4. **종목명 정규화** — AI에 맡길지 vs 사전 매핑. (초기 = AI canonical + 트레이너 보정.)
5. **표시 위치·범위** — 트레이너 먼저 vs 회원앱까지. 회원앱이면 새 안전 뷰(민감컬럼 제외 원칙).
6. **트레이너 보정 UX** — AI 추출이 틀릴 때 고치는 인터페이스(수치 신뢰도 100% 아님 → "자동 + 보정" 전제).

**참고 문서:** `docs/v2-기능요청-가능성판정.md` §3 · `docs/pt-navigator-총정리.md` §3(voice-log) ·§4(스키마) · `docs/v2-실행순서-로드맵.md`.

## 2. 회원 앱 규약 (S1~S3에서 확립 · 회원앱 그래프 붙일 때 준수)

- **회원 데이터 노출 = 안전 뷰만**(원시 테이블 회원 정책 없음). 새 회원 화면은 definer 뷰로만, 민감 컬럼 제외. Supabase "security definer view" 경고는 의도된 설계(무시).
- **격리:** 회원 auth 유저는 trainer 행 없음 → `auth_account_id()`/`auth_is_owner()`=NULL → 매출·타회원·트레이너 데이터 원천 0행. 회원은 `auth_member_id()` 기반만.
- **회원 라우트=`/m`**, 회원 클라 `lib/memberSupabase.js`(storageKey 분리).
