# v2 · 스프린트 2 — 1차 OT 관찰 기록 (데이터 뼈대)

> 대상: Claude Code에 넘기는 구현 브리프
> 원칙(MASTERPLAN): "뼈대(데이터) 없이 AI부터 붙이면 헛돈다. 수동 입력 + DB 저장 뼈대를 먼저 깐다."
> 그리고 "입력하는 행위 자체가 사고의 시간(인지적 각인)" — 단, "입력 자체는 가볍게".
> ⚠️ 이 스프린트에는 AI가 없다. 순수하게 관찰 데이터를 입력받아 저장하는 배관만 만든다.

---

## 1. 목표

1차 OT 수업 직후, 트레이너가 회원을 관찰한 내용을 입력받아 `ot_log` 테이블에 저장한다.
지금까지 하드코딩(김철수 고정)이던 성향·관찰 데이터를, **실제 트레이너 입력 → DB**로 바꾸는 첫 걸음.
이 데이터가 스프린트 3(AI 종합 분석·등록 당위성 브리핑)의 재료가 된다.

## 2. 완료 기준 (Acceptance Criteria)

- [ ] 새 탭 "관찰 기록"이 추가된다 (음성일지 옆).
- [ ] 선택된 회원에 대해 3덩어리(움직임·반응·목적)를 입력할 수 있다.
- [ ] 저장 시 `ot_log`에 기록된다 (`user_id`, `ot_round=1`, `report` jsonb, `goal_type`, `goal_identified`).
- [ ] 같은 회원의 기존 1차 관찰이 있으면 불러와 수정(덮어쓰기)된다 — 중복 행이 안 생긴다.
- [ ] 자유입력/드롭다운 구분이 설계대로 반영된다.
- [ ] 키·supabase 미설정(데모 모드) 시 앱이 죽지 않고 안내로 폴백한다.
- [ ] `npm run build` 통과. 기존 5탭 기능은 그대로 동작.

## 3. 입력 스키마 (확정)

### ① 움직임 관찰 — 1~3개 반복 입력
| 필드 | 입력 방식 | 예시 |
|---|---|---|
| 관찰 내용 | 자유입력(textarea) | "오버헤드 프레스 시 견갑 상방회전 제한 / 승모근 긴장 높음" |
| 회원이 인식함? | 체크박스(boolean) | ☑ (수업 중 스스로 문제 인지) |
| 2차에 풀 것 | 자유입력 | "견갑 가동성 먼저, 그 후 프레스 재시도" |

> "2차에 풀 것" 필드는 스프린트 3의 **등록 당위성 브리핑**(1차 관찰 → 2차 증명 → 클로징 논리) 연결에 쓰인다. 불필요해 보여도 절대 빼지 말 것.

### ② 회원 반응·성향
| 필드 | 입력 방식 | 값 |
|---|---|---|
| 자극 인지도 | 드롭다운 | 잘 느낌 / 보통 / 잘 못 느낌 |
| 태도 태그 | 다중선택 | 겁많음 / 적극적 / 시키는것만 / 재미있어함 |
| 메모 | 자유입력 | 자유 |

### ③ 진짜 목적
| 필드 | 입력 방식 | 값 |
|---|---|---|
| 파악 여부 | 드롭다운 | 파악됨 / 불명확 |
| 목적 유형 | 드롭다운 | 외형변화 / 통증개선 / 건강·체력 / 기구사용법만 / 기타 |
| 상세 | 자유입력 | "가을 결혼식", "무릎 재활 후 복귀" 등 |

> 설계 철학: **핵심(관찰 내용·목적 상세)은 자유입력으로 사고를 유발**, **분류용(목적 유형·자극 인지도 등)은 드롭다운으로 빠르게.**

## 4. DB 저장 구조 (방식 A — 기존 `ot_log` 활용)

`ot_log`는 이미 설계돼 있으나 미사용 상태였다. 이번에 활성화한다.
전체 데이터는 `report(jsonb)`에 담고, **필터·통계에 자주 쓸 2개만 별도 컬럼**으로 승격한다.

### 마이그레이션 (Supabase SQL Editor에서 실행 — git 아님)
```sql
alter table ot_log
  add column if not exists goal_type text,
  add column if not exists goal_identified boolean;
```

### 저장 형태
```
ot_log 한 행 =
  user_id        : (선택된 회원 id)
  ot_round       : 1
  goal_type      : "appearance" | "pain" | "health" | "machine_only" | "other"
  goal_identified: true | false
  report (jsonb) : {
    movements: [
      { observation: string, memberAware: boolean, plan2nd: string }
    ],
    reaction: {
      stimulus: "well" | "normal" | "poor",
      attitudeTags: string[],   // ["timid","active",...]
      memo: string
    },
    goal: {
      identified: boolean,      // goal_identified와 동일값 미러
      type: string,             // goal_type와 동일값 미러
      detail: string
    }
  }
```
- `goal_type` / `goal_identified`는 `report.goal.type` / `report.goal.identified`를 **미러링**(조회 편의용 중복 허용).
- 드롭다운 라벨(한글)과 저장값(영문 키)을 분리 — 나중에 통계·AI가 값으로 다루기 쉽게.

### 업서트 규칙 (중복 방지)
- 탭 진입/회원 선택 시 → `ot_log`에서 `user_id = 선택회원 AND ot_round = 1` 최신 행을 조회.
- 있으면 폼에 프리필(수정 모드) → 저장 시 그 행 **update**.
- 없으면 → 저장 시 **insert**.
- (= 회원당 1차 관찰 1행 유지.)

## 5. 화면 / 컴포넌트

- 새 탭 **"관찰 기록"** 추가. `TABS` 배열에 항목 추가하고, **음성일지 탭 바로 옆**에 배치(수업 마무리 동선을 잇기 위함).
- 컴포넌트: `components/tabs/ObservationTab.jsx` (스프린트 1의 `VoiceLogTab` 분리 패턴 그대로).
- 부모(`app/page.jsx`)에서 현재 선택된 `member`를 prop으로 전달(음성일지와 동일 패턴). 저장은 `member.id`에 묶임.
- 이미 추출된 `Eyebrow`(components/ui) 재사용. 필요한 공용 요소가 더 있으면 복제 말고 추출(스프린트 1 원칙 유지) — 단 이번 슬라이스가 실제로 쓰는 것만.
- ⚠️ **1차 OT 탭(탭 1)은 이번에 건드리지 않는다.** (트레이너가 추후 정리 예정 — 그 계획과 충돌 방지. 관찰 기록을 독립 탭으로 두는 이유가 이것.)

## 6. 기술 주의사항

- **회원 미선택 상태 처리:** 선택된 회원이 없으면(혹은 데모 회원이면) 저장 버튼 비활성 + "회원을 먼저 선택하세요" 안내.
- **데모 폴백:** supabase 미설정 시 저장 대신 안내 문구. 앱이 죽지 않게 try/catch.
- **저장 피드백:** 저장 성공/실패 토스트(음성일지의 showToast 패턴 재사용 가능).
- **jsonb 저장:** supabase insert/update 시 `report`는 객체 그대로 넘기면 됨(문자열화 불필요).
- **커밋 분할:**
  1. `refactor:` 공용 요소 추출(있을 경우만)
  2. `feat:` 관찰 기록 탭 + 입력폼 UI (저장 없이 폼만)
  3. `feat:` ot_log 저장/업서트 연동
  (DB 마이그레이션 SQL은 코드가 아니므로 커밋과 별개로 Supabase에서 실행. 브리프에 SQL 명시됨.)
