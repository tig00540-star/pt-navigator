# v2 · 스프린트 3 설계 재료 메모 (착수 전 정리)

> 이 문서는 스프린트 3(저장된 관찰 → 실 AI 분석 + 등록 당위성 브리핑) 설계 시작 전,
> 트레이너와의 논의에서 나온 방향성을 모아둔 것. 아직 확정 스키마/브리프가 아님.
> 새 채팅으로 스프린트 3 시작 시 이 문서 + MASTERPLAN + 앞 두 스프린트 브리프를 함께 읽을 것.

---

## A. 보안 슬라이스에 추가할 요구사항 (마스터플랜 4순위 구체화)

**"트레이너가 로그인하면 본인 담당 OT/PT 회원만 보인다."**

- 이것은 별도 기능이 아니라 **보안(Auth + RLS) 작업의 자연스러운 결과물**이다.
- 필요 조각:
  1. `user_table`에 `trainer_id` 컬럼 (담당 트레이너 소유권). 마스터플랜 v2 예상 `trainer` 테이블과 연결.
  2. Supabase Auth 로그인 (트레이너 계정 = trainer_id 매핑).
  3. RLS 정책을 `trainer_id = 로그인 사용자` 기준으로 잠금 (DB 레벨 원천 차단, 프론트 필터링 아님).
  4. `/admin` 권한 분리 — 관리자는 전부 보되, 트레이너는 본인 것만.
- **시점:** 스프린트 3(AI) 다음. 지금 anon 전면 개방 상태가 AI 개발 중엔 편하므로, 보안은 마지막에 한 번에 조인다.
- **미리 해둘 값어치(선택):** `user_table`에 `trainer_id` 컬럼만 미리 추가(값은 나중). 스프린트 2에서 ot_log 컬럼 미리 판 것과 같은 논리.
- **보안 설계 때 트레이너에게 물어 정할 것(운영 방식 의존):**
  - 담당 배정을 누가/언제 하나? (데스크 등록 시 / 관리자 배정)
  - 담당 변경 처리 (트레이너 퇴사·회원 이관)
  - 한 회원을 여러 트레이너가 볼 경우 (대타 수업 등) 존재 여부
  - 관리자 전체 열람 vs 트레이너 본인만 — 권한 차등

## B. 1·2차 OT 피드백을 "하나의 대화 흐름"으로 통합 (스프린트 3 방향)

트레이너 요구: 지금은 대본·루틴·팁이 조각조각인데, **60분 수업이 하나의 이어지는 대화처럼**
흘러가도록 AI가 통합적으로 안내해줬으면 함. "이 수업은 이런 방향으로 흘러가야 한다"는 서사(narrative) 형태.

- 즉 AI 출력이 "항목 나열"이 아니라 **"수업 시간대별 대화 아크(arc)"**가 되어야 함.
  예: 도입(라포·목표 재확인) → 중반(움직임 관찰을 회원 스스로 인식시키기) → 후반(자극 체감 확인·가치 각인) → 마무리(다음 수업 예고·클로징 빌드업).
- 저장된 1차 관찰 데이터(ot_log)를 재료로, 2차 OT의 대화 흐름을 AI가 설계.
- ⚠️ **마스터플랜 ⭐⭐ 철학 유지:** AI가 대본을 통째로 던지면 트레이너가 사고 정지 → 하향평준화.
  "정답 대본"이 아니라 "이 회원엔 이 흐름이 왜 맞는지(근거+방향)"를 주고,
  트레이너가 자기 스타일로 채우게. **스파링 파트너지 스크립트 낭독기가 아니다.**
- 1차 관찰의 "2차에 풀 것" 필드 → 2차 대화 흐름의 중심축으로 연결(등록 당위성 브리핑과 한 몸).

## C. 클로징 멘트 — 방향성과 반드시 지킬 선

트레이너 요구: 클로징이 **회원에게 큰 공감을 주고, 거절하기 힘들 만큼 인상 깊게.**

**살릴 방향 (마스터플랜 세일즈 심리 설계와 정렬):**
- 강한 클로징의 원천은 "판매 압박"이 아니라 **"이 트레이너가 나를 정확히 이해했다"는 공감**이다.
  회원 본인도 몰랐던 몸 상태·목표를 짚어주면 신뢰가 생기고, 그 신뢰가 등록으로 이어짐.
- 그래서 클로징은 **1차 관찰 → 2차 증명 → "혼자서는 이걸 못 잡는다"**로 이어지는
  개인화된 논리여야 강력함. 일반 멘트가 아니라 "당신의 데이터"로 말하는 것.
- 마스터플랜 원칙 재확인: "할인해요"(파는 말) ❌ → "지금 멈추면 원점"(책임의 말) ✅.
  손실 회피 + 회당 가치 환기 + 최소 필요량 옵션.

**⚠️ 반드시 지킬 선 (현장 배포 전 검토 필수 — 마스터플랜 5단계 체크리스트와 연결):**
- "거절 못 할 충동구매"를 조작적 압박(FOMO 남발, 허위 긴급성, 죄책감 유발)으로 구현하면 안 됨.
  → 마스터플랜의 "죄책감 없는 세일즈" 철학과 정면충돌. 단기 등록은 되도 신뢰·재등록이 무너짐.
- 의료·과장 표현 금지(치료/완치 등) — 마스터플랜 5단계 체크리스트 명시.
- 목표: "거절 못 하게 몰아붙이는 멘트"가 아니라 **"거절할 이유가 사라질 만큼 깊이 이해받았다고 느끼게 하는 멘트."**
  전자는 압박, 후자는 공감. 결과(강한 클로징)는 같아 보여도 수단이 정반대이고, 지속가능한 건 후자.
- 스프린트 3에서 클로징 AI 설계 시, "설득력"과 "윤리적 선"을 함께 프롬프트에 박을 것.

## D. [나중 목록] 관리자 "행동 가능한" 목표 대시보드 (실데이터화 이후)

트레이너 구상: 관리자 화면을 깔끔·심플하게. 핵심은 죽은 숫자(매출 얼마)가 아니라
**행동 가능한 정보** — 관리자가 보고 즉시 개입할 수 있는 화면.

구체 구상:
- **FC팀 / PT팀**으로 나눠, 각 팀별 **회원권 재등록 대상 / PT 재등록 대상** 회원 리스트.
- 그 대상 중 **금일 수업이 잡힌 회원**을 짚어주고 → **그로 인한 예상 매출 파이** 제시.
- (관리자가 "지금 그 회원 수업 중인 트레이너"에게 클로징 개입 가능 = 행동 가능한 대시보드.)

⚠️ **지금 만들면 안 되는 이유(=나중 목록인 이유):** 이 화면은 현재 없는 4개 테이블 위에 선다.
- FC/PT 팀 구분 → `trainer` 테이블 + 팀 필드 (보안 슬라이스)
- 재등록 대상 판정 → 등록 만료일·잔여 세션 = `member_status` (실데이터화)
- 금일 수업 잡힌 회원 → `session_log` (실데이터화)
- 예상 매출 파이 → `payment` (실데이터화)
지금 만들면 전부 하드코딩(가짜 숫자)로 회귀 = 마스터플랜 "관리자 KPI 전부 가짜" 상태로 되돌아감.
마스터플랜 원칙 위반("뼈대 없이 붙이면 헛돈다").

**언제:** 보안(trainer/팀) + 실데이터화(payment·session_log·member_status) 슬라이스가 끝난 뒤.
이 화면은 별도 기능이 아니라 **그 조각들이 다 모인 종착지**다. 지금은 목적지로만 기록해두고,
payment/session 테이블 설계 시 "이 대시보드를 위해 무엇이 필요한가"의 기준으로 삼는다.

**관리자 화면 정리 원칙(방향만):** "관리자가 매일 아침 3초 보고 오늘 뭘 할지 아는 화면."
실데이터 붙는 시점에 트레이너에게 "관리자가 매일 실제로 뭘 보고 싶은지" 물어 설계.
