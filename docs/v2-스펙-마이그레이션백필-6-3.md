# v2 스펙 — §6-3 마이그레이션 백필 (README + appointment·reg_* 기록본)

> 근거: 리포트 §6-3, §10-4 결정("둘 다: README로 증분 로그 명시 + appointment·reg_*·베이스 테이블 DDL 기록본 백필").
> 성격: **DB 기록본 작성 스펙**. 앱 코드 변경 없음. 새 파일 3개(README 1 + SQL 2). Claude Code는 파일 생성 시 `git add` 먼저.
> 환경/커밋 규약 동일(PowerShell 구버전). 커밋은 대수가 직접.

## ⚠️ 이 작업의 핵심 원칙 (먼저 읽을 것)
`appointment`·`session_log`은 **Supabase UI로 만든 테이블**이라 정확한 타입·제약·**RLS 정책**을 코드만 보곤 100% 확신할 수 없다. 코드 추론 DDL을 그대로 기록하면 **재구축 시 실제와 다른 스키마(특히 RLS)가 생겨 멀티테넌트 격리가 깨질 위험**이 있다(리포트 §7: 격리 100% RLS 의존).

그래서 절차는 **"추출 → 대조 → 기록"**:
1. 아래 **추출 쿼리**를 Supabase SQL Editor에서 돌려 실제 DDL/RLS/인덱스를 뽑는다(대수).
2. 아래 **재구성 초안**과 대조해 타입·제약·정책명·정책조건 차이를 초안에 반영한다.
3. 반영된 **최종본**을 파일로 저장·커밋한다. (초안을 무검증 커밋 금지.)

전부 **멱등**(`if not exists` / `add column if not exists` / `drop policy if exists→create`)이라, 실제 존재하는 DB에 돌려도 **no-op**이고 빈 DB엔 생성된다.

## 스타일 표준 (기존 파일에서 확정)
- 헤더 주석: 무엇/실행일/실행=Supabase SQL Editor 수동/`이 파일 = git 기록본`/멱등/전제/롤백.
- `create table if not exists` · `account_id uuid not null default auth_account_id() references account(id)` · `trainer_id uuid default auth.uid() references trainer(id)` · `user_id ... references user_table(id) on delete cascade`.
- `alter table X enable row level security;`
- `create index if not exists ...;`
- RLS: `drop policy if exists "..." on X; create policy "..." on X for all to authenticated using(account_id = auth_account_id()) with check(...);`
- 하단 검증 쿼리(읽기 전용) + ROLLBACK 주석.

## Part A — `docs/migrations/README.md` (신규 · 그대로 저장 가능)
> 이건 라이브 스키마 의존 없음 → **완성본**. 바로 저장·`git add docs/migrations/README.md`.

```markdown
# docs/migrations — 스키마 증분 로그

이 폴더는 **Supabase SQL Editor에서 수동 실행한 SQL의 git 기록본**이다.
각 파일은 실행 시점의 증분(테이블 신설·컬럼 추가·RLS 변경)이며, 파일명은 `YYYY-MM-DD-<주제>.sql`.

## ⚠️ 이 폴더만으로는 "처음부터 재구축"이 안 된다
초기 베이스 테이블 일부는 **Supabase 대시보드 UI로 생성**돼 `create table` DDL이 이 폴더에 없다:

- `user_table` · `ot_log` · `daily_workout_log` · `center_machine` (베이스 4개)
- `session_log` 본체 · `appointment` (UI 생성 · 컬럼/정책은 이후 증분 파일에 일부만 기록)

이 테이블들의 **컬럼 다수·RLS 초기 정책**은 sprint 산문 문서와 라이브 DB에만 존재한다.
(RLS 잠금 자체는 `2026-07-08-step7*` 파일들에 기록돼 있다.)

## 재구축(새 Supabase 프로젝트) 절차
1. **베이스 스키마 스냅샷을 먼저 적용**한다. 권위 있는 원본은 라이브 DB 덤프:
   - `supabase db dump --schema public > base-schema-snapshot.sql` (Supabase CLI), 또는
   - 대시보드 → Database → 각 테이블 "Definition" 복사, 또는 `pg_dump --schema-only`.
   - 이 스냅샷을 `0000-base-schema-snapshot.sql`로 보관(민감정보 없음 · 스키마만).
2. 그 위에 이 폴더의 증분 파일들을 **날짜순**으로 적용한다(멱등이라 중복 무해).

> 즉 이 폴더 = **레이어(증분)**, 완전 재구축 = **베이스 스냅샷 + 이 레이어**.

## 규약
- **멱등 필수**: `create table if not exists` · `alter table ... add column if not exists` · `drop policy if exists → create policy`. 실제 존재하는 DB에 재실행해도 no-op.
- 트레이너-facing RLS 표준: `using (account_id = auth_account_id())`. 회원-facing: `auth_member_id()`.
- 각 파일 하단에 **검증 쿼리(읽기 전용)** + **ROLLBACK** 주석.
- 신설 테이블은 반드시 `alter table X enable row level security;`(anon 차단).
```

## Part B — `docs/migrations/2026-07-19-appointment-record.sql` (기록본 · 대조 필요)
> 리포트 §6-3 급소: `appointment` DDL이 어디에도 없어 **재구축 시 스케줄 탭이 통째로 죽는다**. 코드 사용 컬럼: `id, account_id, trainer_id, user_id, start_at, status, log_id` (ScheduleBoard·PastDueAppointments). status 값: `booked`·`done`·`canceled`. `log_id` → `daily_workout_log.id`(완료 시 연결).

### B-1. 먼저 추출 (대수 · Supabase SQL Editor)
```sql
-- 컬럼·타입·기본값
select column_name, data_type, is_nullable, column_default
from information_schema.columns
where table_schema='public' and table_name='appointment' order by ordinal_position;
-- 제약(FK·CHECK·PK)
select conname, pg_get_constraintdef(oid) from pg_constraint where conrelid='appointment'::regclass;
-- 인덱스
select indexname, indexdef from pg_indexes where tablename='appointment';
-- RLS 정책(정책명·cmd·조건) ← 특히 이거: account 스코프인지 trainer 스코프인지 확인
select policyname, cmd, qual, with_check from pg_policies where tablename='appointment';
select relrowsecurity from pg_class where relname='appointment';
```

### B-2. 재구성 초안 (inbody_log 템플릿 기반 — **위 추출 결과와 대조 후 확정**)
```sql
-- =============================================================================
-- appointment 기록본 — UI로 생성됐던 예약 테이블의 DDL·RLS를 git에 백필
-- 실행일: 2026-07-19 · 실행: Supabase SQL Editor(수동) · 이 파일 = git 기록본
--
-- ★ 이 파일이 하는 일:
--   appointment(주간 예약 · 스케줄 탭) 테이블/인덱스/RLS를 멱등 기록.
--   기존 라이브 DB엔 이미 존재 → 재실행 시 no-op. 재구축 DB엔 생성.
--   ⚠️ 이 파일은 라이브 스키마 추출(B-1)과 대조해 타입·제약·정책을 확정한 뒤 커밋할 것.
--
-- 전제: account/trainer · auth_account_id() 헬퍼 · daily_workout_log 존재.
-- 멱등: create table if not exists · create index if not exists · drop policy if exists→create.
-- =============================================================================

create table if not exists appointment (
  id          uuid primary key default gen_random_uuid(),
  created_at  timestamptz not null default now(),
  account_id  uuid not null default auth_account_id() references account(id),
  trainer_id  uuid default auth.uid() references trainer(id),          -- 예약 담당 트레이너
  user_id     uuid not null references user_table(id) on delete cascade,
  start_at    timestamptz not null,                                    -- 예약 시각
  status      text not null default 'booked',                         -- booked · done · canceled
  log_id      uuid references daily_workout_log(id) on delete set null -- 완료 시 연결된 수업일지
);
alter table appointment enable row level security;

-- 주간 조회 인덱스(계정·시각 범위)
create index if not exists appointment_account_start_idx on appointment (account_id, start_at);

-- RLS — 계정 스코프(inbody_log/daily_workout_log 동일 수준). ⚠️ 실제 정책명·조건은 B-1로 확정.
drop policy if exists "auth_all_appointment" on appointment;
create policy "auth_all_appointment"
  on appointment for all to authenticated
  using (account_id = auth_account_id())
  with check (account_id = auth_account_id());

-- =============================================================================
-- 검증: 기대 = RLS on · 정책 1행(cmd=ALL) · start_at 인덱스 존재.
-- select relrowsecurity from pg_class where relname='appointment';
-- select policyname, cmd from pg_policies where tablename='appointment';
-- ROLLBACK(신설 재구축 DB에서만): drop table if exists appointment cascade;
-- =============================================================================
```

**대조 체크리스트(B-1 결과 vs 초안):**
- `start_at`/`log_id`/`status` 타입 일치? (timestamptz / uuid / text)
- `status`에 CHECK 제약 있나? 있으면 초안에 `check (status in ('booked','done','canceled'))` 추가.
- RLS가 **account 스코프**가 맞나, 아니면 **trainer 스코프**(`trainer_id = auth.uid()`)인가? → 실제 정책조건 그대로 반영(정책명도 실제 것으로).
- 추가 인덱스(예: `user_id`·`status`)가 있으면 기록.

## 범위 결정 — 베이스 4개 테이블 전체 DDL
§10-4는 베이스 테이블 DDL 백필도 언급했지만, `user_table`·`ot_log`·`daily_workout_log`·`center_machine`는 **컬럼이 여러 sprint에 흩어져 있고 RLS 초기 정책도 복잡**해 손으로 재구성하면 어긋날 위험이 크다(RLS 격리 사고 위험). 그래서 **손 DDL 대신 스냅샷 덤프**를 권장(README Part A에 절차 포함):
- `supabase db dump --schema public` 또는 대시보드 Definition 복사 → `0000-base-schema-snapshot.sql`로 1회 보관.
- 이게 베이스 4개 + `session_log` 본체 + `appointment` 초기형까지 **한 번에 정확히** 잡는다. 이후 이 폴더 증분이 그 위에 얹힘.
- 즉 **appointment·reg_* 기록본(B·C)은 "증분 레이어"로서 계속 기록**하되, 베이스 전체는 스냅샷이 담당 → 이중 안전(스냅샷 있으면 재구축 완결, 없어도 B·C로 급소는 커버).

이 결정으로 "재구축 시 스케줄 탭 죽음"이라는 급소(§6-3)는 **B(appointment)만으로도 해소**되고, 베이스 전체 정합성은 스냅샷으로 완결된다.
