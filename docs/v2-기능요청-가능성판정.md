# v2 기능 요청 4건 — 가능성 판정 · 접근 제안

> 성격: 트레이너 요청 4건의 **구현 가능성 판정 + 어떻게 갈지** 방향 메모(구현 스펙 아님). 현재 코드 배선 확인 기준.
> 요약 판정: ①조건부(데이터 소유권 분리 선행) · ②높음(거의 반배선) · ③중~높음(구조화 저장이 관건) · ④가능하나 대형 에픽(뼈대는 다 있음).

---

## 1. 회원가입 시 센터 선택 / 없으면 개인트레이너 / 이직 시 자료 이전 (이탈 방지)

**판정: 가능하나 "무엇을 가져가나"를 먼저 갈라야 함 (조건부 · 중대형).**

**이미 있는 것:** `account`(type=center/solo) — **개인트레이너=solo는 이미 존재**. `trainer`(account_id·role·active). 센터 합류는 지금 원장 초대(`create-trainer`)로만.

**새로 필요한 것:**
- **가입 시 센터 합류** = 센터 코드/초대 승인 메커니즘(아무나 "그 센터 소속"이라 주장 못 하게 — 원장 승인 or 초대코드). 지금 self-가입은 solo만 깔끔.
- **★이직 자료 이전** = 데이터 소유권 분리 설계(아래).

**★꼭 먼저 정할 것 — 데이터가 두 종류다:**
- **(a) 센터 소유** = 회원 PII·수업일지·인바디·매출·계약. `account`에 묶임 → 이직 시 **안(못) 가져감.** 회원은 센터 자산이고, 트레이너가 회원 데이터를 들고 나가는 건 대표가 오히려 반대할 부분(+개인정보 이슈).
- **(b) 트레이너 소유(포터블)** = 본인 클로징 성향/통계·자기진화 코퍼스(익명)·개인 라이브러리(루틴·대본 템플릿)·자격·프로필. **이게 진짜 '이탈 방지' 자산** — auth 유저(정체성)에 묶으면 센터를 옮겨도 유지.

**구조 변화:** 지금 `trainer` 한 행이 "정체성 + 센터 멤버십"을 한 몸에 갖고 있음. 이직을 깨끗이 하려면 **정체성(auth user + 포터블 자산) ↔ 멤버십(account 소속·역할)**을 분리해야 함 → 중대형 데이터 모델 작업. (B3 카카오/온보딩 프로비저닝 재편과 같이 가면 시너지.)

**결정 요청:** "기존 자료 그대로"의 범위 = **(b) 개인 성장자산만**인지, **(a) 회원 데이터까지**인지. 내 권장은 **(b)만** — 이탈 방지 효과는 거기서 나오고, 회원 데이터 이전은 법적·대표관계상 지뢰. 이 답에 따라 난이도가 갈림.

## 2. 센터 머신·덤벨 등록 → 1·2차 OT·재등록 루틴에 자연 반영

**판정: 높음 (거의 반배선됨). 가장 먼저 하기 좋음.**

**이미 있는 것:** `center_machine`(brand·name) 테이블 존재 + 회원 등록폼에서 픽 + **`ot-brief`가 이미 회원 `machines`를 프롬프트에 주입**(route.js). 즉 "머신 → AI 루틴" 배관 자체가 이미 있음.

**새로 필요한 것:**
- **대표용 center_machine 관리 UI** — 지금은 시드 SQL로 추정(등록폼에 "center_machine 시드 필요" 힌트). 대표가 우리 센터 머신·덤벨·프리웨이트를 CRUD. 덤벨 = center_machine 한 항목(또는 카테고리)로 처리.
- **AI에 '센터 전체 인벤토리' 제공 + 지시** — 지금은 회원이 고른 것만 들어감. 여기에 센터 보유 장비를 주고 "**센터에 있는 장비로만 루틴 구성**" 지시(프롬프트 튜닝, 1·2차·재등록 공통). 없는 장비 추천 방지.

**리스크:** 낮음. 프롬프트가 장비 제약을 실제로 지키는지 검증만. **작고 즉효 → 우선순위 상위 추천.**

## 3. 수업일지 자동 정리 (종목별 · 날짜별 무게·횟수·세트)

**판정: 중~높음 (가치 큼 · 구조화 저장이 관건).**

**이미 있는 것:** 음성일지 AI가 이미 `machines:[{name, detail, method}]`로 **종목·수치를 추출** + `daily_workout_log` 저장.

**새로 필요한 것:**
- **구조화 저장** — 지금 `ai_summary`가 **텍스트**라 집계 불가. `{종목, 세트:[{무게,횟수}]}` 형태로 정규화 저장(예: `daily_workout_log`에 `structured jsonb`, 또는 `workout_set` 테이블 신설).
- **종목명·수치 정규화** — "벤치/인클라인벤치/벤치프레스" canonical 통일, 무게·횟수 숫자화.
- **종목별 날짜순 집계 뷰/차트** — "벤치프레스: 7/16 20kg×12·30kg×12 → 7/20 20kg×15·30kg×15" 자동 정리.

**활용 가치:** 이걸 **회원앱에 "종목별 무게 증가 그래프"**로 붙이면 성과 가시화 = 재등록 유도 강력. AI 게이트(남은>0)·자기진화와도 연결.

**리스크:** 트레이너가 수치를 항상 말하진 않음 + 종목명 오인식. AI 추출이 100%는 아니라 **"자동 정리 + 트레이너 1탭 보정"** 전제(음성일지 현 흐름과 동일). 완전 자동 맹신 금지.

## 4. OT 스케줄링 자동화 + 관리자 배정 + 트레이너 추천

**판정: 가능하나 대형 에픽 (여러 스프린트 · 뼈대는 다 있음).** 사실상 보류했던 **FC(데스크 앞단)를 '관리자 role'로 경량화**한 것과 같음 — 좋은 방향.

**이미 있는 것:**
- `appointment`(user_id·start_at·status[booked/done/canceled]·trainer_id·log_id) = **스케줄 백본 존재.**
- `ScheduleBoard`(주간 그리드·빈슬롯에 회원 배치→appointment insert·완료→수업로그 차감·취소) + **owner 전체보기(trainerFilter)** 존재.
- 회원앱(/m 토큰 인증) = 회원-facing **슬롯 선택 화면**으로 확장 가능.
- 클로징 결과(`closing_result`/`approach`) = **배정 추천의 원천 데이터**(단 ④ 클로징 통계 미완).

**새로 필요한 것(조각별 · 각각 독립 배포 가능):**
1. **트레이너 가능시간 설정** — `trainer_availability`(반복 시간창) 신규. 지금 availability 개념 없음.
2. **회원 슬롯 선택** — (가능시간 − 이미 예약된 슬롯) = 열린 슬롯 → 회원이 체크 → `appointment` insert → 트레이너 스케줄 자동 반영. (회원앱/부킹 토큰 재사용. 동시성 가드 = 같은 슬롯 두 명 방지.)
3. **관리자 OT 파이프라인 뷰** — 신청(미배정) / 스케줄됨 / 미정 3구분. `user_table`(status=ot) + `appointment` 조인.
4. **관리자 등록·배정** — 데스크 오프라인 접수 → 관리자가 OT 회원 등록 + `trainer_id` 배정 → 트레이너는 수동 등록 부담 사라짐. **단 트레이너 수동 OT 추가(MemberForm)는 유지**(필드에서 즉석 OT용, 명시 요청).
5. **배정 추천(선택)** — 회원 성향·목표 ↔ 트레이너 강점(통증방향·soft 세일즈)·클로징률 매칭 → 추천. **대표가 무시 가능**(갑을병정·자기 기준으로도 배정). = FC 문서의 "배정 매칭 AI" 슬라이스 그대로. 선행 = ④ 클로징 통계 + 트레이너 프로필.

**리스크:** 크기(에픽). + `appointment.trainer_id` 배정 배선 정리(현재 "후속"으로 남아있음) + 예약 동시성 가드. 파괴적이진 않고 조각별 점진 배포 가능.

## 5. 종합 — 연결점 · 추천 순서

**서로 얽히는 축:** #4의 배정 추천 · #1의 포터블 '성향 자산' · ④ 클로징 통계 대시보드 = 전부 **"트레이너 성향/클로징 통계"**를 공유. → **④를 먼저 세우면** #4 추천·#1 자산이 그 위에 자연히 얹힘.

**premium 후보:** #2·#3·#4 일부는 B2(프리미엄 게이트) 대상 기능 후보 — 구독 얹을 때 함께 고려.

**추천 착수 순서(내 판단):**
1. **#2 센터 머신·덤벨 → 루틴 반영** — 작고 즉효, 이미 반배선. 대표에게 바로 체감되는 가치.
2. **#3 수업일지 구조화 정리** — 회원앱 성과 그래프·재등록 유도로 파급 큼. AI 게이트와 결.
3. **④ 클로징 통계**(마스터 백로그) — #4 추천·#1 성향자산의 공통 토대라 먼저.
4. **#4 OT 스케줄링 에픽** — 조각 순서: 가능시간 설정 → 회원 슬롯선택 → 관리자 파이프라인 뷰 → 관리자 등록·배정 → (④ 위에) 배정 추천.
5. **#1 센터선택·이직 이전** — 데이터 모델 분리라 신중히. **범위(포터블=개인자산만 vs 회원데이터까지) 확정이 선행.** B3(카카오/온보딩)와 묶으면 효율.

> 원칙(MASTERPLAN): 뼈대(데이터)부터 → AI 얹기 · 데모/실연동 안 섞기 · 광범위 잠금/모델변경은 기존행 백필+단계 배포. #1·#4는 스키마가 커서 특히 그렇게.
