## 0. 설계 원칙 (합의)

- **한 방향, 한 규칙.** 원장이 /admin에서 공지를 쓰고, 트레이너는 앱에서 받는다. **원장→트레이너 단방향**(트레이너 작성 없음 = 읽기 전용). 이미 있는 `auth_is_owner()`로 쓰기 권한을 가른다.
- **공지마다 원장이 강도를 고른다(핵심).** 두 종류다. **① 일반**(기본) = 헤더 벨에 안읽음 배지로 쌓임, 앱 안 막음(짬날 때 확인). **② 필수확인** = 앱 열 때 **전체를 덮는 모달, 확인 누르기 전엔 안 넘어감.** 확인=서버 읽음 행 기록 → 폰·태블릿 어디서 눌러도 다시 안 뜸(기기 무관). "봤나 안 봤나 몰라 무시" 문제를 구조로 막되, **게이트는 드물게 진짜 중요한 것만**(프로모션 시작·규정 변경·긴급 스케줄 등) 써야 닳지 않는다 — 남발되면 "일단 확인 눌러치우기" 반사행동이 돼 정작 중요한 것도 안 읽힘. 그래서 강도를 **원장이 공지 단위로** 정한다(`must_ack` 플래그).
- **타겟팅 v1 포함.** 전체 트레이너 공지(target 비움) + 특정 트레이너 지정 공지 둘 다. 지정은 배열 컬럼 하나(`target_trainer_ids`)로 — **조인 테이블 없이** RLS `= any(...)`로 필터.
- **읽음 테이블 신설.** `announcement_read`(공지×트레이너 1행). 게이트 판정·안읽음 배지·원장 "누가 읽었나" 현황의 단일 근거. (localStorage 안 씀 — 기기 넘어 정확해야 하므로 서버 기록.)
- **신설 테이블 2개뿐 = 회귀 격리.** 기존 테이블·컬럼 무변경. 기존 기능에 손 안 댐. 데모(키 없음)는 `if(!supabase)` 가드로 로컬만 → 크래시 0.
- **C(원장 작성)와 D(트레이너 게이트)를 나눠**, 공지를 실제로 만든 뒤 게이트가 그걸 받는지 순서대로 확인한다.

> 보안 결: `trainer_todo`(할일 A)처럼 **처음부터 스코프가 잡힌 테이블**이다. 읽음은 본인 것만 기록(`with check trainer_id=auth.uid()`), 쓰기는 원장만(`auth_is_owner()`). 총정리 §5의 "트레이너 RLS 부재" 부채를 새로 늘리지 않는다.

## 1. 공지에 들어가는 것 (한눈)

| 요소 | 내용 | 저장 |
|---|---|---|
| **작성** | 원장이 제목+본문 작성, 대상 선택(전체/지정), **강도(일반/필수확인)**, 핀 여부 | `announcement`(신설) |
| **강도** | `must_ack=false` 일반(벨만) · `must_ack=true` 필수확인(게이트) | `announcement.must_ack boolean` |
| **대상** | 비움=계정 전체 트레이너 · 지정=특정 트레이너들(배열) | `announcement.target_trainer_ids uuid[]` |
| **수신(게이트)** | 앱 열 때 내게 온 **필수확인 안읽음** 공지 → 덮는 모달 | 판정=lib 순수함수 |
| **수신(벨)** | 일반·필수확인 **모든** 안읽음 → 헤더 벨 배지 | 판정=lib 순수함수 |
| **확인(읽음)** | 트레이너가 "확인" → 읽음 1행 기록(다시 안 뜸) | `announcement_read`(신설) |
| **재열람** | 헤더 벨 → 내 공지 전체(읽음 포함) 다시 보기 | 위 두 테이블 |
| **읽음 현황** | 원장이 각 공지 "N/M 읽음" 확인 | `announcement_read` 집계 |

> **게이트 정의(필수확인만):** `must_ack=true`인 안읽음 공지가 있을 때만 앱을 덮는다. 모달은 "확인" 전엔 안 닫힘(배경 클릭·X 못 함). 확인 시 표시된 필수확인 공지 **전부**에 읽음 기록하고 닫는다. **일반 공지는 게이트를 안 띄운다** — 벨 배지로만 쌓이고, 트레이너가 벨을 열어 볼 때 읽음 처리. (공지별 개별 확인은 §10 후속 옵션.)

## 2. 데이터 모델 — 마이그레이션 1개 (테이블 2개)

파일: `docs/migrations/2026-07-12-announcement.sql` (파일명 날짜=실제 실행일). Supabase 대시보드 실행 후 동일 SQL을 기록본 커밋.

### 2-1. `announcement` — 공지(원장 작성)

```sql
create table if not exists announcement (
  id                 uuid primary key default gen_random_uuid(),
  account_id         uuid not null default auth_account_id() references account(id),
  author_id          uuid not null default auth.uid()       references trainer(id),  -- 작성자(원장)
  title              text not null,
  body               text not null,
  target_trainer_ids uuid[],                                 -- null/빈배열 = 계정 전체 트레이너, 값 있으면 그 트레이너만
  must_ack           boolean not null default false,         -- ★ true=필수확인(게이트) · false=일반(벨만). 기본 일반.
  pinned             boolean not null default false,         -- 고정(목록·벨·게이트 상단)
  created_at         timestamptz not null default now(),
  updated_at         timestamptz not null default now()
);
alter table announcement enable row level security;

create index if not exists announcement_account_idx on announcement (account_id, created_at desc);
```

### 2-2. `announcement_read` — 읽음(트레이너별 확인)

```sql
create table if not exists announcement_read (
  id              uuid primary key default gen_random_uuid(),
  announcement_id uuid not null references announcement(id) on delete cascade,  -- 공지 삭제 시 읽음도 정리
  account_id      uuid not null default auth_account_id() references account(id),
  trainer_id      uuid not null default auth.uid()       references trainer(id),
  read_at         timestamptz not null default now(),
  unique (announcement_id, trainer_id)                       -- 한 트레이너 한 공지 1행(중복 확인 무해)
);
alter table announcement_read enable row level security;

create index if not exists announcement_read_by_ann     on announcement_read (announcement_id);
create index if not exists announcement_read_by_trainer on announcement_read (trainer_id);
```

### 2-3. RLS — 쓰기는 원장, 읽음은 본인, 열람은 대상자+원장

```sql
-- announcement 조회: 같은 계정 AND (전체공지 OR 내가 대상 OR 원장)
--   원장은 관리 위해 전체 열람(게이트는 클라 순수함수가 '내 대상'만 다시 거른다 = §3).
drop policy if exists "ann_select" on announcement;
create policy "ann_select"
  on announcement for select to authenticated
  using (
    account_id = auth_account_id()
    and (target_trainer_ids is null or auth.uid() = any(target_trainer_ids) or auth_is_owner())
  );

-- announcement 쓰기(insert/update/delete): 원장만.
drop policy if exists "ann_write" on announcement;
create policy "ann_write"
  on announcement for all to authenticated
  using      (account_id = auth_account_id() and auth_is_owner())
  with check (account_id = auth_account_id() and auth_is_owner());

-- announcement_read 기록(insert): 본인 것만(남 대신 읽음 처리 불가).
drop policy if exists "annread_insert" on announcement_read;
create policy "annread_insert"
  on announcement_read for insert to authenticated
  with check (account_id = auth_account_id() and trainer_id = auth.uid());

-- announcement_read 조회: 본인 읽음 OR 원장(누가 읽었나 현황).
drop policy if exists "annread_select" on announcement_read;
create policy "annread_select"
  on announcement_read for select to authenticated
  using (account_id = auth_account_id() and (trainer_id = auth.uid() or auth_is_owner()));
-- update/delete 정책 없음 = 읽음은 append-only(되돌릴 일 없음). '안읽음으로' 필요해지면 그때 추가.
```

> **⚠️ 클로드코드 실행 전 확인 3가지:**
> 1. `auth_is_owner()` 헬퍼가 실재하는지(총정리 §1.3엔 있음 · ⑦-c에서 추가). 없으면 SQL 멈추고 문의. `select auth_is_owner();` 로그인 세션에서 확인.
> 2. `for all` 쓰기 정책에 **`for all`은 select도 포함**한다 → `ann_write`(for all)와 `ann_select`가 공존하면 원장 select는 두 정책의 **OR**로 평가돼 무방(원장은 어차피 전체 열람). 트레이너는 `ann_write` using이 false라 select엔 `ann_select`만 적용 = 의도대로. (동작 확인만.)
> 3. `default auth_account_id()`/`auth.uid()`로 INSERT 시 클라가 소유필드 안 보내도 채워지고 `with check` 통과하는지 실증(insert→`.select()` 1행).

## 3. lib — 순수함수 (`lib/announce.js` 신설)

게이트·벨·현황이 공유. **순수·now 미참조**(비교 재료는 컴포넌트가 주입). memberStatus.js(29KB·회원 라이프사이클)와 도메인이 달라 **새 파일**로 격리.

```js
// lib/announce.js
// 앱내 공지 순수함수. RLS가 이미 '내 계정+내 대상+원장' 범위로 좁혀 내려주지만,
// 원장은 전체를 받으므로(관리용) 게이트/벨 판정은 여기서 '내 대상'만 한 번 더 거른다.

/** 이 공지가 uid에게 온 것인가. target 비면 전체공지 = 모두 대상. */
export function isTargeted(ann, uid) {
  if (!ann) return false;
  const t = ann.target_trainer_ids;
  if (t == null || (Array.isArray(t) && t.length === 0)) return true; // 전체
  return Array.isArray(t) && uid != null && t.includes(uid);
}

/** 정렬: 핀 먼저, 그 안에서 최신순. */
function byPinnedThenNew(a, b) {
  if (!!a.pinned !== !!b.pinned) return a.pinned ? -1 : 1;
  return a.created_at < b.created_at ? 1 : -1;
}

/**
 * 안읽음 전체 — 내게 온 공지 중 (내가 쓴 것 아님) AND (아직 안 읽음). 일반+필수확인 모두.
 * → 헤더 벨 배지 카운트의 근거. 게이트는 이 결과에서 must_ack만 다시 거른다(mustAckUnread).
 * @param {Array} anns  announcement 행들(RLS로 내려온 것)
 * @param {Set|Array} readIds  내가 읽은 announcement_id 집합
 * @param {string} uid  로그인 트레이너 id
 */
export function unreadAnnouncements(anns, readIds, uid) {
  const read = readIds instanceof Set ? readIds : new Set(readIds || []);
  return (Array.isArray(anns) ? anns : [])
    .filter((a) => a && isTargeted(a, uid) && a.author_id !== uid && !read.has(a.id))
    .sort(byPinnedThenNew);
}

/** 게이트용 — 안읽음 중 필수확인만(앱을 막을 대상). 일반 공지는 여기 안 들어옴. */
export function mustAckUnread(anns, readIds, uid) {
  return unreadAnnouncements(anns, readIds, uid).filter((a) => a.must_ack === true);
}

/** 벨(재열람)용 — 내게 온 공지 전체(읽음 포함, 내가 쓴 것 제외). 각 행에 read 플래그 부착. */
export function visibleAnnouncements(anns, readIds, uid) {
  const read = readIds instanceof Set ? readIds : new Set(readIds || []);
  return (Array.isArray(anns) ? anns : [])
    .filter((a) => a && isTargeted(a, uid) && a.author_id !== uid)
    .map((a) => ({ ...a, read: read.has(a.id) }))
    .sort(byPinnedThenNew);
}
```

> 판정을 순수함수로 뺀 이유: 원장 over-fetch(전체 열람)를 클라에서 "내 대상"으로 다시 거르는 규칙이 한 곳에 모여, 게이트·벨이 어긋날 여지가 없다. **게이트=`mustAckUnread`(필수확인만) · 벨 배지=`unreadAnnouncements`(전체 안읽음)** 로 강도를 가른다. **콘솔 수기 검증**(전체/지정/내가 쓴 것/읽음/일반 vs 필수확인).

## 4. UI — 트레이너 수신 (게이트 + 벨)

### 4-1. `components/AnnouncementGate.jsx` (신설) — 필수확인 모달

- **자체 fetch(마운트 시):** `announcement`(RLS 스코프) + `announcement_read`(본인 것) 두 쿼리 → 두 파생:
  - **게이트 대상** = `mustAckUnread(anns, myReadIds, uid)` (필수확인 안읽음만 = 앱을 막을 것).
  - **벨 배지 수** = `unreadAnnouncements(anns, myReadIds, uid).length` (일반+필수확인 전체 안읽음) → 부모로 콜백(§4-2).
  - uid: `supabase.auth.getUser()` 자체 조회(page.jsx의 `myUid` prop으로 넘겨도 됨 — 이미 있음, §5).
  - 데모(`!supabase`): 즉시 `return null`(게이트 없음). 크래시 0.
- **표시:** 게이트 대상 0 → `return null`(일반 공지만 있으면 게이트 안 뜸 = 벨로만). 1개 이상 → **`fixed inset-0 z-50`** 백드롭 + 중앙 카드. 배경 클릭·ESC로 안 닫힘(필수확인). 여러 개면 카드 안에 목록(핀 먼저), 각 제목·본문·작성일.
- **확인:** 하단 "확인했습니다" 버튼 1개 → 표시된 필수확인 공지 **전부** `announcement_read`에 기록:
  ```js
  const rows = unread.map((a) => ({ announcement_id: a.id })); // account_id·trainer_id는 DEFAULT가 채움
  const { data, error } = await supabase
    .from("announcement_read")
    .upsert(rows, { onConflict: "announcement_id,trainer_id", ignoreDuplicates: true })
    .select();
  if (error) { setErr("확인 반영 실패 — 정책 확인"); return; }  // .select() 하드닝
  setUnread([]);  // 성공 후에만 닫음(부분 실패면 다시 뜨는 게 안전)
  ```
  - `upsert + ignoreDuplicates`: 같은 공지 재확인(경합·재시도) 시 unique 충돌을 조용히 흡수. `.select()`로 반영 확인(트러블슈팅: 정책 없으면 0행 조용한 실패).
- **렌더 규칙:** 정적 토큰만(`bg-card`·`border-line`·`text-ink/sub/muted`). 아이콘 `lucide-react`(예: `Megaphone`·`Pin`·`Check`). 동적 클래스 문자열 금지(CLAUDE.md).

### 4-2. 헤더 벨 (재열람 + 안읽음 배지) — page.jsx 헤더에 인라인

- 위치: `app/page.jsx` 헤더 우측 액션줄, **관리자 `<a>`(현재 699~705행) 앞**에 종 버튼 추가.
- 표시: `Bell` 아이콘 + 안읽음 개수 배지(0이면 배지 없음). 클릭 → 게이트 모달을 **재열람 모드**로 오픈(`visibleAnnouncements`, 읽음 포함, 강제 아님·X로 닫힘).
- 구현 선택: **게이트 컴포넌트가 두 모드**(강제=안읽음·닫기 불가 / 재열람=전체·닫기 가능)를 갖게 하고, 벨은 재열람 모드를 연다. 안읽음 카운트는 게이트가 이미 계산하므로 부모(page.jsx)로 콜백(`onUnreadCount`)해 배지에 반영.
  - 가장 값싼 형태: `<AnnouncementGate uid={myUid} onUnreadCount={setUnreadCount} reviewOpen={bellOpen} onCloseReview={()=>setBellOpen(false)} />` 한 컴포넌트가 게이트+재열람 모두 담당. page.jsx는 벨 버튼과 `bellOpen`·`unreadCount` state만.

> **대안(원하면):** 벨 대신 **전용 "공지" 상시 탭**(할일=id 13 패턴, id 14). 게이트는 그대로 두고 탭은 목록·재열람 전용. 벨이 헤더 공간을 덜 먹어 추천이지만, 탭이 더 익숙하면 그쪽으로. → §10 결정 포인트.

### 4-3. 마운트 (`app/page.jsx`)

- 최상위 반환 트리 안, `<header>` **위**(또는 루트 div 직속)에 게이트 배치 → 전체를 덮는 오버레이:
  ```jsx
  return (
    <div className="min-h-screen bg-bg ...">
      <AnnouncementGate uid={myUid} onUnreadCount={setUnreadCount}
        reviewOpen={bellOpen} onCloseReview={() => setBellOpen(false)} />
      <header ...>  {/* 여기에 벨 버튼(unreadCount·setBellOpen) */}
  ```
- `myUid`는 이미 `supabase.auth.getUser()`로 세팅됨(현재 549·585행). 재사용.
- 게이트는 `myUid`가 null(조회 전)이면 아무것도 안 함 → uid 들어오면 fetch. (초기 깜빡임 없음.)

## 5. UI — 원장 작성 (`components/AdminAnnouncements.jsx` 신설)

- 배치: `app/admin/page.jsx`, 원장 게이트(`role==='owner'`) 통과 영역에 `<section className="mb-8"><AdminAnnouncements trainers={trainers} /></section>`. **트레이너 추가(현재 337행 AddTrainerForm) 근처**(원장의 커뮤니케이션 도구 묶음)로.
- `trainers`: admin이 이미 로드하는 `from("trainer").select("id, name")` 결과(현재 `trainers`, 211행). **그대로 prop 전달** → 타겟 선택·읽음 현황의 이름 소스. (name은 시드 시 이메일일 수 있음 — 표시용으론 충분.)
- **작성 폼:** 제목 input + 본문 textarea + 대상 선택 + **강도 체크(필수확인)** + 핀 체크 + "게시" 버튼.
  - 대상 UI: 라디오 "전체 트레이너 / 특정 지정". 지정 시 `trainers` 체크박스 멀티선택 → `target_trainer_ids`. 전체면 `null`로 저장(빈배열도 전체로 처리하나 `null` 권장 = 인덱스·의미 명확).
  - **강도 UI: "필수확인(앱 열 때 확인 요구)" 체크박스 1개.** 체크=`must_ack:true`(게이트), 기본 해제=`false`(벨만·일반). 옆에 한 줄 도움말("드물게 · 꼭 봐야 할 것만" — 남발 방지 넛지).
  - INSERT payload = `{ title, body, target_trainer_ids: 전체?null:[...ids], must_ack, pinned }`(account_id·author_id는 DEFAULT). `.insert().select()` 하드닝 — 0행이면 "게시 실패"(원장 아님/정책).
- **목록:** 최신순(핀 먼저). 각 행 = 제목·본문요약·대상(전체/이름칩)·**강도칩(필수확인/일반)**·**"N/M 읽음"** 배지 + 수정/삭제/핀토글.
  - 읽음 현황: `announcement_read`를 `announcement_id` 별로 집계. M = 대상 트레이너 수(전체면 `trainers.length`, 지정이면 `target_trainer_ids.length`), N = 그 공지의 read 행 수. 간단 카운트로 v1 충분(누가 안 읽었는지 상세는 후속).
  - 수정: title/body/target/pinned UPDATE(`updated_at=now()`) → `.select()` 하드닝. 삭제: DELETE(cascade로 읽음 정리) → 인라인 2단 확인(브라우저 다이얼로그 금지 — 총정리 결).
- 데모(`!supabase`): 로컬 상태만(게시=로컬 추가), 크래시 0. accent는 admin 톤(`fuchsia`) 유지 가능.

## 7. 회귀·안전

- 신설 테이블 2개뿐 → 기존 테이블·컬럼·쿼리 무변경. 기존 기능 영향 0.
- 게이트: **`must_ack=true` 안읽음에만** 반응(일반 공지는 벨만·오버레이 안 뜸). 게이트 대상 0이거나 `!supabase`면 `return null` → 없던 트레이너·데모엔 변화 없음. uid null이면 조회 전 = no-op.
- 읽음은 append-only + unique → 재확인·경합해도 중복/오류 없음(`ignoreDuplicates`).
- 쓰기 원장 한정(`auth_is_owner()`) → 트레이너가 공지 못 만듦/수정 못 함(읽기·읽음기록만).
- 읽음 기록 본인 한정(`with check trainer_id=auth.uid()`) → 남 대신 확인 처리 불가.
- 원장 over-fetch(전체 열람)는 **클라 순수함수가 '내 대상'으로 재필터** → 원장이 트레이너 앱을 열어도 남에게 지정된 공지로 게이트 안 걸림(+ 자기가 쓴 것 `author_id` 제외).
- 데모(키 없음): 게이트·벨·작성 모두 로컬/미표시 → 크래시 없음.

## 8. 테스트 체크리스트

1. `npm.cmd run lint` green (각 서브커밋).
2. **마이그레이션:** 두 테이블·RLS 4정책 생성. 원장 insert→`.select()` 1행. 다른 트레이너로 '특정 지정' 공지 SELECT 0행(대상 필터). 읽음 insert 본인만.
3. **원장 작성:** 전체/지정·**일반/필수확인**·핀 조합 게시 → 목록에 강도칩·대상칩 표시. 수정/삭제 왕복. 데모 크래시 없음.
4. **필수확인 게이트:** `must_ack=true` 공지를 대상 트레이너로 앱 열기 → **모달이 앱을 덮고** 배경/ESC로 안 닫힘. "확인" → 닫히고 `announcement_read` 기록. **새로고침/재접속해도 다시 안 뜸.** 다른 기기로 로그인해도 안 뜸(서버 기록).
5. **일반 공지(핵심 분기):** `must_ack=false` 공지는 대상 트레이너 앱에서 **게이트 안 뜸** — 헤더 벨 배지에만 +1. 벨 열어 보면 읽음 처리 → 배지 감소.
6. **비대상·작성자:** 지정 안 된 트레이너 앱엔 게이트·벨 안 뜸. 원장 본인(작성자) 앱엔 자기 공지로 게이트 안 뜸.
7. **벨/재열람:** 헤더 벨 안읽음 배지 = 일반+필수확인 전체 안읽음 수(확인 후 감소). 벨 클릭 → 읽은 공지 포함 목록, X로 닫힘.
8. **읽음 현황:** 원장 목록의 "N/M 읽음"이 트레이너 확인(게이트·벨 어느 쪽이든)에 따라 증가.
9. **회귀:** 스케줄·회원·할일·내실적·OT/PT 뷰 무변. 데모 모드 크래시 없음.
10. 폰: Vercel 배포 후 하드리프레시 — 원장 게시 → 트레이너 폰: 필수확인은 게이트, 일반은 벨 배지 → 확인 → 안 뜸 육안.

## 9. 열어둔 결정 포인트 (스펙 확정 전 한 번만)

1. **재열람 표면:** 헤더 **벨**(추천·가벼움) vs 전용 **"공지" 탭**(id 14, 할일 패턴). 벨로 잡아놨음 — 탭이 낫다면 §4-2 대안으로 스왑.
2. **게이트 확인 방식:** 여러 공지 시 **"모두 확인" 1버튼**(추천·현재 스펙) vs 공지별 개별 확인(더 강한 "각각 봄" 보장, 클릭 늘어남).
3. **핀/만료:** `pinned`만 넣음. 자동 만료(기간 지나면 내림)는 §10 후속.

> 이 3개는 코드량 작은 갈래라 지금 정하면 그대로 확정. 답 없으면 추천값(벨·모두확인·핀만)으로 진행.
