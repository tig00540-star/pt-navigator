# v2 설계 제안 — OT 여정 '전/후' 재편 (1차·2차·재등록 통일)

> **성격:** 이건 구현 스펙이 아니라 **설계 방향 합의용 제안서**다. 여기서 모델·시점 매핑·공통 패턴을 확정한 뒤, 시점별 **구현 스펙(코드)**을 따로 내린다.
> 역할·흐름: 웹Claude(이 제안) → 트레이너 확정 → 웹Claude(시점별 구현 스펙) → 클로드코드 → …
> 배경 요청: "1차·2차 OT 지원·재등록 지원이 너무 복잡, 정보 많아 한눈에 안 들어온다"(트레이너 피드백) → **전/후 시점으로 나눠 화면·AI출력·프롬프트 전반 재편**, 세 국면을 **비슷한 패턴**으로.

---

## 0. ★먼저 확정해줘 (내가 세운 전제 — 틀리면 한 줄로 뒤집어줘)

이 제안 전체가 아래 3개 전제 위에 서 있다. 여기만 맞추면 나머지는 그 위에서 조정된다.

- **[전제 A] 시점 축 = 각 국면을 '전(수업 준비·AI 브리핑)'과 '후(수업 결과·기록)'로 쪼갠다.** 6개 면: `1차 전`·`1차 후`·`2차 전`·`2차 후`·`재등록 전`·`재등록 후`. (급한불/인바디/운동일지는 이번 스코프 밖.)
- **[전제 B] 전/후를 화면에 어떻게 나눌지 = 탭 vs 세그먼트.** 지금 1차는 이미 **두 탭**(`1차 지원`=전 / `관찰 기록`=후)이고, 2차·재등록은 **한 탭에 전+후가 뭉쳐** 있다. → **추천: 2차·재등록도 한 탭 안에서 `준비 | 기록` 세그먼트 토글로 전/후 분리**(탭 수 안 늘림). 1차는 이미 물리적으로 나뉘어 있으니 그대로 두되 라벨/패턴만 통일. *(대안: 전부 별도 탭 = 탭바 6개로 비대 → 비추천.)*
- **[전제 C] "1차 후" = 관찰 기록(ObservationTab) 시점이다.** 즉 이번 재편에 **관찰기록 입력 간소화(백로그 #6)가 흡수**된다. *(만약 "1차 후"를 관찰과 별개의 새 AI 요약으로 생각한 거면 말해줘 — 설계가 달라짐.)*

이 3개가 OK면 아래를 읽어줘. (특히 B가 이 문서 레이아웃의 뼈대.)

## 1. 진단 요약 (왜 "한눈에 안 들어오나")

세 화면 렌더를 전부 읽고 확인한 5가지:

1. **평면 나열 — 앵커 없음.** 섹션이 전부 같은 무게의 형제 카드. 특히 **2차**는 섹션 8개(케이스거울·클로징·운동대처·브리핑·arc·거절·gaps·결과기록)가 동일 비중으로 쌓여 눈이 시작점을 못 잡는다. 1차만 「핵심 3줄」 앵커가 있고, 2차·재등록엔 없다.
2. **2차의 이중 3분기 = 최대 과부하.** 클로징(yes/partial/no) + 운동대처(yes/partial/no) = **미래 시나리오 6개를 미리 다 머리에 넣으라**는 구조. 실제 수업에선 딱 한 갈래만 벌어지는데.
3. **수업 전/중/후가 한 스크롤에 섞임.** 2차·재등록은 '준비'와 '결과기록'이 한 화면. 시점마다 필요한 게 다른데 전부 노출.
4. **AI 필드를 거의 1:1로 다 렌더.** arc는 비트마다 왜/무엇/예시/톤 4줄, 브리핑 4행, 거절 4유형×4필드… '요약→상세' 단계가 얕다.
5. **세 화면 구조가 제각각.** 1차=3줄+접기7, 2차=섹션나열, 재등록=한 details 안 flat. 화면마다 레이아웃을 새로 학습해야 한다.

## 2. 통일 원칙 — 공통 두 패턴

### 패턴 ①: '전' 화면 = 3층 (모든 준비 브리핑에 동일 적용)
> 현장에서 폰을 '힐끗' 본다는 전제. 3초 안에 "지금 뭐라고 말하지?"에 답이 나와야.

- **L0 · 지금 이 한마디** *(항상 최상단, 큰 글씨, 따옴표):* 이 순간 회원에게 할 멘트/행동 **1개**. 1차·재등록=클로징 진입(enter). 2차=선택한 분기의 진입.
- **L1 · 핵심 카드 3개** *(항상, 3초 스캔):* 그 국면의 승부처 3개. 이모지+한 줄.
- **L2 · 상세** *(접힘, 주제별 ≤4개):* 나머지 전부. 각 카드는 **한 줄 요약 먼저 → 더보기로 풀텍스트**.

### 패턴 ②: '후' 화면 = 최소 입력
> 기록은 다음 단계의 재료다. 사고를 유발하는 **핵심 항목만** 노출, 나머지는 조건부/접힘.

- 결과/관찰 **필수 3~4개**만 첫 화면. 조건부 확장(실패·보류일 때만 사유·재접근일). 저장 = 다음 시점 AI의 근거.

### 공통 컴포넌트로 뽑기 (일관성 + 코드 축소)
`BriefShell`(3층 셸) · `NowLine`(L0 한마디) · `KeyCards`(L1 3카드) · `ClosingSteps`(진입/그림/착지/침묵 카드) · `ObjectionCards`(거절) · `ArcList`(대화흐름) · `BranchPicker`(2차 실시간 분기) · `ResultForm`(후 화면 결과 폼). → 세 화면이 같은 부품을 쓰면 레이아웃이 저절로 통일된다.

## 3. 6개 시점 설계 (골격)

> 각 시점: **목적 / L0·L1·L2(전) 또는 필드(후) / AI출력 변경 / 프롬프트 변경.** 코드는 다음 단계 스펙에서.

### 3-1. 1차 전 — `1차 지원` (FirstOTTab · FirstOTAssist 재편)
- **목적:** 회원 첫 대면 직전, 기본정보로 '가설' + 6단계 흐름 + 추천 프로그램 + 클로징 준비.
- **L0:** 클로징 진입 한마디(`closing.enter`).
- **L1:** ①추천 프로그램(pick 이름·가격) ②오늘의 승부처(`hypothesis` + 자각 포인트) ③클로징 방향(`approach_tag` + `land` 요지).
- **L2(접힘 3):** 대화 흐름(arc·요약형) / 치트키 1개(`movement_cues`) / 거절 대응(요약형). → **`observe_targets`(관찰해올 것)는 '1차 후'로 이동**(맥락상 후에서 체크리스트로 씀).
- **AI출력 변경:** arc 각 비트 4줄→`direction` 1줄 기본 + 더보기. objections 4→핵심 2(price·hesitation) 먼저, 나머지 더보기. `movement_cues` 6필드→3핵심(exercise·why_instant·principle) 먼저.
- **프롬프트 변경:** 분량 상한 더 조임, example 축소. **스키마 필드는 유지**(렌더가 덜 펼칠 뿐) → 구캐시 호환·재검증 부담 최소. *(오늘 이미 「3줄+접기」라 변화 폭 가장 작음.)*

### 3-2. 1차 후 — `관찰 기록` (ObservationTab 간소화 · 백로그 #6 흡수)
- **목적:** 수업 직후 관찰을 최소 노력으로 기록 → **2차 브리핑의 유일 근거.**
- **필드(최소):** 1차 전에서 넘어온 `observe_targets`를 **체크리스트로 상단 표시** → 그것만 빠르게. 핵심 입력 = 자극반응 · 목적파악 · 회원 한마디 · 종합소견(자유). 나머지 상세 태그는 접힘.
- **AI출력(선택 · 신규):** 저장 후 **"이 관찰이면 2차에서 이렇게 갈 수 있어요" 한 줄 프리뷰**(2차 예고). — 이건 새 AI 호출이라 [전제 C] 확정 후 넣을지 결정. 기본안은 **입력 간소화만**, 프리뷰는 옵션.
- **프롬프트 변경:** (프리뷰 넣을 때만) 초경량 요약 프롬프트 신규. 안 넣으면 프롬프트 무변경.

### 3-3. 2차 전 — `2차 브리핑`의 '준비' 세그먼트 (SecondOTTab 브리핑부 재편 · **이중분기 해소**)
- **목적:** 2차 수업 준비 — 등록 당위성 + 클로징 + 자극결과별 운동대처.
- **★핵심 변경 — 실시간 분기 선택기(`BranchPicker`):** 상단에 [잘 옴 / 약하게 / 아직] 세그먼트. **선택한 갈래의 클로징(enter/paint/land/hold) + 운동대처(cause/adjustment/direction)만 L0·L1에 크게.** 나머지 두 갈래는 접힘. → **6개 동시노출 제거, '지금 이 갈래' 하나에 집중.** (기본 선택 = '잘 옴' or 마지막 선택 기억.)
- **L0:** 선택 분기의 클로징 진입 한마디.
- **L1:** ①등록 당위성 한 줄(`closing_logic` or `risk_if_alone`) ②선택분기 클로징 그림·착지 ③선택분기 운동 조정.
- **L2(접힘):** 케이스 거울(`case_feedback`) / 대화흐름 arc / 거절 대처 / data_gaps.
- **AI출력 변경:** closing 3분기·stimulus 3분기 **스키마 유지**(선택기가 소비) — 6개를 '만들되 1개만 크게 보여줌'. briefing 4행→2핵심(risk_if_alone·closing_logic) 먼저. arc·objections 축소는 1차와 동일 결.
- **프롬프트 변경:** 대체로 유지. 분량 상한만 조임. *(스키마를 안 줄이므로 기존 2차 캐시 호환.)*

### 3-4. 2차 후 — `2차 브리핑`의 '기록' 세그먼트 (현 ㉠ 클로징 결과 기록 분리)
- **목적:** 수업 후 클로징 결과 기록(다음: PT 확정 or 재접근).
- **필드(최소):** 결과(성공/실패/보류) → **조건부** 사유(실패·보류) · 재접근일(보류) · 케이스 3박자(선택). 지금 로직 그대로, **'준비'와 분리된 별도 세그먼트**로 이동해 준비 화면을 가볍게.

### 3-5. 재등록 전 — `재등록`의 '준비' 세그먼트 (PtReRegTab 브리핑 + RegBriefView 재편)
- **목적:** 재등록 대화 준비 — 그동안의 근거 + 클로징.
- **L0/L1/L2:** 1차 전과 **동일 3층.** L0=클로징 진입. L1=①확인된 것(`proven_in_pt`) ②멈추면 잃는 것(`risk_if_stop`) ③다음 로드맵(`next_roadmap`). L2=대화흐름·이유별 대처(기록된 `reg_reason` 강조 유지)·클로징 4단계·gaps.
- **AI출력 변경:** RegBriefView가 지금 5블록을 **다 펼침** → 3층으로. 이유별 대처는 기록 사유 1개 먼저, 나머지 접힘.
- **프롬프트 변경:** 스키마 유지, 분량 상한.

### 3-6. 재등록 후 — `재등록`의 '기록' 세그먼트 (현 결과 기록 분리)
- **필드(최소):** reg_result → 조건부 사유·재접근일. 현 로직 유지, '준비'와 분리.

## 4. AI출력·프롬프트 변경 정책 (범위·리스크 관리)

트레이너가 "AI출력·프롬프트도 손보자"고 했으니, **두 단계로 나눠 리스크를 통제**하자:

- **1단계(추천 · 이번 재편의 기본):** **스키마는 유지**하고 (a)렌더를 3층으로 재편 + (b)프롬프트의 **분량 상한만** 조여 각 필드를 짧게. → 기존 캐시(2차 brief·재등록 reg_brief·1차 first_assist) **호환**, 방어 렌더만 하면 됨. 재검증 가벼움. **"한눈에" 문제의 90%는 렌더 3층 + 2차 분기선택기로 해결.**
- **2단계(선택 · 명백한 과잉만):** 필드 자체 축소(예: 2차 arc 6→3, objections 4→2를 **스키마에서** 제거, 1차후 프리뷰 신규). → route.js 프롬프트 변경 + **기존 캐시 구스키마와 어긋남**(방어 렌더 + '재생성' 유도) + phase별 AI 품질 재검증. 범위·리스크 큼.

> **제안:** 우선 **1단계로 전 국면 재편**해서 체감 개선을 먼저 확보하고, 실제로 써보며 "이 필드는 진짜 안 본다" 확신 선 것만 2단계로 스키마 축소. (안 그러면 프롬프트 6군데 동시에 흔들려 회귀 잡기 어려움.) — 1·2단계 중 어디까지 이번에 갈지는 §6에서 물음.

## 5. 캐시·검증·구현 순서

- **캐시:** 1단계는 스키마 유지라 기존 캐시 그대로 렌더(옵셔널 체이닝 방어). 2단계로 필드 빼면 구캐시엔 신필드 없음 → 해당 섹션 조용히 빈 채 '재생성'으로 갱신(기존에도 쓰던 패턴).
- **공통 컴포넌트 먼저:** `BriefShell`·`NowLine`·`KeyCards`·`ClosingSteps`·`ObjectionCards`·`ArcList`·`BranchPicker`·`ResultForm`을 먼저 만들고 → 세 '전' 화면이 갈아끼움. 코드량·회귀면 최소.
- **구현 순서(제안):** ①공통 컴포넌트 → ②2차(가장 아픈 곳 · 분기선택기 효과 큼) → ③재등록 → ④1차(변화 폭 최소) → ⑤1차후(관찰 간소화). 각 단계 폰 검증 후 커밋.
- **검증:** 화면별 — 3초 스캔 되나(L0/L1만으로 판단 가능?) · 접힘 열어야 할 일 드문가 · 2차 분기 전환 매�ک · 구캐시 회원 방어 렌더 · lint.

## 6. 이번에 확정할 것 (§0 전제 OK라는 가정하에)

§0(A·B·C)만 먼저 답해주면 대부분 풀린다. 추가로 2개만:

- **[D] AI출력 변경 범위 = 1단계까지? 2단계까지?** (§4) — 추천: **1단계 전면 + 2단계는 나중에 골라서.**
- **[E] '1차 후' 프리뷰(신규 AI 한 줄) 넣을까?** (§3-2) — 추천: **이번엔 관찰 간소화만, 프리뷰는 다음.**

이 5개(A~E) 확정되면, **시점별 구현 스펙**을 순서대로(공통 컴포넌트 → 2차 → …) 코드까지 내려줄게. 첫 스펙은 '공통 컴포넌트 + 2차 전/후'로 잡을 생각(제일 아픈 곳부터).
