# v2 스펙 — 회원앱 자가입력 M3: 스케줄 체크 (개인운동/PT받은날)

> 성격: 클로드코드 **구현 스펙**. 웹Claude(설계) → 트레이너 → 클로드코드(diff) → 트레이너(git diff) → 웹Claude(검토) → 폰/Supabase 검증.
> 커밋은 트레이너 직접(파일지정 `git add` · `npm.cmd run lint` 통과 후). SQL=Supabase 대시보드 수동(git엔 이 기록본).
> 전제: 회원앱 v1 + B1/B2 + **M1(유산소 · 회원 쓰기 토대)** + M2(사진) 완료. **M1 패턴을 그대로 복제**(스토리지·AI 없음).

---

## 0. 개요

회원이 **개인운동 한 날 / PT 받은 날**을 스스로 체크 → 트레이너가 "회원이 얼마나 스스로 운동하나"를 읽음. 자가입력 3종의 마지막.

> **범위 밖(오늘 안 함):** "개인운동 시 지난 수업 기반 루틴 자동 제시"는 별도 후속(안전판 설계 필요 · §6). M3은 **순수 체크**만.

## 1. 데이터 계약 · 마이그레이션

### 1.1 `schedule_check` (신규 · 회원 소유 · M1 `cardio_log`와 동일 틀)

```
schedule_check (
  id         uuid pk default gen_random_uuid(),
  user_id    uuid not null references user_table(id) on delete cascade,
  on_date    date not null default current_date,
  kind       text not null,     -- 'personal'(개인운동) | 'pt'(PT받은날)
  note       text,
  created_at timestamptz default now()
)
```

### 1.2 마이그레이션

**`docs/migrations/2026-07-16-member-schedule.sql`**

```sql
-- =============================================================================
-- 회원 자가입력 M3 — schedule_check(개인운동/PT 체크) + 회원 CRUD RLS + 트레이너 조회
-- 실행: Supabase SQL Editor(수동) · git 기록본. 멱등. M1 cardio_log와 동일 패턴.
-- 전제: auth_member_id() · auth_account_id().
-- =============================================================================
create table if not exists schedule_check (
  id         uuid primary key default gen_random_uuid(),
  user_id    uuid not null references user_table(id) on delete cascade,
  on_date    date not null default current_date,
  kind       text not null,
  note       text,
  created_at timestamptz default now()
);
create index if not exists schedule_check_user_idx on schedule_check (user_id, on_date desc);
alter table schedule_check enable row level security;

-- 회원: 본인 것만 CRUD (추가·삭제 위주 · update 불필요)
drop policy if exists "member_sched_select" on schedule_check;
create policy "member_sched_select" on schedule_check for select to authenticated
  using (user_id = auth_member_id());
drop policy if exists "member_sched_insert" on schedule_check;
create policy "member_sched_insert" on schedule_check for insert to authenticated
  with check (user_id = auth_member_id());
drop policy if exists "member_sched_delete" on schedule_check;
create policy "member_sched_delete" on schedule_check for delete to authenticated
  using (user_id = auth_member_id());

-- 트레이너: 자기 account 회원 것 조회(읽기만)
drop policy if exists "trainer_sched_select" on schedule_check;
create policy "trainer_sched_select" on schedule_check for select to authenticated
  using (exists (
    select 1 from user_table u where u.id = schedule_check.user_id and u.account_id = auth_account_id()
  ));

-- 검증: 정책 4개(member 3 + trainer 1). 회원=본인 것만, 트레이너=account 회원.
-- 롤백: drop table if exists schedule_check cascade;
```

> 스토리지 없음(순수 테이블) → M2 같은 버킷/서명URL 이슈 없음. `member_photo` 테이블 RLS와 동일 구조.

## 2. 회원앱 UI — `app/m/[token]/page.jsx` (스케줄 섹션 추가)

**위치:** "비포애프터 사진"(M2) 아래 **"운동 스케줄"** 섹션. **M1 `CardioSection` 구조 복제.**

**동작:**
- **로드:** `loadHome` Promise.all에 `schedule_check` 조회 추가(`select("*").order("on_date",{ascending:false}).limit(60)`) · state `schedule` · `loadSchedule` 재조회 콜백(M1 `loadCardio` 패턴) · `signOut`에 `setSchedule([])`.
- **추가 폼:** 날짜(기본 오늘) · **구분 select(개인운동/PT)** · 메모(선택) → [추가] → `insert({ user_id: me.id, on_date, kind, note })`(하드닝 `.select()`).
- **목록:** 날짜 · **구분 배지**(개인운동=primary톤 / PT=sub톤) · 메모 · 삭제(휴지통, `.select()` 하드닝).
- **빈 상태:** "아직 스케줄 기록이 없어요. 오늘 운동을 체크해보세요."
- 데모/키 가드, me 로드 후 활성, 회원 톤(큰 글씨). lucide(`CalendarCheck`/`Plus`/`Trash2`).
- **구분 라벨 맵**(정적·purge 안전): `const SCHEDULE_KINDS = { personal: "개인운동", pt: "PT" };`

**주의:** 배지 색은 **동적 조립 금지** — `kind==='personal'`이면 primary 클래스, `'pt'`면 sub 클래스를 **정적 분기**(삼항)로.

## 3. 트레이너 조회 — `components/views/MemberScheduleSummary.jsx` (신규 · 읽기)

M1 `MemberCardioSummary` 패턴.

- `supabase.from("schedule_check").select("*").eq("user_id", member.id).order("on_date",{ascending:false}).limit(20)`(트레이너 정책=account 스코프).
- **표시:** 이번 달 요약(**개인운동 N회 · PT N회**) + 최근 목록(날짜·구분 배지·메모). 읽기 전용.
- **기록 0건·데모면 자체 숨김.** `PtWorkoutTab`에 `<MemberScheduleSummary member={member} />` 삽입(M1·M2 요약 옆).

## 5. 주의·엣지

- **M1 패턴 복제** — cardio_log와 구조 동일(스토리지 없음). 새 위험 요소 없음.
