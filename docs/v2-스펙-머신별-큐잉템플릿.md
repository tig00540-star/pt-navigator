# v2 스펙 — 머신별 큐잉 템플릿 (트레이너 큐 자산화)

## 0. 한 줄 요약
머신마다 **트레이너 정본 실행 큐**를 1회 저장해두고, 음성일지 생성 시 그 큐를 권위 있는 기준으로 재생한다. AI는 "음성→일지" 편의와 그날 말한 세션별 포인트만 얹는다. 저장 큐가 없는 머신은 지금처럼 AI가 생성(무회귀).

## 1. 배경 / 왜
- 운동일지는 회원 만족도가 가장 높은 지점이자 일반회원 vs PT회원의 최대 차별점. 그 핵심은 **트레이너의 시그니처 큐잉**이다.
- 현재 `voice-log`의 `method` 큐는 AI가 매번 생성 → 유사 계열(로우로우/하이로우/시티드로우)이 "비슷비슷"하게 나옴. 어느 앱이든 낼 수 있는 일반 큐.
- 큐를 AI 품질에 맡기지 않고 **트레이너 자산으로 소유**하면, 매 일지가 "이 트레이너의 큐"로 고정된다. 방금 종목명=머신명으로 통일(드롭다운)했으므로 큐를 머신에 붙일 **조인 키가 깨끗**해져 지금이 얹기 좋은 타이밍.

## 2. 대전제 / 제약 (인계서 준수)
- 기존 로직·RLS·payload **최소 변경**. 신규 컬럼 + 설정 UI + 프롬프트 주입만. `method` 배열의 스키마/모양·저장 구조는 불변(생성 소스만 바뀜).
- 저장 위치 = **`center_machine` 공용 1벌**(대표 관리). solo(1인)면 대표=트레이너라 곧 본인 큐. 다중 트레이너에서 트레이너별 큐가 필요해지면 후속으로 오버레이 테이블 추가.
- 저장 큐 없는 머신 = **기존 AI 생성 폴백**. 어떤 경우에도 기존 동작이 나빠지지 않는다.

## 3. 데이터 (Phase 1)
- 마이그레이션: `center_machine`에 `cues text[]` 컬럼 추가(nullable). 멱등, 데이터 무변경.
  ```sql
  alter table center_machine add column if not exists cues text[];
  ```
- RLS: **정책 변경 없음**. 컬럼만 추가라 기존 `center_machine` 정책(대표만 쓰기 / 계정 열람) 그대로 적용됨.
- 타입 선택: `text[]`(단순·순서 보존) 추천. 큐별 메타(강조/카테고리)가 필요해지면 그때 `jsonb`로 승격.

## 4. 설정 UI (Phase 1) — `CenterMachineSettings` 확장
- 머신 추가/수정 폼에 **"실행 큐"** 편집 추가. 줄 단위 입력(textarea, 한 줄 = 큐 1개) → 저장 시 `cues text[]`로. 대표(`isOwner`)만 편집, 소속 트레이너는 열람.
- (선택·권장) **"AI 초안 생성" 버튼**: 머신 이름·종류로 큐 3~6개 초안을 뽑아 textarea에 채움 → 트레이너가 다듬어 승인 후 저장. 초기 입력 부담을 크게 줄이면서 최종본은 트레이너 표현으로 남음.
  - 신규 경량 라우트 `POST /api/machine-cues` (Claude, `{name, kind}` → `{cues: string[]}`). `requireTrainer` + owner 게이트. `voice-log` 프롬프트의 method 규칙(구체 실행 큐)과 톤 통일.
- 등록된 장비 칩에 **큐 개수 배지**(예: `큐 4`) 정도로 상태 표시.

## 5. 주입 (Phase 2) — `voice-log/route.js`
- 라우트가 `center_machine`(이름 + `cues`)을 로드. (현재는 `member.machines` 문자열만 힌트로 씀 → 센터 머신 큐까지 확장.)
- summary 프롬프트에 **"[머신 정본 큐]"** 블록 주입: 트레이너가 말한 종목이 등록 머신과 매칭되면, 그 머신의 `cues`를 **method의 우선 기준**으로 쓰고 그날 말한 포인트를 얹어 재구성. 매칭 안 되면 **기존대로 AI 생성**.
- 매칭 키: 종목명=머신명 통일 상태 활용. `name` 우선, 필요 시 `canonical` 보조. 정확/부분 일치 규칙은 구현 시 확정.
- payload·`method` 스키마 불변. 생성 소스만 "AI 순수 생성 → 정본 큐 우선"으로 바뀜.

## 6. 단계 구분
- **Phase 1** (자산화 시작): 스키마 + 큐 편집 UI(+AI 초안). 이것만으로도 "내 큐 저장"이 시작되고, 설정에서 큐를 쌓을 수 있음. 일지 반영은 아직 없음.
- **Phase 2** (일지 반영): voice-log 주입. 실제 일지에 정본 큐가 재생됨.
- Phase 1 완료 후 큐를 어느 정도 채운 뒤 Phase 2를 켜는 순서 권장(빈 큐로 주입 켜봤자 폴백뿐).

## 7. 열린 결정 (구현 전 확정)
1. `cues` 타입: `text[]`(추천) vs `jsonb`.
2. "AI 초안 생성" 버튼 포함 여부(권장: 포함).
3. 매칭 정밀도: `name` 정확 일치 우선 / `canonical` 폴백 / 부분 일치 허용 범위.
4. 공용 1벌로 시작 확정(다중 트레이너 오버레이는 백로그).

## 8. 검증 (구현 후)
- Phase 1: 대표 계정에서 머신에 큐 저장/수정/삭제, 소속 트레이너 열람 전용 확인. AI 초안 → 편집 → 저장 라운드트립.
- Phase 2: 큐 등록된 머신으로 음성일지 생성 → method가 **정본 큐 기반**으로 나오는지, 큐 없는 머신은 기존 AI 생성 유지되는지(폴백 무회귀), 저장 payload/그래프 불변 확인.
