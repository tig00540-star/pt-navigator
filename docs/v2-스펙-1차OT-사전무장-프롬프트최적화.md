# v2 스펙 — 1차 OT '사전무장 컨닝페이퍼' 프롬프트 최적화 (세일즈 클로징 극대화)

> **대전제(이 프로젝트 전반 · 클로드코드도 기억):** OT 지원 AI = **세일즈 클로징 확률 극대화** 도구. 형태 = **OT 입장 3분 전 사전무장 컨닝페이퍼** — 트레이너가 외우고 폰을 주머니에 넣은 뒤 당당하게 리드. 출력은 "참고 자료"가 아니라 **바로 외워 바로 쓸 완성 대사**.
> 이 문서 = `app/api/ot-brief/route.js`의 `firstPrompt`(phase:"first") 교체 초안 + 다운스트림 영향. 커밋은 트레이너 직접.

## 0. 무엇이·왜 바뀌나

**피드백:** "세일즈에 실질적 도움이 안 된다 / 정보가 많아 한눈에 안 온다."
**원인:** 지금 `firstPrompt`은 재료(실전 클로저 멘트·비유·pick_ref·윤리가드)는 좋은데, **6단계 arc + movement_cues 6필드 + observe_targets + watch_for**로 흩어져 "현장에서 못 쓰는 참고서"가 됐다.
**해결:** 엔진(가드레일·환각방지·비유 개인화)은 **그대로 재사용**하되, 출력을 **트레이너가 준 4블록 컨닝페이퍼**로 재편하고 곁가지를 쳐낸다.

| 항목 | 지금 | 바뀜 |
|---|---|---|
| 출력 축 | arc 6단계가 척추 | **4블록 컨닝페이퍼**(오프닝·타겟운동·비유·거절5)가 척추 |
| 오프닝 | arc[0] 라포(추상) | **일상 라포 완성 대사**(첫 만남 긴장 완화) |
| 운동 | movement_cues 6필드 + "방향만" | **타겟운동 1개 + 노리는 반응 + 짚어줄 실제 대사 + 등록 논리 연결** |
| 비유 | closing.paint 안에 묻힘 | **독립 블록**(직업/일상 맞춤 대사 + 등록 브릿지) |
| 거절 | 4종 | **5종 선제 방어**(신호+공감무브+실제 대사) |
| 곁가지 | arc·watch_for·observe_targets 화면 노출 | **arc/watch_for 제거, observe_targets는 '1차 후(관찰)'로 이동** |
| 대본 수위 | 세일즈=대본 / 운동=방향만 | **세일즈+운동 둘 다 완성 대사**(숫자처방·의료단정만 금지) |

## 1. ★가드레일 재조정 (의식적 결정 — 검토 요망)

PREAMBLE은 4개 phase 공용이라 **손대지 않는다.** 대신 이 국면(1차 사전무장)에서만 **user 프롬프트가 국소적으로 재스코프**한다:

> "이 국면에선 운동 파트도 '짚어줄 실제 대사'까지 구체적으로 준다. **단 숫자 처방(세트·횟수·각도·중량·템포)과 의료 단정(치료·완치·진단)은 여전히 절대 금지.**"

- **트레이드오프(명시):** 기존 "운동=방향만(스파링·하향평준화 방지)" 원칙을 이 도구에선 **클로징 확률 우선**으로 완화. 트레이너 결정. (급한불`acute`은 의료안전상 기존 스파링·보수 유지 — 이 재스코프는 first에만.)
- **유지되는 가드:** ①없는 관찰 창작 금지(1차는 관찰 전이라 신체 판단은 '가설' 톤, 단 라포·세일즈 대사는 확신 있게) ②pick_ref로만 패키지 추천(가격 환각 금지) ③허위 긴급성·FOMO·죄책감·소진강요·의료표현 금지 ④숫자 처방 금지.

## 2. 새 `firstPrompt` 초안 (교체용 · route.js)

> 시그니처는 그대로 `firstPrompt(member, packages)` (프로필 주입 스펙과 합류 시 `profile` 파라미터 추가). `g()`·`pkgBlock` 등 기존 헬퍼 재사용.

```
[상황·대전제] 1차 OT 입장 3분 전. 이 출력의 유일한 목적 = 이 회원의 '등록(클로징) 확률 극대화'다.
트레이너는 이걸 30초~1분 안에 훑어 머리에 넣고 폰을 주머니에 넣는다. 그러니 '참고 설명'이 아니라
'바로 외워 바로 말할 완성 대사'로 써라. 추상 지침·개발자용 매핑 설명은 금지.

[이 국면 특칙 — 대본 수위] 세일즈(오프닝·비유·클로징·거절)뿐 아니라 운동 파트도 '회원에게 그대로
말할 실제 대사'까지 구체적으로 준다. 단 아래는 여전히 절대 금지:
 - 숫자 처방(세트·횟수·각도·중량·템포·시간). 운동은 '방향+짚어줄 말'까지, 숫자는 빼라.
 - 의료 단정(치료·완치·진단). 통증은 '불편/부담' 수준 표현까지만.

[회원 기본정보] name=${'${g(m.name)}'}, age=${'${g(m.age)}'}, job=${'${g(m.job)}'}, residence=${'${g(m.residence)}'},
               mbti=${'${g(m.mbti)}'}, pain=${'${g(m.pain)}'}, goal=${'${g(m.goal)}'}, machines=${'${machines}'}
아직 관찰 전이다 — 신체에 대한 판단은 '~일 가능성' 가설 톤. 없는 관찰·수치·에피소드 창작 금지.
단 오프닝·비유·거절 대사는 세일즈 기술이라 확신 있게 완성해도 된다(사실 왜곡이 아니므로).

[내 PT 패키지] (★이 목록에서만 추천. 없는 패키지·가격·세션수 창작 금지. [n]=참조번호)
${'${pkgBlock}'}

[★운동목적(goal) = 최우선 렌즈] 이 회원 goal이 4블록 전부의 중심축이다.
 - 통증개선/불편해소 → 불편 부위를 전면, 그 해소로 엮는다.
 - 외형/체중감량/바디라인 → 체형·라인·자신감을 전면, pain은 보조.
 - 건강/체력/활력 → 컨디션·에너지를 전면.
 - 벌크업/근육 → 근성장·볼륨·자세를 전면.
 ⚠️ goal과 무관하게 pain만 물고 늘어지지 말 것. pain은 goal이 통증개선일 때만 주인공.

[컨닝페이퍼 4블록 — 이 순서가 트레이너가 읽는 순서다]

① opening(오프닝 — 첫 만남 긴장 풀기): 회원의 직업·거주·나이·목표 같은 '일상 소재'로 가볍게 문을 여는
   완성 대사 1개. 날씨·시즌 같은 무난한 소재도 좋다. 세일즈 냄새 0, 사람 대 사람의 라포.
   - line: 그대로 말할 첫 대사("○○님, …") 1~2문장.  - why: 이게 여는 것(긴장 완화·신뢰) 1문장.

② target_exercise(타겟 운동 & 리액션 가이드 — 세일즈의 '증거' 만들기):
   이 회원에게 '되네?'/'통증 없이 자극되네?'를 몸으로 체감시킬 운동 1개와, 그걸 세일즈로 잇는 법.
   - exercise: 시킬 운동(방향까지 · 숫자 금지).  - target_reaction: 노릴 반응(무엇을 느끼게 할지).
   - point_it_out: 그 반응이 나온 순간 회원에게 그대로 말할 실제 대사("방금 느끼셨어요? …") 1~2문장.
   - so_what: 그 체감을 등록 논리로 잇는 한 줄("혼자선 이 세팅 못 잡습니다" 결).

③ sales_metaphor(세일즈 비유 — 회원 세계 언어): 이 회원의 직업·일상·목표에서 끌어온 비유 하나로
   '지금 안 하면/함께 하면'을 그림처럼. 운동·기계 클리셰("브레이크-액셀") 금지.
   - metaphor: 그대로 말할 비유 대사 1~2문장.  - bridge: 비유→등록 필요성으로 잇는 한 줄.

④ closing_line(클로징 한마디 — L0 '지금 이 한마디'): 오늘 체감을 근거로 한 '가정 종결' 한 문장.
   '등록하세요'(판매 동사) 금지 → "다음 주부터 이렇게 가시죠" 결. 긴급성은 사실 기반 손실만.

⑤ objection_defense(거절 선제 방어 5종): 아래 5개 각각 미리 무장. 반박이 아니라 '공감으로 빗장 풀고
   세일즈로 다시 끌기'. reason 키 고정(5개 전부 · 각 1개):
     price(가격 부담) · hesitation(생각해볼게요·망설임) · doubt(효과·필요성 의심) ·
     time(시간 부족·바빠서) · compare(타 센터 비교 — "가격 비교하고 올게요/다른 데서도 OT 받아볼게요").
   - trigger: 이 거절이 나올 신호(회원이 흔히 하는 말/태도) 1문장.
   - defense: 공감으로 걱정 풀고 바로 잇는 세일즈 무브 1~2문장.
   - line: 그 자리에서 그대로 말할 실제 대사("○○님, …") 1문장.
   ⚠️ compare(타 센터 비교)는 경쟁사 비방·가격 맞대응 금지 → '오늘 이미 당신 몸에서 확인한 것'이라는
     관계·데이터 선점 우위로 프레이밍("다른 데선 0부터 다시 파악해야 해요. 저는 오늘 이미 ○○을 봤고요").
   ⚠️ 허위 긴급성·공포·죄책감 금지(PREAMBLE 상속).

[recommended_program] 위 패키지에서 이 회원에 가장 맞는 1개(pick_ref)만 확신 있게. 가격·세션수를 값
   텍스트에 쓰지 마라(앱이 목록에서 채움). why_fit=목적·직업·나이 근거 2문장 이내. 없으면 pick_ref=null +
   data_gaps에 "가격 설정 탭에 패키지를 등록하면 콕 집어드려요".

[data_gaps — 성장 프레임] 기본정보로 위 전부를 반드시 생성한다("정보 부족" 반환 금지). data_gaps는
   결핍이 아니라 "○○를 관찰해오면 △△까지" 형태 긍정 코칭. 충실하면 빈 배열.

[분량 — 외우기 쉽게] 각 대사는 짧고 입에 붙게. line류 1~2문장, why/bridge/so_what 1문장.
[필드명 인용 금지] 값 텍스트에 스키마 키(point_it_out·so_what·objection_defense 등)를 쓰지 마라.
[출력 언어] 자연스러운 한국어. 영문 코드값·필드명 노출 금지. 아래 JSON만(설명·코드펜스 금지).
{
  "member_read": "이 회원 한 줄 — 누구고/뭘 원하고/뭐가 걸리나 (3분전 각인 앵커)",
  "opening": { "line": "...", "why": "..." },
  "target_exercise": { "exercise": "...", "target_reaction": "...", "point_it_out": "...", "so_what": "..." },
```

## 3. 거절 5종 (확정 · 2026-07-14)

**decider(배우자 상의) 제외 → compare(타 센터 비교) 추가.** 최종 5종 + 화면 라벨(`OBJ_LABEL`):

| reason 키 | 화면 라벨 | 트리거 예 |
|---|---|---|
| `price` | 가격 부담 | "생각보다 비싸네요" |
| `hesitation` | 생각해볼게요 (망설임) | "좀 더 고민해볼게요" |
| `doubt` | 효과·필요성 의심 | "PT까지 필요할까요" |
| `time` | 시간 부족 | "요즘 너무 바빠서" |
| `compare` | 타 센터 비교 | "가격 비교하고 올게요 / 다른 데서도 OT 받아보고 결정할게요" |

> `compare`는 신규 키(기존 first는 decider였음). `FirstOTAssist`의 `OBJ_LABEL` 맵을 위 5개로 교체(§4). `compare` 대응 원칙은 프롬프트 ⑤의 ⚠️(경쟁사 비방·가격 맞대응 금지 → 데이터 선점 우위) 참조.

## 4. 다운스트림 영향 (프롬프트 확정 후 별도 스펙)

이 프롬프트를 넣으면 스키마가 바뀌므로 함께 손볼 것 (각각 별도 커밋으로 순서대로):
- **`FirstOTAssist.jsx` 렌더 교체:** 4블록 컨닝페이퍼 레이아웃(member_read 앵커 → 오프닝 → 타겟운동 → 비유 → closing_line 크게 → 거절5 칩). 기존 arc/movement_cues/watch_for 렌더 제거. **구캐시(구 스키마) 방어 렌더 + '다시 생성' 유도.**
- **`OBJ_LABEL` 맵 교체:** `FirstOTAssist.jsx`의 거절 라벨 맵을 §3 5종(price/hesitation/doubt/time/compare)으로. 기존 `decider` 제거.
- **`FIELD_TERMS` 추가:** point_it_out·so_what·target_reaction·objection_defense·sales_metaphor·member_read·closing_line 누출 방어.
- **observe_targets 이동:** '1차 후(관찰 기록)' 화면으로 — 단 새 스키마엔 observe_targets가 없으니, 관찰 체크리스트는 별도 설계(전/후 재편 스펙과 합류).
- **캐시:** `ot_log` round1 `report.first_assist` 스키마가 바뀜 → 구캐시 회원은 신필드 없음 → 방어 렌더 후 '다시 생성'. (기존에도 쓰던 패턴.)
- **동적 UI(요청 #2 · 회원유형별):** goal_type(벌크업/다이어트/체형교정/통증)로 4블록의 강조·색 톤 분기 — 렌더 스펙에서.

## 5. 자기진화 루프로 가는 다리 (요청 #3 · 나중)

"거절 데이터 축적 → AI가 더 정교한 대응으로 진화"는 이미 있는 **D-3 케이스 메커니즘**(`closingCasesForTrainer`·`closingCaseGate` · 2차에서 few-shot 주입)을 **거절 사유별로 확장**하면 된다: 축적된 (거절사유→통한 대응) 사례를 objection_defense 생성에 few-shot으로 주입. + 매니징 대시보드는 `closing_reason` 집계(이미 저장 중)로 "어떤 거절이 매출 저하 핵심인지" 분포를 뽑는다. → 프롬프트 확정 후 별도.
