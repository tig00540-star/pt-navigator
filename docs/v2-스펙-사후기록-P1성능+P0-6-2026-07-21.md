## P1-9 · `components/views/RegisterDueToday.jsx` — 재조회 루프 차단

**문제:** `useEffect` deps가 `members`(배열 **참조**)라, 부모(`app/page.jsx`)의 어떤 state 변경(탭 전환·회원 선택·알림)에도 새 배열 참조가 내려와 `session_log` + `daily_workout_log`를 **통째로 재조회**했다. 형제 위젯 `ReapproachToday.jsx`(`:36` `memberKey = (members||[]).map(m=>m.id).join(",")` + `:55` `[today, memberKey]`)는 이미 옳은 패턴.

**처방(적용됨):**
- deps를 배열 참조 → **id 문자열 키**로. 여기선 "재등록 대상 = pt 회원 집합"만 관심이므로 pt로 좁힌 키: `const ptKey = (members||[]).filter(m=>viewFor(m)==="pt").map(m=>m.id).join(",")` → `useEffect(..., [ptKey])`.
- `select("*")` → **실제 읽는 컬럼만**: `session_log` = `id, user_id, started_at, created_at, sessions_total, service_sessions` · `daily_workout_log` = `user_id, contract_id, voided`(`activeContract`·`remainingSessions`·`reregisterDue`가 읽는 셋 = `ChurnRiskToday`와 동일). max-rows·전송량 낭비 제거.
- setState는 IIFE 내부에서만 + `cancelled` 가드(동기 setState 0 → set-state-in-effect 회피, `ReapproachToday` 골격 그대로).

**주의:** 데모(`!supabase`)는 `[]` 유지 → 카드 미표시(라이브 전용 위젯). 정렬은 잔여 유료 적은 순(급한 순).

## P1-10 · `components/views/MyStats.jsx` — 파생 집계 useMemo 래핑

**문제:** `useMemo`/`useCallback` 0개였다. 컴포넌트에 `reportOpen` 모달 state가 있어, **월간 리포트를 여닫을 때마다(그 외 어떤 렌더에도)** `O(계약×로그)` reduce(`remAll`)와 로그 전체 순회 여러 개가 통째로 다시 돌았다. 참고 정답: `app/admin/page.jsx`(useMemo 다수).

**처방(적용됨):**
- 파생 집계 전체를 **하나의 `useMemo`**로 묶음(`ym`·`rev`·`closing`·`pay`·`remAll`·`doneAll`·drill-down 등 한 번에 반환). deps는 입력만: `[members, contracts, logs, otRows, schemes, runs, uid, goals, contractNames]`.
- 하위 헬퍼(`revenueByTrainer`·`remainingSessions`·`closingStats` 등)가 **순수**(입력만 읽고 부수효과 없음)라 메모이즈 안전. 훅은 early return(`loading`) **앞**이라 호출 순서 불변(Rules of Hooks).

**함께 들어간 것(P0-6과 교차):** 이 파일의 `session_log`·`daily_workout_log` 조회는 `fetchAllRows`로 감쌈(아래 P0-6). 급여·잔여·매출의 입력이라 잘리면 **돈 숫자가 틀린다**.

## P1-12 · `app/api/ot-brief/route.js` — 프롬프트 입력 상한(심층방어)

**문제:** 이 라우트는 `requireTrainer`(인증+구독)로 이미 게이트되지만, **프롬프트에 직삽입되는 필드**에 길이·개수 상한이 없으면 인증 통과자가 거대한 본문으로 토큰을 폭주시켜 비용·지연 공격이 가능하다(심층방어 공백).

**처방(적용됨) — `POST` 초입:**
- `MAX_BODY_BYTES = 256 * 1024` — `JSON.stringify(body).length`가 초과면 거부(순환참조 등 stringify 실패도 거부). 정상 페이로드(회원+관찰+케이스 몇 건)는 수 KB라 256KB는 넉넉.
- `MAX_CLOSING_CASES = 20` — 클로징 케이스 배열은 `slice(0, 20)`으로 **앞쪽(우선순위 높은)만** 통과, 초과분 조용히 버림(토큰이 배열 길이에 선형 비례 → 개수 상한이 실질 방어. 통상 한 자리수).
- 장비 블록도 `machines.slice(0, 120)` 상한(프롬프트용 soft 목록).
- (참고) `maxTokens`는 phase별: first 8192 · second+케이스 6144 · 그 외 5120. `maxDuration=180`.

**주의:** 상한 초과는 **거부(4xx)**이지 조용한 잘림이 아니다(케이스 개수만 예외적으로 슬라이스). 정상 사용에는 절대 안 걸리는 값.

## P0-6 · `lib/fetchAllRows.js` 신설 — 1000행 잘림 원천차단

**문제(시한폭탄):** 필터·limit 없는 `select`는 PostgREST **Max rows(기본 1000)에서 에러·경고 없이 잘린다**. 그 행을 세어 계산하는 급여·매출·잔여·이탈 숫자가 "틀린 채로 멀쩡히" 뜬다. `daily_workout_log`는 수업 1건=1행이라 가장 빨리 참(하루 15~16 수업이면 센터 단위로 몇 주, 트레이너 1인도 두 달여). `ORDER BY` 없어 비결정 잘림도 동반.

**처방(적용됨):**
- `fetchAllRows(makeQuery, { pageSize = 1000, orderColumn = "id" })` — `.range()`로 페이지를 끝까지 훑어 전 행 수집. **유니크 컬럼(기본 id=PK)으로 정렬 강제**(페이지 경계 중복·누락 방지 · `created_at`류는 동시각 충돌로 부적합). 반환 `{ data, error }` = supabase와 동형 **드롭인**. `error`는 그대로 전달.
- 빌더는 한 번만 await되므로 **thunk로 받는다**: `fetchAllRows(() => supabase.from("daily_workout_log").select("*"))`.
- **계산식 불변** — 잘려 있던 입력을 완전하게 채울 뿐이라 숫자가 더 정확해지기만 한다.

**적용 대상(무필터 전체표 4쿼리 2파일):**
- `app/admin/page.jsx` — 센터 전체 `session_log`·`daily_workout_log`(가장 빨리 참).
- `components/views/MyStats.jsx` — 본인 `session_log`·`daily_workout_log`.
- 나머지는 `.eq("user_id")` 단일회원(소량)이거나 이미 필터 있어 **대상 아님**.

**검증(커밋 시점):** lint·prod build green. 실렌더 — 내 실적 급여·관리자 매출이 적용 전과 **동일**(현재 전 테이블 <1000이라 결과 불변 = 드롭인 충실 확인), 에러 0.
