## 1. 제품 정의

### 1.1 한 줄

**개인 PT 트레이너가 OT부터 재등록까지 회원 한 명을 끝까지 책임지도록 돕는 AI 세일즈·관리 도구.**

### 1.2 해결하는 문제

트레이너가 회원을 못 만드는 이유는 운동을 못 가르쳐서가 아닙니다. **자기 가치를 말하지 못해서**입니다.
그리고 그 원인은 스킬이 아니라 정체성 충돌입니다 — "나는 파는 사람이 아니다", "거절이 두렵다".

| 문제 | 현장에서 벌어지는 일 |
|---|---|
| 세일즈 저항 | 등록 얘기를 꺼내지 못하고 수업만 잘하다 회원을 놓침 |
| 근거 없는 클로징 | 감으로 밀어붙이거나 할인으로 도망감 |
| 거절 공포 | "생각해볼게요"를 들으면 관계가 끝났다고 느끼고 다시 연락하지 못함 |
| 기록의 증발 | 1차에서 본 것을 2차에서 못 씀. 매번 처음부터 |
| 재등록 누락 | 잔여 세션이 끝나가는 걸 뒤늦게 알아챔 |

### 1.3 해결 방식

**세일즈를 다시 정의합니다 — 파는 행위가 아니라 책임지는 행위로.**

그리고 그 재정의를 구호가 아니라 제품 구조로 구현합니다.

| 원칙 | 제품에서의 구현 |
|---|---|
| 설득이 아니라 증명 | 1차 관찰 입력 → 2차 브리핑의 유일한 근거로 사용 |
| AI는 스파링 파트너 | 트레이너가 먼저 관찰·입력해야 AI가 동작. 빈 입력으로 생성 불가 |
| 거절은 과정의 한 점 | 보류 사유 + 재접근일을 기록 → 그날 '오늘' 화면에 자동 등장 |
| 회원이 만족해야 재등록 | 회원앱으로 본인 변화를 직접 보여줌 |

### 1.4 하지 않는 것

- **트레이너 대신 판단하지 않습니다.** 관찰 입력을 건너뛴 자동 생성 기능은 만들지 않습니다.
- **의료 영역을 침범하지 않습니다.** 진단·치료·교정 표현 금지.
- **수치를 처방하지 않습니다.** 세트·횟수·중량·각도는 AI가 정하지 않고 방향과 원리만 제시합니다.
- **성과를 보장하지 않습니다.** 매출·체중 같은 숫자를 약속하지 않습니다.
- **회원을 압박하지 않습니다.** 허위 긴급성·기한 조작·죄책감 유도는 AI 프롬프트 단계에서 금지돼 있습니다.

## 2. 사용자

### 2.1 주 사용자 — 개인 PT 트레이너

| 항목 | 내용 |
|---|---|
| 상황 | 혼자 회원을 만들고 지켜야 함. 조직의 세일즈 교육을 받지 못함 |
| 사용 맥락 | **수업 입장 직전 3분**, 폰으로 흘긋. 헬스장은 밝고 손은 바쁨 |
| 원하는 것 | 지금 이 회원에게 무슨 말을 해야 하는지 |
| 원하지 않는 것 | 읽어야 할 자료, 공부거리, 정답 강요 |

이 사용 맥락이 제품 전반의 제약을 만듭니다 — 밝은 테마 전용, 결론 먼저, 바로 말할 수 있는 완성 대사,
하단 탭 고정.

### 2.2 부 사용자 — 센터 원장

담당 트레이너들의 실적·급여·공지를 관리합니다. 매출과 클로징 전환은 실데이터로 보지만,
**트레이너 QC 지표는 아직 데모**입니다(§7.2).

### 2.3 최종 수혜자 — 회원

트레이너가 발급한 링크로 자기 기록을 봅니다. **프리미엄 구독 전용**입니다.
회원이 자기 변화를 눈으로 보는 것이 재등록의 동기가 된다는 것이 이 축의 가설입니다.

## 3. 핵심 사용자 흐름

### 3.1 OT → PT 등록 (제품의 중심 흐름)

```
[등록]  회원 문진 입력 (17필드 · 종이 문진표 → 앱 옮겨적기)
   │
[1차 OT 준비]  AI 사전무장 — 오프닝·타깃 종목·클로징 대사·거절 5종 방어
   │
[1차 OT 수업]  현장 메모
   │
[1차 피드백]  ★ 트레이너가 관찰을 입력 ★  ← 제품의 심장
   │           (관찰 · 반응/성향 · 진짜 목적 · 종합 소견 · 1차 클로징 결과)
   │
[2차 OT 준비]  AI 브리핑 — 1차 관찰을 유일한 근거로 등록 당위성 생성
   │
[2차 OT 수업]  '증명' 수업 → 클로징
   │
[클로징 결과 기록]  성공 / 보류 / 실패 + 접근 방향 + 사유
   │
   ├─ 성공 → [PT 등록 확정] 계약 금액 입력 → 상태 전환 → PT 뷰
   └─ 보류 → 재접근일 기록 → 그날 '오늘' 화면에 자동 등장
```

**중요한 설계 결정 — 클로징 성공은 자동으로 PT 등록이 되지 않습니다.**
트레이너가 계약 금액을 넣고 명시적으로 확정해야 상태가 바뀝니다. "성공했다고 말한 것"과
"실제로 결제된 것"은 다른 사건이고, 이걸 섞으면 매출 집계가 오염되기 때문입니다.

### 3.2 PT 회원 관리 → 재등록

```
[수업]  → 음성 일지(녹음 → AI 구조화) → 카톡 복사 → 저장 + 세션 차감
   │
[누적]  세트·무게 · 인바디 · 사진 · 회차
   │
[잔여 세션 임박]  '오늘' 화면에 재등록 도래로 자동 등장
   │
[재등록 준비]  AI 브리핑(계약·빈도·최근 일지 근거) → 재등록 대화
   │
[결과 기록]  성공 / 보류 / 실패
```

### 3.3 회원의 흐름

```
트레이너가 링크 발급 (프리미엄) → 회원이 링크 + 휴대폰 뒤 4자리로 로그인
   │
[내 기록]  오운완 · 달력 · 수업일지 · 인바디 · 무게 추이
[기록 남기기]  개인운동 · 유산소 · 사진
   │
기록이 쌓임 → 오운완 연속·뱃지·트레이너 포상 → 다시 옴
```

## 5. 오운완 (출석 게임화)

재등록 동기를 만드는 회원 참여 장치입니다.

### 5.1 판정 규칙 — 설계의 핵심

**오운완은 버튼이 아닙니다. 실제 기록에서 파생됩니다.**

한 회원의 하루(한국 시간 기준)에 아래 중 **하나라도** 있으면 그날 1개로 셉니다.

1. 유산소 기록
2. 개인운동 체크
3. 수업 일지 (단, 무효 처리·노쇼 제외)

버튼으로 만들면 누르기만 하면 되므로 부정의 원천이 됩니다. 기록에서 파생시키면
**출석을 조작하려면 실제로 기록을 남겨야 합니다.**

### 5.2 지표와 보상

| 지표 | 정의 |
|---|---|
| 누적 | 전체 오운완 일수 |
| 이번 달 | 당월 일수 |
| 연속 | 연속 일수. 오늘이 아직 없으면 어제부터 세어 "오늘 아직"도 연속을 유지 |

**뱃지 5종** — 첫 오운완(1) · 불꽃 7일(연속 7) · 꾸준 10회 · 든든 30회 · 골드 60회.
앞쪽을 촘촘하게 둔 것은 초반 이탈을 막기 위해서입니다.

**포상** — 트레이너가 누적 N회 도달 시 줄 보상을 직접 정의합니다. 회원앱에 진행바로 표시됩니다.

### 5.3 랭킹

트레이너만 봅니다. **회원에게는 랭킹을 공개하지 않습니다** — 하위권 회원의 사기를 떨어뜨려
목적(동기 부여)과 반대로 작동하기 때문입니다. 회원은 자기 진행률만 봅니다.

### 5.4 한계

유산소·개인운동은 회원 자기신고입니다. 신뢰 기반이며 검증 수단이 없습니다.
포상 지급 마킹은 미구현이라 실제 지급 여부는 앱 밖에서 관리됩니다.

## 6. 비기능 요구사항

### 6.1 권한과 격리

**계정 단위 멀티테넌트.** 모든 데이터 테이블이 계정 스코프로 잠겨 있고, 정책은 로그인 세션에서
계정을 도출하는 헬퍼 함수를 축으로 삼습니다.

| 역할 | 범위 |
|---|---|
| 원장 | 계정 전체 회원·계약·매출. 추가로 장비·공지·급여 설정 쓰기 |
| 트레이너 | 계정 내 **본인 담당 회원만** |
| 회원 | 자기 것만. 읽기는 전용 안전 뷰, 쓰기는 자가입력 3종 |

**회원은 트레이너용 데이터에 원천적으로 접근할 수 없습니다** — 회원 계정에는 트레이너 레코드가
없어 트레이너 스코프 정책이 전부 거짓이 되고, 결과가 0행이 됩니다.

**금지 규율** — 누구나 열리는 정책(`using(true)`)을 추가하지 않습니다. 보안 모델이 무너집니다.

### 6.2 구독 게이팅 — 2층

| 층 | 조건 | 막히는 것 |
|---|---|---|
| 1층 | 구독 활성 + 미만료 | 앱 전체. 로그인은 되지만 모든 데이터가 0행 |
| 2층 | 프리미엄 등급 | 회원앱 |

1층은 **별도 게이트 코드가 아니라 계정 도출 헬퍼 안에 조건으로 심어** 전 테이블을 한 번에 잠급니다.
게이트를 빠뜨린 테이블이 생길 수 없는 구조입니다.

2층은 **이중 잠금**입니다 — 링크 발급이 서버에서 거절되고, 회원 로그인 시에도 등급을 재확인합니다.
따라서 프리미엄을 해지하면 **이미 배포된 링크도 그날부터 열리지 않습니다.**

구독 상태 조회가 실패하거나 결과가 비면 **잠그는 쪽으로 처리합니다.** 조회 실패를 통과로 두면
결제벽이 우회되기 때문입니다.

### 6.3 데이터 무결성

- **쓰기 실패는 조용할 수 있습니다.** 권한에 막히면 에러 없이 0행이 돌아옵니다.
  모든 쓰기는 갱신된 행을 돌려받아 확인하고 실패를 사용자에게 알려야 합니다.
- **캐시와 결과의 공존.** 2차 OT 행에는 AI 브리핑 캐시와 클로징 결과가 함께 있습니다.
  각 저장 경로는 자기 필드만 갱신해야 하며, 통째로 덮어쓰면 다른 쪽이 사라집니다.
- **집계는 서버에서.** 오운완 누적은 클라이언트에서 세면 조회 한도에 갇혀 누락됩니다.
  전용 함수로 서버 집계합니다.

### 6.4 AI 안전장치

| 장치 | 내용 |
|---|---|
| 도메인 분리 | 운동 파트는 스파링, 세일즈 파트는 실전. 프롬프트에서 역할을 나눔 |
| 윤리 가드 | 허위 긴급성 · 압박 · 죄책감 유도 금지 |
| 환각 방지 | 가격은 AI가 만들지 않고 앱이 조회. 없는 정보는 "부족한 정보"로 명시 |
| 응답 파싱 | 3단 방어. **부분 파싱 결과는 채택하지 않음** |
| 의료 경계 | 급한불은 단정하지 않고 보수적으로. 필요 시 전문의 안내 |

### 6.5 접근성 · 표현

- **밝은 테마 전용.** 헬스장 조명에서 어두운 화면은 반사로 더 안 보입니다.
- 본문 명도 대비 **4.5:1 이상**. 배경 3종(페이지·카드·입력칸) 모두에서 측정합니다.
- 색은 역할입니다. 레드 채움 = 누르는 곳, 레드 텍스트 = 보는 곳. 금액을 초록으로 칠하지 않습니다.
- 서체는 Pretendard 한 벌. 굵기와 자간으로만 위계를 만듭니다.
- → 상세는 브랜드 가이드라인

## 7. 알려진 리스크

### 7.1 AI 응답 지연 — 최우선

**2차 브리핑 실측 약 45초, 1차는 그보다 김.** 스트리밍이 없어 그동안 화면이 비어 있습니다.

이것은 성능 문제가 아니라 **제품 콘셉트와의 충돌**입니다. 이 제품은 "수업 입장 직전 3분에
훑는 컨닝페이퍼"를 표방하는데, 그중 1분을 빈 화면으로 씁니다. 과거 실제로 타임아웃이 나서
브리핑이 아예 표시되지 않은 사고도 있었습니다.

**대응 방향** — 스트리밍 도입 / 사전 생성(수업 전날 배치) / 대기 중 유용한 정보 표시.
셋 중 하나는 베타 중에 결정해야 합니다.

### 7.2 관리자 QC 데모가 실측처럼 보임

QC 섹션은 전부 하드코딩인데 **화면에 데모 표시가 없습니다.** 원장 대상 시연에서 실측으로
오인될 수 있고, 이는 신뢰 사고로 직결됩니다.

**대응** — 베타 전에 배지를 붙이거나 섹션을 숨깁니다. 실데이터화는 행동 계측 수집이 선행돼야 하므로
단기간에 불가능합니다.

### 7.3 인증 고착

토큰 갱신이 실패하면 앱 전체가 로딩 화면에 영구 정지합니다. 타임아웃도 재시도도 없습니다.
현장에서 "앱이 안 켜져요"로 나타납니다.

### 7.4 관측성 부재

트레이너가 "안 돼요"라고 했을 때 서버에 남는 흔적이 부족합니다. 일부 로깅이 추가됐으나
요청 추적은 되지 않습니다. 베타 중 장애 대응 속도를 결정하는 요인입니다.

### 7.5 프롬프트 인젝션

트레이너·회원의 자유 입력이 그대로 프롬프트에 들어갑니다. 방어가 없습니다.
현재는 자기 계정 데이터만 영향받으므로 피해 범위가 제한적이지만, 미해결 과제입니다.

### 7.6 회원 계정 생성 중단

회원 최초 로그인은 여러 단계를 거치는데, 중간에 끊기면 인증 계정만 생기고 연결이 안 돼
**해당 회원이 영구히 로그인하지 못하는** 상태가 될 수 있습니다. 발생 가능성은 낮게 평가됩니다.

## 8. 현재 상태와 다음 단계

### 8.1 지금 상태

**베타 준비 완료.** 트레이너 코어(OT/재등록 AI · 음성일지 · 수업 차감 · 급여 · 인바디 · 추이)와
회원앱 v1, 수익화 스위치(구독 2층 게이트)가 모두 동작합니다.

### 8.2 미구현 — 세 덩어리

**① 리포트·통계** — 공통 토대는 **클로징 통계**입니다. 원천 데이터(클로징 결과·접근 방향·사유·
회원 스냅샷)는 **이미 쌓이고 있는데** 이를 읽는 화면이 없습니다. 하나 만들면 그 위에 트레이너
강약점 · 센터 강약점 · 팀 KPI 실데이터화가 얹힙니다.

이것이 **자기진화 루프의 핵심 레버**입니다 — 트레이너가 자기 클로징 패턴을 알게 되면
제품이 "도구"에서 "코치"가 됩니다.

**② 스케줄 자동화** — 뼈대(예약·보드)는 있고, 트레이너 가능시간 설정 → 회원 슬롯 선택 →
관리자 배정이 미착수입니다.

**③ 회원 자가입력 심화 + 회원 AI** — 식단 사진 분석, 루틴 자동 생성. 대부분 프리미엄 AI라
수익화 게이트가 이미 깔린 지금 자연스럽게 열립니다.

### 8.3 수익화 잔여

| 항목 | 상태 |
|---|---|
| 구독 스키마 · 2층 게이트 | 완료 |
| 결제 자동화(웹훅) | **미구현 — 활성/강등이 수동** |
| 카카오 로그인 | **미구현** — 가입 마찰 감소용 |
| 관리자 활성/등급 버튼 | 미구현 — 현재 수동 SQL |

**결제 자동화 미구현이 베타 확장의 실질적 병목입니다.** 계정이 늘면 수동 활성이 감당되지 않습니다.

### 8.4 권장 순서

```
1. 베타 전 위생   — QC 데모 라벨 · AI 대기 UX · 인증 고착 방어
2. 결제 자동화     — 수동 활성 병목 해소
3. 클로징 통계     — 리포트류 공통 토대 + 자기진화 루프
4. 스케줄 에픽
5. 회원 AI
```

1번은 **베타 시작 전에** 끝나야 합니다. 나머지는 베타 피드백을 보고 순서를 바꿔도 됩니다.

## 9. 성공 지표

### 9.1 제품이 작동하는지

| 지표 | 의미 |
|---|---|
| **OT → PT 전환율** | 제품의 존재 이유. 도입 전후 비교가 유일하게 중요한 숫자 |
| 1차 관찰 입력률 | 이게 낮으면 2차 AI가 재료 없이 돌고 제품 철학이 무너짐 |
| 2차 브리핑 생성률 | 실제로 무장하고 들어가는지 |
| 클로징 결과 기록률 | 통계·자기진화의 원천. 낮으면 ①이 불가능 |
| 보류 → 재접근 실행률 | "거절은 과정의 한 점"이 실제로 작동하는지 |

### 9.2 리텐션

| 지표 | 의미 |
|---|---|
| 주간 활성 트레이너 | 도구가 습관이 됐는지 |
| 음성 일지 작성률 | 가장 반복되는 행동 |
| 회원앱 재방문율 | 회원 축 가설의 검증 |
| 오운완 연속 유지율 | 게임화가 실제로 동기를 만드는지 |

### 9.3 경계해야 할 지표

**AI 생성 횟수를 성공 지표로 삼지 않습니다.** 많이 생성하는 것이 목적이 아니라 트레이너가
잘 말하는 것이 목적입니다. 생성 횟수는 비용 지표로만 봅니다.

## 10. 열린 질문

1. **AI 지연을 어떻게 해결할 것인가** — 스트리밍 / 사전 생성 / 대기 UX 중 무엇인가
2. **클로징 통계를 어디까지 보여줄 것인가** — 트레이너 본인에게만인가, 원장도 보는가.
   원장이 보면 감시 도구가 되어 "트레이너 편"이라는 브랜드 약속과 충돌할 수 있음
3. **회원 자기신고를 얼마나 믿을 것인가** — 오운완 부정이 실제로 문제가 되는가
4. **가격 모델** — 좌석당 월정액 초안. 회원 수는 과금축이 아니고 수업 수 종량제는 배제
5. **인바디 자동 연동** — 협약이 필요해 현재는 종이 → 수동 입력. 언제 시도할 것인가
