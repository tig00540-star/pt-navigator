## 0. 설계 원칙 (합의)

- **"오늘 뭐 하지"를 한 곳에.** 지금 트레이너의 '해야 할 것' 신호(재접근·재등록·미마감)가 **회원 탭 상단에 위젯 3개로 흩어져** 있고, 첫 화면(스케줄)엔 없다. 이걸 **새 상시 탭 "할일" 하나로 통합**한다.
- **자동 파생 + 수동 메모, 한 리스트.**
  - **자동** = 기존 데이터에서 규칙으로 뽑음(테이블 신설 없음). 순수함수는 `lib/memberStatus.js`에.
  - **수동** = 트레이너가 직접 적는 자유 할일(체크박스). 유일한 신설 테이블 `trainer_todo`.
- **★ 중복 해소 = 흡수(재구현 아님).** 기존 3위젯(`ReapproachToday`·`RegisterDueToday`·`RegisterReapproachToday`)을 **할일 탭 안으로 옮겨** 렌더하고, **회원 탭(`MemberListTab`)에서는 제거**한다. 한 회원이 두 군데 뜨는 중복을 없앤다. **컴포넌트 자체는 그대로 재사용**(검증된 fetch·판정 로직을 다시 안 짬 = 회귀 최소).
- **스코프 = 트레이너 개인.** 원장→트레이너 할일 배정은 이번 범위 밖(협업/권한 별건). ⑦에서 `trainer_id`·RLS·"내 회원" 자동 필터가 이미 완비돼 개인 스코프가 자연스럽다.
- **첫 화면은 스케줄 유지.** 할일은 스케줄 **다음** 상시 탭. 하루의 축(스케줄 보드)은 건드리지 않는다.
- **회귀 안전:** 자동 신호는 기존 파생 함수·컴포넌트 재사용이라 로직 변화 0. 수동은 새 테이블·새 컴포넌트라 기존과 격리. **C(추가)와 D(제거)를 나눠** 통합이 붙는 걸 먼저 확인한 뒤 위젯을 뗀다.

## 2. 데이터 모델 — 수동 할일 테이블 (마이그레이션 1개)

### 2-1. `trainer_todo` — 트레이너 개인 할일

```sql
-- docs/migrations/2026-07-12-trainer-todo.sql  (파일명 날짜는 실제 실행일 기준)
create table if not exists trainer_todo (
  id          uuid primary key default gen_random_uuid(),
  account_id  uuid not null default auth_account_id(),
  trainer_id  uuid not null default auth.uid(),   -- 소유자 = 로그인 트레이너
  body        text not null,
  done        boolean not null default false,
  due_date    date null,                           -- 선택: 마감일(있으면 도래분을 '오늘' 강조)
  member_id   uuid null,                           -- 선택: 회원 연결(탭 시 그 회원으로)
  created_at  timestamptz not null default now(),
  done_at     timestamptz null
);

create index if not exists trainer_todo_owner_ix on trainer_todo(account_id, trainer_id);
create index if not exists trainer_todo_open_ix  on trainer_todo(trainer_id) where done = false;

alter table trainer_todo enable row level security;

-- ★ 본인 것만 전권(SELECT·INSERT·UPDATE·DELETE). 프로젝트 최초의 trainer_id 스코프 테이블.
--   토글(done)·삭제가 있으므로 UPDATE·DELETE 정책 필수(총정리 트러블슈팅: 정책 없으면 조용한 실패).
create policy trainer_todo_all on trainer_todo for all
  using  (account_id = auth_account_id() and trainer_id = auth.uid())
  with check (account_id = auth_account_id() and trainer_id = auth.uid());
```

- **왜 `for all` 단일 정책:** 개인 소유 데이터라 읽기·쓰기 축이 동일(`account+본인`). 페이롤(원장만 쓰기)과 달리 역할 분기 없음.
- **`default auth_account_id()`/`auth.uid()`:** INSERT 때 클라가 소유 필드 안 보내도 서버가 채움(⑦ 헬퍼 재사용 · pay_scheme 패턴과 동일). 클라 INSERT payload = `{ body, due_date?, member_id? }`만.
- **레거시/회귀:** 신설 테이블이라 기존 무영향. 데모(키 없음)에선 `if(!supabase)` 가드로 로컬 상태만.

> ⚠️ Supabase 대시보드에서 실행 후 동일 SQL을 `docs/migrations/`에 기록본 커밋(git엔 기록만). `auth_account_id()`가 있는지, `default`로 넣어도 RLS `with check`를 통과하는지 실증(insert 후 `.select()`로 1행 확인).

## 3. lib — 순수함수 (`lib/memberStatus.js`)

기존 재사용(변경 0): `reapproachToday` · `reregisterReapproachToday` · `reregisterDue` · `activeContract` · `remainingSessions` · `viewFor` · `latestContract`. **신규 2개만 추가**(순수·now 미참조·today/now 주입):

```js
/**
 * 할일 4 — "미확정 클로징": OT 회원 중 2차(round=2) 관찰은 했으나 클로징 결과 미기록.
 * ot_log round=2 행이 있고 closing_result가 비었으면 대상. member view가 'ot'인 회원만(성공→PT 전이면 제외).
 * @param {Array} r2rows  ot_log round=2 행 [{ user_id, ot_round, closing_result }]
 * @param {Set|Array} otMemberIds  현재 view==='ot'인 회원 id 집합(호출부가 viewFor로 산출)
 * @returns {Array<{user_id}>}  결과 미기록 회원.
 */
export function unclosedClosings(r2rows, otMemberIds) {
  const ids = otMemberIds instanceof Set ? otMemberIds : new Set(otMemberIds || []);
  const rows = Array.isArray(r2rows) ? r2rows : [];
  const out = [];
  for (const r of rows) {
    if (!r || r.user_id == null) continue;
    if (r.ot_round !== 2) continue;
    const empty = r.closing_result == null || r.closing_result === "";
    if (empty && ids.has(r.user_id)) out.push({ user_id: r.user_id });
  }
  return out;
}

/**
 * 할일 5 — "미처리 예약": 지난 예약(start_at < now)을 완료/취소 안 한 것.
 * appointment status='booked' + start_at 과거. 완료(done)·취소(canceled)는 제외.
 * @param {Array} appts  appointment 행 [{ id, user_id, start_at, status }]
 * @param {string} nowISO  기준 시각 ISO(호출부 주입 · 모듈 내 now 금지).
 * @returns {Array}  과거 미처리 예약(start_at 오름차순 = 오래된 것 먼저).
 */
export function pastDueAppointments(appts, nowISO) {
  const rows = Array.isArray(appts) ? appts : [];
  return rows
    .filter((a) => a && a.status === "booked" && typeof a.start_at === "string" && a.start_at < nowISO)
    .sort((a, b) => (a.start_at < b.start_at ? -1 : 1));
}
```

> `start_at`은 `toISOString()`(UTC Z) 저장이라 `nowISO`도 `new Date().toISOString()`로 주입하면 사전식 비교 안전(reapproach 날짜비교와 동일 결). 컴포넌트가 `now`를 읽어 주입, 판정은 순수함수.

## 4. UI — `components/views/TodoTab.jsx` (신설)

트레이너 상시 탭. **자동 섹션(1~5) + 수동 섹션**을 위→아래로. 각 자동 항목 탭 = `onSelect(user_id[, tab])`로 그 회원·해당 탭 이동.

### 4-1. 자동 섹션 (1~3 = 기존 위젯 재사용)

```jsx
<ReapproachToday members={members} onSelect={(id) => onSelect(id, 1)} />        {/* 1차 OT */}
<RegisterDueToday members={members} onSelect={(id) => onSelect(id, 11)} />      {/* 재등록 */}
<RegisterReapproachToday members={members} onSelect={(id) => onSelect(id, 11)} />
```

- 세 컴포넌트는 **각자 빈배열이면 `return null`**(이미 그렇게 구현됨) → 할 일 없으면 섹션 자체가 안 뜸. 그대로 이관.
- 현재 `onSelect(id)` 시그니처는 유지(위젯 내부는 안 건드림). **탭 목적지 매핑은 TodoTab이 감싸서** 결정: 재접근(OT)=1차 OT(tab 1), 재등록 계열=재등록(tab 11). (page.jsx의 기존 `onSelect`는 무조건 tab 1로 갔지만, 할일에선 소스별로 알맞은 탭.)

### 4-2. 자동 섹션 (4·5 = 신규)

기존 위젯과 **같은 시각 패턴**(둥근 카드 · 색 헤더 · count 배지 · 회원명 + 컨텍스트 + `ChevronRight`)으로 인라인 렌더. 빈배열이면 미표시.

- **미확정 클로징**(amber/rose 계열, OT 미마감 톤): `unclosedClosings(round2Rows, otIds)` → 회원명 + "2차 클로징 결과 미기록" → 탭 시 `onSelect(id, 2)`(2차 OT).
- **미처리 예약**(slate/zinc 중립 톤): `pastDueAppointments(appts, nowISO)` → 회원명 + `start_at`(월/일 시:분) + "N일 지남" → 탭 시 `onSelect(id, 9)`(스케줄).
  - fetch: `appointment` `status='booked'` + `start_at < now` (트레이너 RLS 자동 스코프). 주간창 제한 없이(과거 전체) 당기되 `limit`으로 상한.

### 4-3. 수동 섹션

- **입력줄:** text 인풋 + (선택) 마감일 date + "추가" 버튼 → `trainer_todo` INSERT(`{ body }` 최소, `due_date` 있으면 포함) → `.select()` 하드닝 → 성공 시 리스트 앞에 낙관적 추가.
- **리스트:** 미완료 먼저(생성 역순), 완료분은 흐리게 아래(or 토글로 숨김). 각 행 = 체크박스(토글 `done`/`done_at`) + body + (있으면) 마감일 배지 + 삭제 버튼(인라인 확인).
  - 토글 UPDATE·삭제 DELETE → `.select()`로 `data.length>0` 확인(정책 없으면 조용한 실패 = 트러블슈팅). 마이그레이션의 `for all` 정책이 이걸 보장.
- **회원 연결(선택·후속 여지):** v1은 `member_id` 컬럼만 두고 UI 연결은 생략 가능(입력줄에 회원 선택 셀렉트는 다음 증분). 최소 버전 = 자유텍스트 체크리스트.
- **빈 상태:** 자동·수동 다 없으면 "오늘 할 일 없음 — 스케줄만 소화하면 돼요" 류 한 줄.

### 4-4. 렌더 규칙 (CLAUDE.md 준수)

- 색·클래스는 **정적 토큰/`C` 맵만**(`text-ink/sub/muted/primary-strong`, `bg-card/elevate`, `border-line` 등). 동적 클래스 문자열 금지 — 기존 위젯이 쓰는 토큰 그대로.
- 아이콘 `lucide-react`만(예: `CheckSquare`·`ListTodo`·`CalendarClock`·`Plus`).
- 데모 가드: `if(!supabase)` 시 자동은 기존 위젯이 알아서 미표시, 수동은 로컬 상태만(빈 리스트+추가 데모).

## 8. 회귀·안전

- 자동 1~3은 **컴포넌트·판정 로직 재사용**(변경 0) → 위치만 회원 탭→할일 탭. 표시 결과 동일.
- 자동 4~5는 **신규 순수함수 + 신규 섹션**, 빈배열이면 미표시 → 없던 회원엔 아무 변화 없음.
- 수동은 **신설 테이블·컴포넌트**로 기존과 격리. 정책 없으면 토글/삭제가 조용히 실패하므로 **A에서 `for all`(UPDATE·DELETE 포함) 정책 필수**.
- `trainer_todo`는 `trainer_id=auth.uid()` 스코프라 남의 할일 안 보임(프로젝트 최초 trainer 스코프 테이블 — 개인 메모라 적정).
- C·D 분리로 통합이 붙는 걸 본 뒤 위젯 제거 → 중간에도 앱 안 깨짐(잠깐 중복만).
- 데모(키 없음): 자동 위젯 미표시 + 수동 로컬만 → 크래시 없음.

## 9. 테스트 체크리스트

1. `npm.cmd run lint` green (각 서브커밋).
2. **마이그레이션:** `trainer_todo` 생성 + `for all` RLS. 다른 트레이너 계정으로 남의 할일 SELECT 0행 확인. insert→`.select()` 1행.
3. **자동 흡수:** 할일 탭에 오늘 재접근·재등록 타이밍·재등록 재접근이 **회원 탭에서 보이던 것과 동일하게** 뜨는지. 회원 탭엔 더 이상 안 뜨는지(중복 제거 확인).
4. **미확정 클로징(4):** 2차 관찰만 하고 클로징 결과 안 남긴 OT 회원 → 뜨는지. 결과 기록 후 사라지는지. PT 전이 회원은 안 뜨는지.
5. **미처리 예약(5):** 어제 이전 `booked` 예약 → 뜨는지. 완료(done)·취소(canceled) 처리하면 사라지는지. 미래 예약은 안 뜨는지.
6. **수동:** 추가→체크(done 토글)→삭제 왕복. 새로고침 후 상태 유지. 마감일 있는 것 표기.
7. **네비게이션:** 각 자동 항목 탭 → 알맞은 회원·탭(재접근=1차 OT, 재등록=재등록, 미확정 클로징=2차 OT, 미처리 예약=스케줄).
8. **회귀:** 회원 탭 목록·검색·세그먼트 정상. 스케줄·내실적 무변. 데모 모드 크래시 없음.
9. 폰: Vercel 배포 후 하드리프레시 육안(할일 탭·수동 왕복·자동 표시).
