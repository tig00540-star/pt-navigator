# v2 설계 — 자동 스케줄링 (고정 슬롯 + 예외 관리 하이브리드)

> 성격: **방향 합의용 설계 제안**(구현 스펙 아님). OK면 S1부터 구현 스펙(SQL + 라우트/화면)으로 내려간다.
> 협업: 웹Claude(설계) → 트레이너 → 클로드코드(diff) → 트레이너(git diff) → 웹Claude(검토) → 폰/Supabase 검증. 커밋은 트레이너 직접. SQL=Supabase 대시보드 수동(git엔 기록본).
> ★대전제(컨벤션): account 스코프 RLS · 데모 null가드 · purge-safe 정적클래스 · `.select()` 하드닝 · **anon-open(`using(true)`) 금지** · KST · lucide.
> ★개정: 초안(전면 재분배)에서 **고정 슬롯 + 예외 관리 하이브리드**로 전환(현장 정합·양측 만족·구현 경량). 이 파일이 최신본.
> ★확정된 설계 결정(트레이너):
> 1. 기본 모델 = **고정 슬롯 auto-roll + 변경요청**(전면 재분배 아님). 변동 회원만 가용제출+자동분배.
> 2. 주간 횟수 = **트레이너가 회원별로 지정**.
> 3. 강화 2종 **둘 다 채택**: 인앱 변경요청 루프 + 선호/가능 구분.
> 4. 카톡 = **"회원페이지 확인!" 단순 안내 broadcast**.
> 5. 자동화 = **수동 트리거 버튼 + 인앱 확정 배너**(cron/푸시 없음).

## 0. 큰 그림 — 왜 하이브리드인가

**초안(전면 재분배)의 약점:** 모든 회원이 매주 가용을 재제출하고 알고리즘이 매주 처음부터 재분배 → ① 회원은 안 변하는 스케줄을 매주 재입력(마찰) ② "가능"만 보고 배정해 *가능하지만 안 반가운* 시간이 잡힘 ③ "배정당한 느낌"(주도권↓) ④ 트레이너는 미제출자 쫓기 + 결국 수동 조율.

**하이브리드의 뼈대 = 3층:**
1. **고정 슬롯(backbone)** — 대다수 회원은 스케줄이 안정적(매주 월·목 19시). 한 번 합의해 저장 → **매주 다음주로 auto-roll**. 회원·트레이너 재입력 0. 침묵=유지.
2. **변동 꼬리(tail)** — 매주 진짜 바뀌는 소수만 가용제출 → **자동분배 엔진**이 *남은 슬롯*에 배치. 무거운 알고리즘은 여기만 돌아 리스크↓.
3. **예외 루프(exceptions)** — 확정 후 회원이 인앱 **변경요청** → 트레이너 함(inbox). 앱 밖 카톡에 흩어지던 개별 연락을 앱 안으로.

**이미 있는 것(재사용):** `appointment`+`ScheduleBoard`(주간 예약 보드·완료·취소·카톡 clipboard) · 회원앱 `/m/[token]`(회원 세션 `auth_member_id()`·자가입력 패턴) · `schedule_check`(지난 기록 — **가용 아님**, 별개).
**신규:** `recurring_slot` · `member_availability`(+선호) · `schedule_change_request` · `trainer_availability`(+일일상한) · `user_table.weekly_target` · `appointment.source` 확장 + **회원 read RLS**.

## 1. 주간 사이클 & 타임라인

`week_start` = **월요일 0시(KST)**(ScheduleBoard `mondayOf` 재사용). "차주" = 다음 월요일.
★**핵심 의존순서: 공급(트레이너 가용) 먼저 → 수요(회원 가용) 나중.** 트레이너가 다음주 가용을 픽스해야 회원 그리드가 **그 안에서만** 열린다.

```
1회 세팅   트레이너: 반복 가용 템플릿 + 일일 최대 타임. 회원별 고정 슬롯·주간횟수 합의·저장.
~금요일    트레이너: ★[다음주 가용 픽스] — 템플릿 프리필 → 이번주 조정(일요일 기본 열림·못하면 닫기·연차) → 확정.
           = 이게 안 되면 회원 제출 그리드·[스케줄 생성] 비활성(공급이 먼저).
토~일요일  회원: 차주 가용+선호 제출(변동 회원) / 고정 회원은 변경 있을 때만 변경요청.
           ★제출창 **일요일까지 열림**(하드 6시컷 아님). 그리드 = 트레이너 픽스 가용 안에서만 선택.
일요일     트레이너: [다음주 스케줄 생성] → ① 고정 auto-roll ② 변동 분배 ③ 용량/경합/미배치 리포트.
           회원앱: 차주 "확정" 배너 · 트레이너: [카톡 안내 복사]
주중       변동: 회원 인앱 변경요청 → 트레이너 함. 또는 ScheduleBoard 직접 배치/취소.
```

> 실제 마감 = 트레이너가 **일요일 [다음주 스케줄 생성]** 누르는 시점(제출창은 그때까지 열림). 토·일 6시는 안내 문구.
> ★게이트: 트레이너 [다음주 가용 픽스]가 **안 되면** 회원 그리드·[스케줄 생성] 비활성(공급 미확정 상태로 수요 안 받음).

## 2. 데이터 계약 (신규 테이블 · 컬럼 · RLS)

> 마이그레이션 후보(멱등·git 기록본): `docs/migrations/2026-07-20-auto-schedule.sql` (아래 7개 DDL+RLS 한 파일 · 검증 쿼리 포함).

### 2.1 `trainer_availability` — 트레이너 가용(반복 템플릿) + 일일 상한
```
trainer_availability (
  id, account_id default auth_account_id() references account(id),
  trainer_id default auth.uid(),        -- FK 없음(appointment 관례)
  weekday int,                           -- 0=월…6=일
  start_hour int, end_hour int,          -- 근무창(시, end 미포함)
  max_per_day int,                       -- ★일일 최대 타임(과부킹·번아웃 방지 · nullable=무제한)
  created_at timestamptz default now(),
  check (weekday between 0 and 6), check (start_hour>=0 and end_hour<=24 and start_hour<end_hour)
)
```
- 요일별 근무창 여러 행 + 일일 상한. **반복 템플릿**(주 기본값 · week_start 없음).
- RLS = `appointment` 결(account 격리 + owner/본인 가시성).

**`trainer_week_availability` — 다음주 가용 픽스(주별 override · §1 금요일 단계 · ★공급 확정)**
```
trainer_week_availability (
  id, account_id default auth_account_id() references account(id),
  trainer_id default auth.uid(),
  week_start date not null,               -- 이 주에만 적용
  weekday int, start_hour int, end_hour int,
  confirmed_at timestamptz,               -- 픽스(확정) 시각 · null=미확정
  created_at timestamptz default now(),
  check (weekday between 0 and 6), check (start_hour>=0 and end_hour<=24 and start_hour<end_hour)
)
create index twa_week_idx on trainer_week_availability (account_id, week_start);
```
- **[다음주 가용 픽스]** = 반복 템플릿을 이 주로 **프리필**(일요일 포함) → 트레이너 조정(일요일 닫기·연차 등) → **확정**(`confirmed_at`).
- **"그 주 효과적 가용"** = 이 주 확정 rows 있으면 그것, 없으면 반복 템플릿. **회원 그리드·분배 엔진 둘 다 이 효과적 가용만 읽음.**
- 회원 그리드·[스케줄 생성]은 `confirmed_at` 있어야 활성(§1 게이트). RLS = `trainer_availability`와 동일.

### 2.2 `recurring_slot` — 회원 고정 슬롯(backbone · 트레이너 관리)
```
recurring_slot (
  id, account_id default auth_account_id() references account(id),
  trainer_id default auth.uid(),
  user_id uuid not null,                 -- 회원(FK 없음 · appointment 관례)
  weekday int, hour int,                 -- 매주 이 요일·시각 고정
  active boolean default true,           -- 일시중지(삭제 대신)
  created_at timestamptz default now(),
  check (weekday between 0 and 6), check (hour>=0 and hour<24)
)
create index recurring_slot_acc_idx on recurring_slot (account_id, user_id);
```
- 한 회원 여러 행(주2회면 2행). **auto-roll의 소스.** 트레이너 CRUD.
- RLS: account 스코프 CRUD(트레이너/owner) + **회원 read 본인**(`user_id = auth_member_id()` · 회원앱 "내 고정 시간" 표시용, 읽기만).

### 2.3 `member_availability` — 변동 회원 차주 가용 + 선호(week_start별 · 회원 소유)
```
member_availability (
  id, user_id references user_table(id) on delete cascade,
  week_start date, weekday int, start_hour int, end_hour int,
  preference text not null default 'possible'
     check (preference in ('preferred','possible')),   -- ★선호 vs 가능
  created_at timestamptz default now(),
  check (weekday between 0 and 6), check (start_hour>=0 and end_hour<=24 and start_hour<end_hour)
)
create index member_avail_week_idx on member_availability (week_start, user_id);
```
- 재제출 = (user_id, week_start) 기존행 delete→insert(스냅샷 교체 · 하드닝).
- RLS = `schedule_check` 결(회원 본인 CRUD + 트레이너 account read).

### 2.4 `user_table.weekly_target` — 회원별 주간 횟수(트레이너 지정)
```
alter table user_table add column if not exists weekly_target int;   -- null=미지정(분배 제외)
```
- 고정 슬롯 회원은 슬롯 수가 곧 횟수 → 주로 **변동 회원 분배 목표**로 사용. 정책 변경 없음.

### 2.5 `schedule_change_request` — 인앱 변경요청 루프(회원→트레이너 inbox)
```
schedule_change_request (
  id, user_id references user_table(id) on delete cascade,
  appointment_id uuid,                   -- 대상 예약(null=일반 요청)
  week_start date,
  kind text check (kind in ('confirm','change','cancel')),  -- 좋아요/변경/취소요청
  note text,                             -- "목요일로 바꿔주세요"
  status text not null default 'pending' check (status in ('pending','resolved')),
  created_at timestamptz default now()
)
create index scr_pending_idx on schedule_change_request (status, week_start);
```
- 회원 insert/select(본인) · 트레이너 select/update(account 회원 · resolve). RLS 양측.
- 트레이너 '오늘/스케줄'에 **"변경요청 N건"** 뱃지 → 처리 시 ScheduleBoard에서 재배치 후 `status='resolved'`.

### 2.6 `appointment` 확장 — `source` 3값 + 회원 read RLS
```
alter table appointment add column if not exists source text not null default 'manual'
  check (source in ('manual','recurring','auto'));
--  manual=수동 배치 · recurring=고정 슬롯 auto-roll · auto=변동 엔진 분배
alter table appointment add column if not exists matched_preference boolean;
--  배정 슬롯이 회원 '선호(preferred)'와 일치했나 — 형평 부채(§5.3 D) 추적용 · nullable(고정/수동은 null)

drop policy if exists "member_read_appointment" on appointment;
create policy "member_read_appointment" on appointment for select to authenticated
  using (user_id = auth_member_id());     -- 회원 확정 배너 열람(읽기만 · insert/update/delete 회원 불가)
```
- `source`가 **재실행 안전의 핵심**(§3.3): 재생성은 `recurring`+`auto` booked만 갈아끼움 · `manual`·`done`·`canceled` 불변.
- 회원 read: 현재 appointment는 회원이 못 읽음(account 정책뿐) → 이 정책이 회원 유일 통로. **격리 유지**(쓰기 전부 회원 차단 · 회원 세션은 `auth_account_id()` null).

## 3. 고정 슬롯 auto-roll (backbone)

### 3.1 [다음주 스케줄 생성] — 1단계
- 대상 week_start(차주 월). 각 `active` recurring_slot → 그 주 해당 요일·시각으로 appointment insert(`source='recurring', status='booked'`).
- **변경요청 반영**: 그 (user_id, week_start) `kind='cancel'` 요청 있으면 skip, `kind='change'`면 배치 후 트레이너 조율 플래그.
- 일괄 insert 하드닝(`.select()` · 부분실패 요약 반영).

### 3.2 미리보기 → 확정(2-step)
- 버튼은 **미리보기 먼저**(생성 예정 목록·경합·용량 표시) → 트레이너 [확정]에서 실제 insert. 실수 방지.

### 3.3 재실행 안전
- [생성] = ① 그 주 `source in('recurring','auto') and status='booked'` delete → ② 재생성.
- `manual`·`done`·`canceled` 불변. 여러 번 눌러도 수동·완료분 보존.

## 4. 변동 꼬리 자동분배 (엔진)

고정 슬롯이 락된 **뒤 남은 슬롯**을 두고 변동 회원(고정 없음 or `weekly_target>0`)만 배치.

**입력:** 트레이너 가용 슬롯 − 고정/수동 점유 = **남은 슬롯** · 변동 회원별 가용(선호/가능) · `weekly_target`.
**제약:** 1슬롯=1회원 · 슬롯 ⊆ (트레이너 남은 ∩ 회원 가용) · 같은 날 중복 금지(기본) · **트레이너 일일 상한 준수**.
**우선순위(결정형 그리디 · 경합 공정규칙과 동일 §5.3):**
1. 유연도 = (배치가능 슬롯) − (목표). 낮은(제약 심한) 회원부터.
2. 회원별로 목표만큼 선택: **선호 슬롯 먼저 → 요일 분산 → 남은 슬롯 제거**.
3. 못 채우면 **미배치/대기** 리포트.

**출력:** `source='auto', status='booked'` 일괄 insert(하드닝) + 미배치·용량 요약(§5).

## 5. ★ 혼잡 · 용량 문제 (피크 경합) — "다 겹치면 어떻게?"

> 현실: 트레이너 1명 하루 6~8타임 → 주 ~48슬롯. 50명×주2회=100세션. **알고리즘 이전에 수요가 공급의 2배**, 피크(18~21시)엔 더 몰림. **없는 용량은 못 만든다.** 해법 = 병목 가시화 + 공정 배분 + 수요 분산 + 대기열 + 상한/경고.

### 5.1 용량 대시보드(정직함)
- [다음주 스케줄 생성] 미리보기에 **수요 vs 공급**: 총(수요 100 · 공급 48) · **시간대별 히트맵**(피크 18~21시 수요 40 vs 공급 12 = 초과 28). 트레이너가 크런치를 눈으로.

### 5.2 고정 슬롯이 경합을 먼저 줄인다
- 단골 고정 피크 슬롯 = **먼저 락(선점권)**. 경합은 *남은 슬롯 두고 변동 회원끼리*만 → 100명 문제가 작은 꼬리로 축소. 하이브리드의 최대 이점.

### 5.3 경합 우선순위 — 형평 점수(explainable priority)
남은 슬롯 경합 시 변동 회원에게 **형평 점수**를 매겨 높은 순 배정. ★고정 슬롯 회원은 경합 대상 아님(§5.2) — 이 점수는 **변동 꼬리에만** 적용(안정성 유지). 네 신호:

| 신호 | 뜻 | 데이터 출처 |
|---|---|---|
| **대기 W** — 최근수업 오래된 회원↑ | 오래 못 온 회원 우선(starvation 방지) | `daily_workout_log` 마지막 `session_at` — **이미 있음**(memberStatus 14일무수업과 동일 소스) |
| **희소도 S** — 가용 적은 회원↑ | 주1회만 가능 등 대체 슬롯 없는 회원 우선(안 주면 0회) | `member_availability`(계산) |
| **부채 D** — '가능'에 주로 배정됐던 회원↑ | 그동안 선호 양보한 회원 보상(형평 debt) | appointment `matched_preference` 이력(신규 추적 · §2.6) |
| **선호 P** — 이 슬롯을 '선호'로 찍음 | 선호 신호 = 게이트/보너스 | `member_availability.preference` |

- 배정 = **선호(P) 슬롯 먼저** → 경합이면 **점수 = w1·W + w2·S + w3·D** 높은 순 → 동률 user_id 안정정렬(결정형·재현가능).
- 진 회원 → 다음 선호 슬롯 → 다 막히면 **대기/미배치**(§5.5) + **부채 D 누적 → 다음주 상위(= "다음주 선호 우선권")**.
- ★**설명가능성**(회원 수용의 핵심): "이번주는 오래 못 오신 분·시간 빠듯한 분 우선이라 가능시간대로 갔고, 다음주 선호시간 우선권 드려요" — 트레이너가 근거 제시 가능.
- 구현 순서: **W·S(기존 데이터)** S3에 바로 · **D(선호매칭 추적)** 는 §2.6 컬럼 깐 뒤 가중치 추가.

### 5.4 수요 분산 넛지
- 회원 가용제출 화면에 **"이 시간대는 여유 있어요"** 표시(비피크 하이라이트) → 회원이 스스로 비피크 선택 유도(강제 아님). 근원에서 몰림↓.

### 5.5 대기열(waitlist)
- 피크 탈락자 = 대기 등록 → 취소·노쇼로 슬롯 비면 **자동 제안**(변경요청 루프 재사용). 경쟁을 줄서기로. (S 후반·선택)

### 5.6 일일 상한 + 정직한 경고
- `trainer_availability.max_per_day`로 과부킹 방지.
- 구조적 초과(수요≫공급)면 미리보기에 **"이 인원은 현재 가용으로 다 수용 불가 — N세션 초과"** 경고 → 증원·인원캡·시간확대 = **사업 결정을 트레이너에게**. 앱이 억지로 안 우겨넣음.

## 7. OT 특례 — 주간 풀 제외 + 직후 체이닝·픽스

- **OT 회원(`ot_active`)은 auto-roll·자동분배·경합 풀에서 제외**(1차/2차 여정이라 주간 배치에 안 맞음).
- **직후 체이닝**: OT 예약 **완료** 시(ScheduleBoard `complete`) → 회원 상태 OT면 "**다음 OT 예약**" 퀵액션 → 날짜/시간 골라 즉시 insert(`source='manual'` · 고정). 수업 끝나자마자 다음 걸 못박음.
- **픽스**: `source='manual'`이라 §3.3 재생성에서 **안 지워짐** → 자동분배와 충돌 없이 고정.

## 8. 회원앱 (`app/m/[token]/page.jsx`)

**A. 내 고정 시간**(읽기 · `recurring_slot` 본인) — "매주 월19시·목19시" 표시(합의된 고정 확인).

**C. 확정 배너**(자동분배 후 · appointment 본인 read) — "다음주 스케줄 확정" + 슬롯 목록 + [좋아요/변경요청/취소요청](§6). **읽기전용**(회원 appointment 못 씀).

## 12. 스프린트 분해 (하나씩 배포·검증)

1. **S1 — 스키마·RLS 뼈대**: `2026-07-20-auto-schedule.sql`(7 DDL). SQL 검증(정책 수·회원 격리). 화면 없음.
2. **S2 — 고정 슬롯 + auto-roll**: 트레이너 반복 가용/일일상한 + **[다음주 가용 픽스]**(주별 override·게이트) · 회원 고정슬롯 관리 · **[다음주 스케줄 생성]**(recurring→appointment·미리보기·재실행 안전) · 회원앱 "내 고정 시간". → **backbone만으로 이미 실사용 가능**.
3. **S3 — 변동 꼬리 + 경합/용량**: 회원 가용제출(선호/가능·여유힌트) · 자동분배 엔진(§4) · 용량 대시보드·공정규칙·미배치 리포트(§5.1~5.4).
4. **S4 — 확정·안내·변경요청**: 회원앱 확정 배너 + 변경요청 루프(§6) + 카톡 안내(§10) + 트레이너 변경요청 함.
5. **S5 — OT 체이닝 + 대기열**: 완료 모달 "다음 OT 잡기"(§7) · 대기열 자동제안(§5.5).

> S2까지만 해도 "고정 스케줄 자동 굴리기"가 돌아 대다수 회원 커버. S3~S5는 변동·경합·예외의 꼬리를 얹는 순서. 리스크 큰 알고리즘(S3)이 뒤라 안전.

## 13. 남은 결정 / 엣지 (합의 후 S1)
- **가용 입력 단위**: 1시간 슬롯 체크 권장(트레이너 근무창·ScheduleBoard 시간格과 교집합 단순). 시간대(6~9시) 방식 원하면 알려줘.
- **형평 점수 가중치(w1·w2·w3)**: 대기·희소도·부채 비중은 S3에서 실데이터로 튜닝. 추적은 **§2.6 `matched_preference` + `daily_workout_log` 마지막수업**으로 계산(별 테이블 불필요).
- **owner(멀티 트레이너)**: recurring/분배는 trainer_id별 독립. S2·S3는 단일 트레이너 먼저 검증 후 owner 확장.
- **대기열 세부**·**노쇼 패널티** = 후속. (트레이너 주별 가용 override는 §2.1로 코어 편입 완료.)
- week 경계 전부 KST 월요일(클라 `date.js` 결 · 서버시각 안 씀).

이 설계 OK면 **S1(스키마·RLS) 구현 스펙**부터 내려갈게 — `2026-07-20-auto-schedule.sql` 7개 DDL+RLS + 검증 절차.
