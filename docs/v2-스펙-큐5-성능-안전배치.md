## ⚠️ 리포트와 어긋난 2가지 (스펙에 반영됨 · 먼저 읽기)

**(1) `tab-anim`은 지금 "죽은 클래스"다.** 리포트 §5는 "`PTView.jsx:64`가 이미 CSS(`tab-anim`)로 같은 효과를 낸다"고 했지만, **`app/globals.css`에 `.tab-anim` 정의가 없다**(전수 확인). 즉 PTView 탭 전환엔 **지금 아무 애니메이션도 없다**(무동작 클래스만 붙어 있음). 따라서 motion 제거는 "PTView처럼 하면 됨"이 아니라, **globals.css에 실제 `.tab-anim`을 신설**해야 하고 — 그러면 app/page.jsx의 번들(~120KB)이 빠지는 동시에 **PTView도 비로소 애니메이션이 생긴다(공짜 이득)**.

**(2) 날짜창(date window)은 이번 대상에서 뺀다.** 리포트 §5/§10-5는 `ChurnRiskToday`·관리자·`MyStats`에 "날짜 필터"를 제안했지만, `lib/memberStatus.js` 확인 결과 이 위젯들은 **월-스코프 지표(이달 매출·이달 수업)와 생애-스코프 지표(총 잔여·클로징률·이탈 판정)를 한 데이터셋으로 동시 계산**한다:
- `ChurnRiskToday`: "14일+ 무수업"을 판정하려면 회원의 **마지막 수업(임의로 오래됨)**이 필요 → 최근 N일로 자르면 이탈 회원이 **이탈 목록에서 사라진다**(정반대 결과).
- `MyStats` PT현황/잔여 카드: `remainingSessions`는 **계약 전체 이력**을 봐야 정확 → 월 창으로 자르면 잔여 과대계산.
- `closingStats`: ot_log **전기간** 회원별 클로징 결과 → 월 창이면 클로징률 왜곡.

→ 블랭킷 날짜창은 **정확도를 깨는 의미 변경**이다. 진짜 절감은 **서버측 집계 RPC**인데 이건 §10-5에서 이미 "베타 후"로 결정됨. 그래서 이번엔 **의미 불변 최적화(컬럼 축소·useMemo·읽기 캐시)만** 하고, 날짜창/RPC는 후속으로 남긴다(§후속-D).

## 배치 5-A — 순수 이득 (의미 변화 0)

### 5-A-1. `motion`(framer) 제거 → 실제 CSS `.tab-anim` 신설
**목적:** `motion`(≈120KB gz)을 탭 전환 1곳에만 쓰는 걸 제거하고, CSS 애니메이션으로 대체. 부수효과로 PTView 애니메이션도 살아남.

**(a) 선행 확인(클로드코드).** 제거 전, 레포 전체에서 `motion`이 정말 이 한 곳만인지 grep:
```
git grep -n "motion/react\|framer-motion\|<motion\.\|AnimatePresence"
```
`app/page.jsx` 외 결과가 나오면 **멈추고 대수에게 보고**(스펙 전제 깨짐). 없으면 진행.

**(b) `app/globals.css`** — 파일 하단(마지막 `input[type="date"]` 블록 뒤)에 추가:
```css
/* 탭 전환 진입 애니메이션 — motion(framer) 대체(번들 ~120KB 제거). app/page.jsx·PTView 공용.
   key 변경 시 요소가 재마운트되며 1회 재생. exit 애니메이션은 없음(enter-only). */
@keyframes tabIn {
  from { opacity: 0; transform: translateY(12px); }
  to   { opacity: 1; transform: translateY(0); }
}
.tab-anim { animation: tabIn 0.26s cubic-bezier(0.22, 1, 0.36, 1) both; }
@media (prefers-reduced-motion: reduce) { .tab-anim { animation: none; } }
```

**(c) `app/page.jsx`** — import 라인 제거:
```js
import { motion, AnimatePresence } from "motion/react";
```
그리고 `<main>` 안의 래퍼를 교체. **현재:**
```jsx
<AnimatePresence mode="wait" initial={false}>
  <motion.div
    key={tab}
    initial={{ opacity: 0, y: 12 }}
    animate={{ opacity: 1, y: 0 }}
    exit={{ opacity: 0, y: -8 }}
    transition={{ type: "spring", stiffness: 420, damping: 34, mass: 0.8 }}
  >
    {tab === 9 ? (
      ... (내부 전부 그대로) ...
    )}
  </motion.div>
</AnimatePresence>
```
**교체 후:**
```jsx
<div key={tab} className="tab-anim">
  {tab === 9 ? (
    ... (내부 전부 그대로) ...
  )}
</div>
```
- 내부 JSX(`{tab === 9 ? ... : ...}` 전체 트리)는 **한 글자도 안 바꾼다.** 래퍼만 교체.
- `key={tab}` 유지가 핵심 — 탭 바뀌면 div 재마운트 → CSS 애니메이션 1회 재생(PTView와 동일 패턴).
- **의미 변화(수용):** 나가는 탭의 exit 페이드(y -8)만 사라짐. PTView는 원래 exit 없었고, 진입 느낌은 유지. 앱 체감 거의 동일.

**(d) (선택 · 대수 판단)** 다른 사용처가 없으면 `npm.cmd rm motion`으로 의존성 제거 가능. import만 지워도 번들에선 빠지므로 **필수는 아님**. package.json 변경은 별도 판단.

**커밋 경계:** 5-A-1 단독 1커밋. 예: `perf(bundle): motion(framer) 제거 → CSS tab-anim 신설(≈120KB↓, PTView 애니 복구)`

### 5-A-2. `center_machine` 읽기 캐시 (`lib/centerMachines.js` 신설)
**목적:** 거의 안 변하는 설정 데이터를 세 곳이 마운트마다 각자 조회(§5). **읽기 소비처 2곳**(VoiceLogTab·PtWorkoutTab)을 모듈 캐시 1벌로. 편집기(CenterMachineSettings)는 실시간 CRUD라 **직접 조회 유지 + 변경 시 캐시 무효화**.

**(a) 새 파일 `lib/centerMachines.js`** (`git add` 먼저):
```js
import { supabase } from "@/lib/supabaseClient";

/* center_machine 읽기 전용 공유 캐시 — 거의 안 변하는 설정 데이터.
   읽기 소비처(VoiceLogTab·PtWorkoutTab)가 마운트마다 각자 조회하던 걸 SPA 세션 1벌로.
   편집기 CenterMachineSettings는 직접 조회 유지 + save/remove 후 invalidate() 호출.
   RLS/account 스코프는 기존과 동일(select에 account 필터 없음 = DB DEFAULT+RLS 위임 유지). */
let cache = null; // Promise<rows[]> | null

export function loadCenterMachines() {
  if (!supabase) return Promise.resolve([]); // 데모: 빈 목록(= 기존 동작)
  if (!cache) {
    cache = supabase
      .from("center_machine")
      .select("*")
      .order("kind", { ascending: true })
      .order("name", { ascending: true })
      .then(({ data, error }) => {
        if (error) { cache = null; return []; } // 실패 시 캐시 비워 다음 마운트 재시도
        return data || [];
      });
  }
  return cache;
}

export function invalidateCenterMachines() { cache = null; }
```

**(b) `components/tabs/VoiceLogTab.jsx`** — center_machine 조회 effect. **현재:**
```js
if (!supabase) return;
const { data } = await supabase.from("center_machine").select("name, cues");
if (cancelled) return;
const list = (data || [])
  .filter((r) => r?.name && Array.isArray(r.cues) && r.cues.length)
  .map((r) => ({ name: r.name.trim(), cues: r.cues }));
setMachineCues(list);
```
**교체 후:**
```js
const data = await loadCenterMachines();
if (cancelled) return;
const list = (data || [])
  .filter((r) => r?.name && Array.isArray(r.cues) && r.cues.length)
  .map((r) => ({ name: r.name.trim(), cues: r.cues }));
setMachineCues(list);
```
- `loadCenterMachines()`가 데모(supabase null)도 처리하므로 `if (!supabase) return;`는 제거(effect가 `setMachineCues([])`로 자연 degrade). 파생 로직(filter/map)은 **그대로** — 캐시는 `select("*")`라 `name`·`cues` 컬럼 포함, 결과 동일.
- 파일 상단에 `import { loadCenterMachines } from "@/lib/centerMachines";` 추가.

**(c) `components/views/PtWorkoutTab.jsx`** — center_machine 조회 effect. **현재:**
```js
if (!supabase) return; // 데모면 빈 목록 = 자유 타이핑
const { data } = await supabase.from("center_machine").select("name").order("name", { ascending: true });
if (cancelled) return;
const names = [...new Set((data || []).map((r) => (r.name || "").trim()).filter(Boolean))];
setMachineOptions(names);
```
**교체 후:**
```js
const data = await loadCenterMachines();
if (cancelled) return;
const names = [...new Set((data || []).map((r) => (r.name || "").trim()).filter(Boolean))]
  .sort((a, b) => a.localeCompare(b, "ko"));
setMachineOptions(names);
```
- 기존엔 `.order("name")`로 이름순이었음 → 캐시는 kind·name 순이라, datalist 이름순 유지를 위해 dedupe 후 `localeCompare("ko")` 정렬 추가(동작 동일하게 보존).
- `import { loadCenterMachines } from "@/lib/centerMachines";` 추가.

**(d) `components/views/CenterMachineSettings.jsx`** — 편집기. 조회 effect는 **그대로 직접 조회 유지**(편집기는 항상 최신·자체 ordering 필요). 다만 mutation 성공 후 캐시 무효화:
- 상단 `import { invalidateCenterMachines } from "@/lib/centerMachines";` 추가.
- `save`의 insert/update **성공 분기**(각각 `setRows(...)` 직후, `showToast("수정됨")`/`showToast("추가됨")` 부근)와 `remove`의 성공 분기(`setRows((p) => p.filter(...))` 직후)에 `invalidateCenterMachines();` 한 줄씩 추가. 데모(supabase null) 분기엔 불필요.
- **이유:** 원장이 장비를 바꾸면 다음에 VoiceLog/PtWorkout이 마운트될 때 새 목록을 받게. (이미 떠 있는 탭 즉시 갱신은 원래도 안 했으니 무변.)

**커밋 경계:** 5-A-2 단독 1커밋. 예: `perf(center_machine): 읽기 공유 캐시(lib/centerMachines) — VoiceLog·PtWorkout 마운트별 중복조회 제거 + 편집 시 무효화`

## 배치 5-B — 집계 안정화 (의미 변화 0)

### 5-B-1. `ChurnRiskToday` — `risky` 집계 `useMemo` + session_log 컬럼 축소
**목적:** '오늘' 탭이라 앱 열 때마다 렌더되는데, 회원×로그 **O(n²) 집계를 렌더 본문에서 매 렌더 실행**(§5). + `session_log.select("*")` 전량.

**(a) `import { useEffect, useState } from "react";` → `useMemo` 추가:**
```js
import { useEffect, useMemo, useState } from "react";
```

**(b) session_log 조회 컬럼 축소.** **현재:**
```js
supabase.from("session_log").select("*"),
```
**교체:**
```js
supabase.from("session_log").select("id, user_id, started_at, created_at, sessions_total, service_sessions"),
```
- 근거(memberStatus 확인): `remainingSessions`=`id`·`sessions_total`·`service_sessions`, `activeContract`=`started_at`, 본체=`user_id`·`active.started_at`·`active.created_at`. 이 6개면 충분.
- `daily_workout_log` 조회는 **그대로 두기**(이미 컬럼 축소됨: `user_id, session_at, created_at, voided, source`). ⚠️ **여기 contract_id를 추가하지 말 것** — 추가하면 `remainingSessions`의 `used` 계산이 바뀌어 **잔여 표시 값이 달라진다(의미 변경)**. (아래 §관찰 참고 — 이건 별건.)

**(c) `risky` 계산을 useMemo로.** **현재**(렌더 본문, `if (!supabase) return null;` 아래):
```js
const risky = members
  .filter((m) => viewFor(m) === "pt")
  .map((m) => { ...전체... })
  .filter(Boolean)
  .sort((a, b) => b.gap - a.gap);
```
**교체:** 동일 식을 `useMemo`로 감싸기(내용 **동일**, deps만 명시):
```js
const risky = useMemo(() =>
  members
    .filter((m) => viewFor(m) === "pt")
    .map((m) => { ...전체 동일... })
    .filter(Boolean)
    .sort((a, b) => b.gap - a.gap),
  [members, contracts, logs]
);
```
- ⚠️ **훅 순서 규칙:** `useMemo`는 `if (!supabase) return null;` **위**로 올려야 한다(early return 뒤에 훅 두면 규칙 위반). `useMemo` 안에서 `logs`/`contracts` 빈 배열이면 자연히 `risky=[]`가 되고, `if (!supabase) return null;`은 그대로 유지(그 아래 `if (loading || risky.length === 0) return null;`도 유지). 즉 **useState 3개 → useEffect → useMemo → early returns** 순서(모든 훅이 early return 앞).
- `daysSince`가 `Date.now()`를 쓰지만, 세션 중 재계산은 데이터 변경 시에만 → 이탈은 '일' 단위라 무해(허용).

**커밋 경계:** 5-B-1 단독 1커밋. 예: `perf(ChurnRiskToday): O(n²) 집계 useMemo + session_log 컬럼 축소(의미 불변)`

### 5-B-2. `MyStats` (선택 · 낮은 우선순위)
파생 집계가 많지만 데이터가 1회 로드라 useMemo 이득이 작고, 실질 무게인 6개 `select("*")` 컬럼 축소는 memberStatus 의존이 넓어 **이번 배치에선 보류**. 굳이 한다면 `rev/priceSum/closing/sessionRows/revRows/refundRows` 등을 `[contracts, logs, otRows, members, uid, ...]` deps로 useMemo만(의미 불변). **권장: 스킵**하고 5-A/5-B-1에 집중.

## §후속-D — 날짜창/서버 집계 (의미 변경 · 이번 미구현)
위 §어긋남(2)의 이유로 블랭킷 날짜창은 정확도를 깬다. 안전한 대안은 **서버측 집계 RPC**(월-스코프 지표는 DB에서 집계, 생애-스코프는 별도)인데 §10-5에서 **베타 후**로 결정됨. 관리자 대시보드(`admin/page.jsx`의 7테이블 `Promise.all` `select("*")`)도 같은 함정 — **컬럼 축소는 안전하나** 날짜창은 위험. 관리자 렌더 전체가 정말 월-스코프만 쓰는지 전수 확인이 선행돼야 하므로 **별도 스펙**으로 분리. 이번엔 손대지 않음.
