## 0. 미니 에픽 개요 (자가입력 3종 · AI는 베타 후)

> 베타용 회원앱 완성 = **읽기(완료) + 자가입력 저비용 3종**. 식단/루틴 AI는 베타에서 사용 검증 후.

| 스프린트 | 내용 | 크기 | 비고 |
|---|---|---|---|
| **M1**(이번) | **회원 쓰기 토대 + 유산소 기록** | 중 | 회원 INSERT 패턴을 제일 단순한 기능으로 확립 |
| M2 | 비포애프터 사진 | 중~상 | Supabase Storage + RLS. 토대 위에 얹음. 최고 만족도 |
| M3 | 스케줄 체크(개인운동/PT날) | 소~중 | M1 쓰기 패턴 재사용 |

**M1이 토대인 이유:** 지금 회원은 **열람 전용**(SELECT는 안전뷰, 쓰기 정책 없음). M1에서 **회원이 자기 데이터를 INSERT/수정/삭제하고, 트레이너가 그걸 읽는** 패턴(테이블 + RLS 4종 + 트레이너 조회)을 처음 세운다. M2·M3는 이 패턴을 복제.

## 1. 데이터 계약

### 1.1 `cardio_log` (신규 테이블 · 회원 소유)

```
cardio_log (
  id            uuid pk default gen_random_uuid(),
  user_id       uuid not null references user_table(id) on delete cascade,
  performed_on  date not null default current_date,   -- 운동 날짜
  kind          text,      -- 종류(러닝/사이클/걷기/기타 · 자유 입력)
  minutes       int,       -- 시간(분)
  note          text,      -- 메모(선택)
  created_at    timestamptz default now()
)
```

- **민감 컬럼 없음**(전부 회원 본인 입력) → 회원이 raw 테이블 직접 SELECT 정책으로 읽음(S2의 definer-view 규약은 트레이너-작성·민감데이터용 · 여기선 불필요).
- `account_id` 컬럼 **안 둠** — 트레이너 스코프는 `user_table` 조인으로(회원 INSERT를 단순하게, 스푸핑 여지 제거).
- 프리미엄 게이트: **회원은 premium일 때만 로그인됨(B2)** → 로그인된 회원 = premium. cardio_log에 별도 plan 게이트 불필요.

### 1.2 마이그레이션

**`docs/migrations/2026-07-1x-member-cardio.sql`**

```sql
-- =============================================================================
-- 회원 자가입력 M1 — cardio_log(유산소) 테이블 + 회원 쓰기 토대(RLS 4종) + 트레이너 조회
-- 실행: Supabase SQL Editor(수동) · git 기록본. 멱등.
-- 전제: auth_member_id()(회원 세션→user_table.id) · auth_account_id()(트레이너 세션→account_id).
-- =============================================================================
create table if not exists cardio_log (
  id           uuid primary key default gen_random_uuid(),
  user_id      uuid not null references user_table(id) on delete cascade,
  performed_on date not null default current_date,
  kind         text,
  minutes      int,
  note         text,
  created_at   timestamptz default now()
);
create index if not exists cardio_log_user_idx on cardio_log (user_id, performed_on desc);

alter table cardio_log enable row level security;

-- 회원: 본인 것만 CRUD (user_id = 내 회원 id)
drop policy if exists "member_cardio_select" on cardio_log;
create policy "member_cardio_select" on cardio_log for select to authenticated
  using (user_id = auth_member_id());
drop policy if exists "member_cardio_insert" on cardio_log;
create policy "member_cardio_insert" on cardio_log for insert to authenticated
  with check (user_id = auth_member_id());
drop policy if exists "member_cardio_update" on cardio_log;
create policy "member_cardio_update" on cardio_log for update to authenticated
  using (user_id = auth_member_id()) with check (user_id = auth_member_id());
drop policy if exists "member_cardio_delete" on cardio_log;
create policy "member_cardio_delete" on cardio_log for delete to authenticated
  using (user_id = auth_member_id());

-- 트레이너: 자기 account 회원 것 조회(읽기만). user_table 조인으로 account 스코프.
drop policy if exists "trainer_cardio_select" on cardio_log;
create policy "trainer_cardio_select" on cardio_log for select to authenticated
  using (exists (
    select 1 from user_table u
    where u.id = cardio_log.user_id and u.account_id = auth_account_id()
  ));

-- 검증:
--   (회원 세션) insert/select/update/delete 본인 것만 됨. 타 user_id insert 거부.
--   (트레이너 세션) 자기 account 회원 cardio 조회됨 · 타 account 0행.
-- 롤백: drop table if exists cardio_log cascade;
```

> **회원 INSERT 안전:** `with check (user_id = auth_member_id())` — 클라가 `user_id`를 남의 것으로 넣어도 RLS가 거부. 회원 클라는 자기 `me.id`만 넣음.

## 2. 회원앱 UI — `app/m/[token]/page.jsx` (유산소 섹션 추가)

**위치:** "인바디 변화" 또는 "종목별 무게 변화" 아래에 **"유산소 기록"** 섹션 추가.

**동작:**
- **로드:** `loadHome()`에 `cardio` 조회 추가 — `memberSupabase.from("cardio_log").select("*").order("performed_on", { ascending: false }).limit(30)`. state `cardio`.
- **추가 폼(간단):** 날짜(기본 오늘)·종류(텍스트, 예: 러닝)·시간(분, number)·메모(선택) → [기록] → `memberSupabase.from("cardio_log").insert({ user_id: me.id, performed_on, kind, minutes: Number(minutes)||null, note })` → 성공 시 목록 앞에 낙관 추가(또는 재조회). 회원용이라 **글씨 크게·입력 최소**.
- **목록:** 날짜·종류·분·메모 카드. 각 항목 삭제(휴지통) → `delete().eq("id", ...)`(RLS로 본인 것만).
- **빈 상태:** "아직 유산소 기록이 없어요. 오늘 운동을 남겨보세요."
- **데모/키 없음:** `!memberSupabase` 가드(기존 패턴).
- **에러 하드닝:** insert/delete 시 `.select()`로 반환 확인(0행이면 실패 토스트 — 교훈1 패턴).

**톤/스타일:** 기존 회원앱 토큰(`bg-card`·`border-line`·`Eyebrow`·`Button`·큰 글씨). lucide 아이콘(예: `Activity`/`Footprints`). 새 색·동적클래스 금지.

**주의(React):** `me`가 로드된 뒤에만 폼 활성(user_id 필요). set-state-in-effect 회피 패턴 유지(기존 `loadHome`과 동일).

## 3. 트레이너 앱 — 회원 유산소 조회 (읽기 · 작게)

트레이너가 "회원에게 안 물어봐도" 유산소 활동을 봄(이 기능의 절반의 가치).

- **위치:** PT 뷰의 회원 상세(인바디/일지 근처) 또는 회원 상세 하단에 "유산소 기록(회원 입력)" 요약.
- **조회:** `supabase.from("cardio_log").select("*").eq("user_id", member.id).order("performed_on", {ascending:false}).limit(10)` (트레이너 정책으로 자기 account 회원만).
- **표시:** 최근 N건 리스트(날짜·종류·분) + 이번 주 합계(분) 정도. 읽기 전용(트레이너는 수정 안 함).
- **범위 최소화:** M1에선 **간단 리스트/합계만**. 차트·추이는 후속(원하면).

> 트레이너 UI가 부담되면 M1은 **회원 입력 + 회원 조회까지**로 하고 트레이너 조회는 M1b로 분리 가능. 단 RLS(트레이너 select 정책)는 이번에 함께 깔아 둠(재작업 없게).

## 5. 주의·엣지

- **회원 쓰기 토대 = 이 패턴이 M2·M3의 틀:** 새 테이블 + 회원 CRUD 4종(`user_id = auth_member_id()`) + 트레이너 select(account 조인). M2(사진)는 여기에 **Storage**만 추가, M3(스케줄)은 이 패턴 복제.
- **민감정보 없음** → 회원 raw SELECT 정책 허용(S2 definer-view 규약은 트레이너-작성 데이터용). 만약 뒤에 민감 필드가 붙으면 그때 뷰로.
- **프리미엄/AI 비용:** 유산소는 **AI 없음 = COGS 0**. (식단/루틴 AI만 회원당 비용 → 베타 후 AI 게이트로.)
- **트레이너 쓰기 안 함:** 트레이너는 cardio_log 읽기만(회원 소유 데이터). 트레이너가 대신 입력하는 시나리오는 범위 밖.
- **on delete cascade:** 회원 삭제(user_table) 시 cardio_log 자동 정리. 'PT 종료'(auth 유저만 삭제)는 회원 행 보존이라 유산소도 보존(재개통 시 그대로).
